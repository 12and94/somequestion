## 题目：实现对象池

### 标准准确的说法
- 目标：减少频繁分配释放带来的开销与碎片。
- 核心结构：
- 预分配内存块 + 空闲链表。
- 借出对象时从空闲链取块，归还时回收。
- 生命周期策略：
1. 每次借出构造/归还析构（语义严格）。
2. 仅重置字段，减少构造析构成本（需确保资源安全）。
- 并发场景要考虑锁或无锁空闲队列。

### 通俗易懂的理解
- 对象池像“共享单车车棚”：先准备好一批车，借还都在车棚内流转，不每次造新车。

### 面试回答简版模板
`对象池通过预分配和复用把分配成本前置，核心是空闲块管理和生命周期策略。实现时要明确并发安全和资源清理语义。`

### 对应示例
```cpp
template<class T>
class ObjectPool {
public:
    explicit ObjectPool(size_t n) {
        storage_.resize(n);
        for (size_t i = 0; i < n; ++i) free_.push_back((int)i);
    }

    template<class... Args>
    T* acquire(Args&&... args) {
        if (free_.empty()) return nullptr;
        int id = free_.back(); free_.pop_back();
        T* p = reinterpret_cast<T*>(&storage_[id]);
        new (p) T(std::forward<Args>(args)...);
        return p;
    }

    void release(T* p) {
        if (!p) return;
        int id = (int)(reinterpret_cast<Slot*>(p) - storage_.data());
        p->~T();
        free_.push_back(id);
    }

private:
    using Slot = std::aligned_storage_t<sizeof(T), alignof(T)>;
    std::vector<Slot> storage_;
    std::vector<int> free_;
};
```

## 题目：排行榜设计（Top100 + 自身排名）

### 标准准确的说法
- 需求拆解：
- 实时更新分数。
- 快速返回 Top100。
- 快速查询某用户当前排名。
- 常见结构：
- `unordered_map<uid, score>` 存当前分。
- 有序结构（平衡树/跳表）按 `(score desc, uid asc)` 维护全局顺序。
- Top100 直接取有序结构前 100。
- 自身排名可用 order-statistics tree（支持 k-th / rank）或 Fenwick/线段树（离散分值）。

### 通俗易懂的理解
- 一张“用户分数字典”负责查个人，一张“按分数排好的榜单”负责取 Top 和排名。

### 面试回答简版模板
`排行榜常用“哈希 + 有序结构”双索引：哈希做用户定位，有序结构做 TopK 与 rank。核心是更新时要先删旧值再插新值。`

### 对应示例
```cpp
// 思路示意：
// 1) map: uid -> score
// 2) ordered set: (-score, uid)
// 更新：erase(old pair), insert(new pair)
// Top100：ordered set 前 100 个
// rank(uid)：有序结构里求 order_of_key((-score, uid)) + 1
```

## 题目：设计定时器

### 标准准确的说法
- 常见实现：最小堆按到期时间排序。
- 新任务入堆，调度线程睡到最近到期时间，触发后执行并处理重复任务（周期任务）。
- 关键点：
- 时钟选择（单调时钟避免系统时间回拨影响）。
- 取消语义（惰性删除或版本号）。
- 回调执行策略（直接执行或投递线程池）。

### 通俗易懂的理解
- 定时器像“闹钟堆”：最近要响的闹钟放最上面，系统只盯最早那个。

### 面试回答简版模板
`定时器常用最小堆维护到期时间，调度线程等待最近任务。工程上要处理取消、周期任务和回调线程模型。`

### 对应示例
```cpp

    int64_t dueMs;
    int id;
    std::function<void()> cb;
    bool operator>(const TimerTask& o) const { return dueMs > o.dueMs; }
```

## 题目：设计线程池

### 标准准确的说法
- 核心组件：
- 线程安全任务队列。
- Worker 线程循环。
- 条件变量唤醒。
- 停机协议（立即停机 / drain）。
- 关键策略：
- 队列满时行为（阻塞/拒绝/caller-runs）。
- 任务异常隔离。
- 监控指标（队列长度、等待时延、拒绝次数）。

### 通俗易懂的理解
- 线程池像“固定工人 + 待办队列”，重点在调度规则和停机规则，而不是线程数量本身。

### 面试回答简版模板
`线程池设计要点是任务队列、worker 循环、同步唤醒和停机语义，工程重点是背压、异常隔离和可观测性。`

### 对应示例
```cpp
class ThreadPool {
public:
    explicit ThreadPool(size_t n);
    ~ThreadPool();
    template<class F>
    void submit(F&& f);
    void shutdown(bool drain = true);
private:
    std::mutex m_;
    std::condition_variable cv_;
    std::queue<std::function<void()>> q_;
    std::vector<std::thread> workers_;
    bool stopping_ = false;
    bool drain_ = true;
};
```

## 题目：设计背包系统

### 标准准确的说法
- 常见需求：
- 物品堆叠（同类可叠加）。
- 容量限制（格子数、重量、体积）。
- 原子操作（加/删/移动物品要么全成功要么失败）。
- 常见数据模型：
- `Slot`：`itemId + count + meta`。
- `Inventory`：固定槽位数组 + `itemId -> 槽位索引集合` 辅助索引。
- 并发与一致性：
- 多线程/多请求下要加锁或串行化，避免重复扣减和脏读。

### 通俗易懂的理解
- 背包系统像“有限格子的货架”：要解决“放哪格、能不能叠、满了怎么办、交易失败要回滚”。

### 面试回答简版模板
`背包设计核心是数据模型和一致性：槽位结构 + 叠加规则 + 容量约束 + 原子更新。并发场景要保证操作可回滚、状态不脏。`

### 对应示例
```cpp
struct Slot {
    int itemId = 0;
    int count = 0;
};

class Inventory {
public:
    explicit Inventory(int n): slots_(n) {}

    bool addItem(int itemId, int cnt, int stackLimit) {
        // 先填已有堆，再找空位，放不下返回 false
        for (auto& s : slots_) {
            if (s.itemId == itemId && s.count < stackLimit) {
                int put = std::min(cnt, stackLimit - s.count);
                s.count += put; cnt -= put;
                if (cnt == 0) return true;
            }
        }
        for (auto& s : slots_) {
            if (s.itemId == 0) {
                int put = std::min(cnt, stackLimit);
                s.itemId = itemId; s.count = put; cnt -= put;
                if (cnt == 0) return true;
            }
        }
        return false;
    }

private:
    std::vector<Slot> slots_;
};
```
