## 题目：异常处理（C++ exception 体系）
>[!NOTE]- 标准准确的说法
>- 先给定义：**异常是“函数无法在本层处理错误时，把错误沿调用栈向上传递”的机制**。
>- 传播流程是 `throw -> stack unwinding -> catch`：
>  - `throw`：构造异常对象并开始查找匹配的 `catch`。
>  - `stack unwinding`：从抛出点向外层函数逐层退出，沿途局部对象析构（这就是 RAII 在异常下仍安全的原因）。
>  - `catch`：找到第一个类型匹配的处理器后进入处理逻辑。
>- 类型规则（面试高频）：
>  - 推荐 **throw by value, catch by const reference**。
>  - `catch` 顺序要“**先子类、后父类**”，否则父类会提前吞掉子类。
>  - 需要继续向上抛时用 `throw;`（保留原异常），不要写 `throw e;`（可能切片/丢失动态类型）。
>- `noexcept` 语义：函数承诺不抛。若异常逃出 `noexcept` 函数，程序直接 `std::terminate`。
>  - 常见应 `noexcept` 的位置：析构函数、移动构造/移动赋值、低层关键路径。
>- 析构函数原则：**析构函数不要抛异常**。
>  - 尤其在栈展开期间再抛新异常，会立即 `std::terminate`。
>- 异常安全等级（必须会说）：
>  - `basic guarantee`：操作失败后对象仍处于可析构、无资源泄漏的有效状态。
>  - `strong guarantee`：操作失败不改变原状态（事务语义，commit-or-rollback）。
>  - `nothrow guarantee`：操作保证不抛异常。
>- 什么时候该用异常：
>  - 用：构造失败、资源获取失败、不可在本层恢复的错误。
>  - 不用：正常业务分支（如“查不到数据”“权限不足”），这类更适合返回值/错误码。
>- 工程治理建议：**在模块边界做异常翻译和收敛**（底层技术异常 -> 业务异常/错误码），避免异常类型在系统里无约束扩散。
>- 线程边界注意：异常不能“自动跨线程”传播。
>  - 例如 `std::thread` 里未捕获异常会终止线程/程序；`std::async` 会把异常保存到 `future.get()` 时再抛。
>- 性能认知：多数实现是“正常路径零额外分支成本，异常路径成本高”。因此异常适合“低频错误”，不适合热路径分支控制。
>- 术语补充（流程三词）：
>  - `throw`：抛出异常对象，当前函数从抛出点中断。
>  - `stack unwinding`：调用栈逐层回退；每退一层会析构该层局部对象。
>  - `catch`：外层匹配到处理器并处理；若始终找不到，最终 `std::terminate`。
>- 术语补充（函数栈帧）：
>  - 栈帧是“一次函数调用在栈上的现场记录”，常含返回地址、参数副本、局部变量。
>  - 若 `main -> A -> B -> C`，则 `C` 在栈顶；`C` 内 `throw` 后先退出 `C` 栈帧，再退 `B/A`，直到某层 `catch`。

>[!tip]- 代码示例
>```cpp
>#include <iostream>
>#include <stdexcept>
>#include <string>
>
>// 观察栈展开顺序用的 RAII 对象
>struct Guard {
>  std::string name;
>  explicit Guard(std::string n) : name(std::move(n)) {
>    std::cout << "enter " << name << "\n";
>  }
>  ~Guard() noexcept { // 析构建议 noexcept
>    std::cout << "leave " << name << "\n";
>  }
>};
>
>class ConfigError : public std::runtime_error {
>public:
>  explicit ConfigError(const std::string& msg) : std::runtime_error(msg) {}
>};
>
>int parse_port_inner(const std::string& s) {
>  Guard g("parse_port_inner");
>  size_t pos = 0;
>  int v = std::stoi(s, &pos); // 可能抛 invalid_argument/out_of_range
>  if (pos != s.size() || v <= 0 || v > 65535) {
>    throw ConfigError("invalid port: " + s); // throw by value
>  }
>  return v;
>}
>
>int parse_port(const std::string& s) {
>  Guard g("parse_port");
>  try {
>    return parse_port_inner(s);
>  } catch (const std::invalid_argument&) { // catch by const reference
>    // 模块边界翻译：技术异常 -> 领域异常
>    throw ConfigError("port is not a number: " + s);
>  } catch (const std::out_of_range&) {
>    throw ConfigError("port out of range: " + s);
>  }
>}
>
>bool init_server(const std::string& s, int& port) noexcept { // 边界收敛为返回值
>  try {
>    port = parse_port(s);
>    return true;
>  } catch (const ConfigError& e) {
>    std::cerr << "[config] " << e.what() << "\n";
>    return false;
>  }
>}
>
>int main() {
>  int port = 0;
>  init_server("abc", port);
>  // 输出会看到 leave parse_port_inner / leave parse_port，证明抛异常时栈对象被正常析构
>}
>```

>[!quote]- 通俗易懂的理解
>- 你可以把异常当成“故障专用通道”：代码一旦进入这个通道，程序会自动回退并清理局部资源。
>- 最容易混淆的点：
>  - “异常”不是普通 `if/else`，它不适合表达常态分支。
>  - “能抛”不等于“该抛”，关键在边界设计。
>- 一个好记的三层原则：
>  - **底层库层**：抛出具体技术异常。
>  - **模块边界层**：翻译异常类型，统一语义。
>  - **接口层**：收敛成稳定契约（异常或错误码二选一）。
>- 面试官真正看的是：你是否能保证“失败时不泄漏、状态可控、边界一致”。

>[!example]- 面试回答简版模板
>`C++ 异常是 throw-栈展开-catch 机制。我的实践是：异常仅用于不可恢复路径；遵循 throw by value / catch by const reference；析构和关键移动操作保持 noexcept；用 RAII 保证栈展开时资源自动回收；在模块边界做异常翻译并统一收敛到稳定接口。`
