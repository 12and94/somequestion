- 日期：2026-03-25
- 题目：throw by value, catch by const reference 什么意思？
- 补充知识点：
  - 这句话的含义是：
    - 抛异常时，用“值”抛：
      - `throw MyError("msg");`
    - 捕获异常时，用“`const` 引用”接：
      - `catch (const MyError& e)`
  - 为什么这样做：
    - 避免对象切片（slicing）：按值捕获基类时，派生类信息可能丢失。
    - 避免不必要拷贝：按值捕获会额外拷贝异常对象；按引用不会。
    - `const` 保证只读语义：通常只读取 `what()` 等信息，不修改异常对象。

- 对比例子：
```cpp
struct Base : std::exception {
  const char* what() const noexcept override { return "Base"; }
};
struct Derived : Base {
  const char* what() const noexcept override { return "Derived"; }
};

try {
  throw Derived{}; // throw by value
}
// 不推荐：按值捕获基类，可能发生切片
catch (Base e) {
  // e 很可能只剩 Base 视图
}

// 推荐：按 const 引用捕获
catch (const Base& e) {
  // 保留动态类型信息，what() 可正确多态分发
}
```

- 通俗解释：
  - `throw` 像“把完整错误包裹扔出去”；`catch const&` 像“只读地接住原包裹，不拆不复制”。

- 面试可复述模板：
  - `异常处理建议 throw by value, catch by const reference。这样既避免切片，又避免额外拷贝，还能保留动态类型信息。`

- 日期：2026-03-25
- 题目：对象切片是什么？
- 补充知识点：
  - 对象切片（object slicing）是指：
    - 把派生类对象按“值”赋给基类对象时，派生类那部分数据被切掉，只剩基类子对象。
  - 发生后常见结果：
    - 基类成员还在；
    - 派生类新增成员丢失；
    - 当对象已经变成独立基类对象时，调用表现也按基类对象语义执行。

- 最小示例：
```cpp
struct Base {
  int b = 1;
  virtual const char* name() const { return "Base"; }
};

struct Derived : Base {
  int d = 2;
  const char* name() const override { return "Derived"; }
};

Derived der;
Base base = der; // 发生切片：只拷贝 Base 那一部分
```

- 与引用/指针对比：
```cpp
Base& rb = der;   // 不切片
Base* pb = &der;  // 不切片
```
  - 引用/指针只是“指向同一对象”，不复制对象本体，所以不会切片。

- 在异常中的对应关系：
  - `catch (Base e)`：按值捕获，可能切片（不推荐）。
  - `catch (const Base& e)`：按引用捕获，不切片（推荐）。

- 通俗解释：
  - 切片就是“把大对象装进小盒子”，装不下的派生部分会被直接截掉。

- 面试可复述模板：
  - `对象切片是派生对象按值转成基类对象时丢失派生部分。规避方式是用基类引用/指针，异常里则用 catch(const Base&)。`

- 日期：2026-03-25
- 题目：noexcept 部分没看懂
- 补充知识点：
  - `noexcept` 是函数对外的异常契约：函数承诺“不把异常抛出函数边界”。
  - 语义上它不只是提示：如果异常从 `noexcept` 函数逃逸，程序会直接 `std::terminate()`。
  - 这意味着 `noexcept` 不是“建议”，而是“违反即终止”的硬约束。
  - 常见误区：
    - 误区1：`noexcept` 里完全不能写会抛异常的代码。
      - 更准确：可以调用可抛代码，但必须在函数内部 `try/catch` 收敛，不能让异常逃出边界。
    - 误区2：`noexcept` 只是为了性能。
      - 更准确：它会改变错误传播行为（逃逸即 terminate），同时才可能带来优化收益。
  - 典型使用位置：
    - 析构函数（通常应不抛）。
    - 移动构造/移动赋值（容器会据此选择更优策略）。
    - 你能明确保证不抛的底层函数。

- 对比例子：
```cpp
#include <iostream>
#include <stdexcept>

void g() { // 没有 noexcept
  throw std::runtime_error("x");
}

void h() noexcept { // 承诺不抛
  throw std::runtime_error("y"); // 异常一旦逃出，terminate
}

int main() {
  try {
    g();
  } catch (...) {
    std::cout << "caught g\n"; // 能执行
  }

  try {
    h();                         // 这里会 terminate，下面 catch 通常到不了
  } catch (...) {
    std::cout << "caught h\n";
  }
}
```

- 边界收敛示例：
```cpp
bool init() noexcept {
  try {
    may_throw();
    return true;
  } catch (...) {
    return false; // 或记录日志并转错误码
  }
}
```

- 通俗解释：
  - `noexcept` 就像“本函数对外保证不冒泡异常”。
  - 你可以在函数内部处理异常，但不能让它越界跑出去。

- 面试可复述模板：
  - `noexcept 是异常契约，不是普通注释。它保证异常不越过函数边界；若异常逃逸会直接 terminate。工程上常用于析构、移动操作和边界收敛函数。`

- 日期：2026-03-25
- 题目：为什么析构函数、移动构造/移动赋值、低层关键路径这些不要抛异常
- 补充知识点：
  - 共同原因：这些位置一旦异常“逃出边界”，通常没有安全恢复路径，结果可能是 `std::terminate`、状态破坏或性能退化。
  - 析构函数：
    - 析构常发生在栈展开期间（已经有异常在传播）。
    - 若析构再抛新异常，运行时通常直接 `std::terminate`。
    - 因此析构职责应是“尽力清理资源，不再抛出”。
  - 移动构造/移动赋值：
    - 标准容器（如 `vector`）扩容/重排时依赖移动语义。
    - 若移动是 `noexcept`，容器可优先走移动路径。
    - 若移动可能抛，容器常退回拷贝路径以维持强异常安全，带来性能损失和语义复杂度。
  - 低层关键路径（内存、锁、清理、边界层）：
    - 被高频调用，异常扩散成本高。
    - 常处于“必须完成清理/释放”的阶段，不适合再引入异常控制流。
    - 跨模块/跨语言边界传播异常不稳定，通常应收敛为错误码或稳定契约。
  - 关键结论：
    - 不是“函数内部绝对不能抛”，而是“不能让异常逃逸出这类函数边界”。

- 边界收敛示例：
```cpp
bool close_resource() noexcept {
  try {
    may_throw_cleanup();
    return true;
  } catch (...) {
    log_error();
    return false;
  }
}
```

- 通俗解释：
  - 这些函数像系统“基础设施层”，核心目标是可预测与稳态，不是把错误继续往外冒泡。

- 面试可复述模板：
  - `这几类函数要强调可预测性：析构抛异常会在栈展开时触发 terminate；移动操作不 noexcept 会让容器退化到拷贝路径；低层关键路径应在边界收敛错误，避免异常跨层扩散。`
