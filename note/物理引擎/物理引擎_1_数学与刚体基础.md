# 物理引擎 1：数学与刚体基础

## 1. 线性代数与空间表示
### 1.1 向量、点、方向
- 点表示位置，向量表示位移/方向；点和向量在物理意义上不同。
- 两点相减得到向量；点加向量得到新点。
- 面试重点：你是否能解释点和法线为何不能同样用变换矩阵处理（尤其非均匀缩放）。

### 1.2 矩阵与坐标变换
- 物理与渲染都会用到局部空间/世界空间切换。
- 刚体姿态常由旋转矩阵或四元数表示。
- 常见坑：行向量/列向量约定混乱导致乘法顺序错误。

### 1.3 四元数与旋转
- **一句话先记住**
- 四元数是“用 4 个数表示旋转”的方法，本质是旋转算子，不是点坐标。

- **四元数是什么**
- 常写成 `q = (w, x, y, z)`，也可记成 `q = (w, v)`（`v` 是三维向量部分）。
- 在物理引擎里，表示旋转时通常要求 `q` 是单位四元数（长度为 1）。
- 轴角到四元数（面试高频）：
  - 已知单位轴 `n=(nx,ny,nz)` 和旋转角 `theta`
  - `w = cos(theta/2)`
  - `x = nx * sin(theta/2)`
  - `y = ny * sin(theta/2)`
  - `z = nz * sin(theta/2)`

- **为什么不用欧拉角，常用四元数**
- 欧拉角在某些姿态会出现万向节锁（gimbal lock）。
- 连续旋转累积时，四元数通常更稳定。
- 插值更自然（slerp/nlerp），适合动画与物理过渡。

- **如何用四元数旋转向量**
- 把向量 `p` 写成纯虚四元数 `p_q=(0, p)`。
- 旋转结果：`p' = q * p_q * conjugate(q)`。
- 注意：如果 `q` 不是单位四元数，结果会混入缩放误差。

- **刚体里怎么更新旋转（工程常用）**
- 已知角速度 `omega` 和时间步 `dt`：
- 1) 计算角位移：`angle = |omega| * dt`
- 2) 构造增量四元数：`dq = quat(axis=normalize(omega), angle=angle)`
- 3) 更新姿态：`q_new = normalize(dq * q_old)`（左右乘取决于约定）
- 每帧归一化很关键，防止长时间漂移。

- **小白理解（一步步）**
- 1) 你可以把四元数想象成“旋转指令包”，不是物体位置。
- 2) 这个指令包说的是：绕哪根轴，转多少角。
- 3) 物体每帧都会收到一个“很小的旋转指令”（由角速度产生）。
- 4) 把这些小指令不断叠加，就得到连续旋转。
- 5) 叠加时要定期“校准长度=1”，否则会越算越歪。

- **小白数字例子（绕 Y 轴转 90°）**
- 轴：`n=(0,1,0)`，角度：`theta=90°=pi/2`
- `theta/2 = pi/4`
- `cos(pi/4)=0.7071`，`sin(pi/4)=0.7071`
- 所以四元数约为：`q=(0.7071, 0, 0.7071, 0)`
- 这个四元数表示“绕 Y 轴转 90°”的旋转。

- **四元数示例（C++）**
```cpp
#include <cmath>

struct Vec3 {
    float x, y, z;
};

struct Quat {
    float w, x, y, z;
};

static inline Quat Normalize(const Quat& q) {
    float n = std::sqrt(q.w*q.w + q.x*q.x + q.y*q.y + q.z*q.z);
    return {q.w/n, q.x/n, q.y/n, q.z/n};
}

static inline Quat Conjugate(const Quat& q) {
    return {q.w, -q.x, -q.y, -q.z};
}

static inline Quat Mul(const Quat& a, const Quat& b) {
    return {
        a.w*b.w - a.x*b.x - a.y*b.y - a.z*b.z,
        a.w*b.x + a.x*b.w + a.y*b.z - a.z*b.y,
        a.w*b.y - a.x*b.z + a.y*b.w + a.z*b.x,
        a.w*b.z + a.x*b.y - a.y*b.x + a.z*b.w
    };
}

static inline Quat FromAxisAngle(const Vec3& axis, float rad) {
    float h = 0.5f * rad;
    float s = std::sin(h);
    return Normalize({std::cos(h), axis.x * s, axis.y * s, axis.z * s});
}

static inline Vec3 RotateVec3(const Quat& qUnit, const Vec3& v) {
    Quat p{0.0f, v.x, v.y, v.z};
    Quat r = Mul(Mul(qUnit, p), Conjugate(qUnit));
    return {r.x, r.y, r.z};
}

// 示例：绕 Y 轴旋转 90 度
// Quat q = FromAxisAngle({0,1,0}, 3.1415926f * 0.5f);
// Vec3 out = RotateVec3(q, {1,0,0});
```

- **常见坑（非常容易被问）**
- 1) 每帧不归一化：长时间后姿态漂移。
- 2) 乘法顺序写反：旋转结果完全不同。
- 3) 世界角速度和局部角速度混用：更新公式出错。
- 4) 欧拉角与四元数频繁来回转换：容易引入数值噪声。

- **面试可复述模板**
- 四元数是单位旋转算子，常用于刚体姿态表示与更新。它能避免欧拉角万向节锁，组合旋转更稳定。工程中通常用角速度构造增量四元数更新姿态，并在每步归一化，避免漂移。
## 2. 力学基础
### 2.1 线性运动
- 牛顿第二定律：F = m a。
- 状态更新核心：
  - 速度由加速度积分得到。
  - 位置由速度积分得到。
- 物理引擎常用固定时间步保证稳定。

### 2.2 角运动
- 角运动可以看成线性运动的旋转版本：
- 线性里是 `F -> a -> v -> x`。
- 角运动里是 `τ(扭矩) -> α(角加速度) -> ω(角速度) -> 姿态`。

- 核心量与关系：
- 扭矩：`τ = r × F`（力臂叉乘力）。
- 角动量：`L = I * ω`。
- 角加速度（简化理解）：`α = I^{-1} * τ`。
- 更完整刚体形式常写为：`α = I^{-1} * (τ - ω × (Iω))`（3D 刚体高速旋转时更准确）。

- 惯性张量为什么重要：
- 质量决定“线性难不难推”，惯性张量决定“旋转难不难拧”。
- 同一个物体绕不同轴旋转，难度不同，所以 `I` 不是单个标量而是矩阵。

- 世界空间与局部空间惯量：
- 刚体局部惯量常是常量：`I_body`。
- 实际求解通常要转到世界空间：`I_world = R * I_body * R^T`。
- 这也是 3D 角运动比 2D 难很多的关键点。

- 每帧更新（常见离散形式）：
- `ω_{t+dt} = ω_t + α * dt`
- 姿态由 `ω` 积分更新四元数（见 1.3），并做归一化。

- 工程常见附加项：
- 角阻尼（angular damping）抑制高频抖动。
- 角速度上限（clamp）防止数值爆炸。

- 最小示例（扭矩与角速度更新）
```cpp
// 输入：世界空间 force, 作用点 point, 质心 center
Vec3 r = point - center;
Vec3 torque = Cross(r, force);          // τ = r x F
Vec3 alpha  = IworldInv * torque;       // 简化: α = I^-1 τ
omega += alpha * dt;                    // 更新角速度

// 后续：用 omega 更新四元数姿态，再归一化
```

- 小白理解（一步步）：
- 1) 你推门时，如果推在门把手上（离铰链远），门更容易转；这就是“力臂”作用。
- 2) 同样大小的力，推在铰链附近（力臂短）就不容易转，扭矩更小。
- 3) 物体“转得快不快”不仅看力，还看它“抗转动能力”（惯性张量）。
- 4) 所以角运动不是“给个力就转”，而是“力 + 施力位置 + 物体形状质量分布”共同决定。

- 小白数字例子：
- 假设同一个力 `F=10N`，
- 作用点 A 距质心 `r=0.2m`，作用点 B 距质心 `r=1.0m`（方向合适时）。
- 近似扭矩大小可理解为 `τ≈r*F`：
- A 点 `τ≈2`，B 点 `τ≈10`，B 明显更容易把物体转起来。

- 常见坑：
- 忽略 `I_world` 更新，导致旋转方向/速度异常。
- 角速度积分后不归一化姿态四元数，长期漂移。
- 把局部空间和世界空间向量混算。
### 2.3 冲量（Impulse）
- 冲量是“在极短时间内累积的力”，定义：`J = ∫F dt`。
- 它的工程意义是：不需要显式模拟极短碰撞力过程，直接给速度一个瞬时修正。

- 与力的区别：
- 力（Force）：持续作用，通常通过积分逐步改变速度。
- 冲量（Impulse）：瞬时作用，直接改变速度。

- 线速度修正：
- `v' = v + J * invMass`

- 角速度修正：
- `ω' = ω + I^{-1} * (r × J)`
- 其中 `r` 是接触点到质心的向量。

- 碰撞里为什么常用冲量：
- 碰撞时间很短，直接用冲量可高效近似接触响应。
- 实时引擎里接触与约束迭代求解基本都在“冲量空间”完成。

- 法向冲量（简化公式）常见形式：
- `j = -(1+e) * v_rel_n / K`
- 其中：
- `e` 是反弹系数（restitution）。
- `v_rel_n` 是相对速度在法线方向分量。
- `K` 是有效质量项（包含线性与角向贡献）。

- 最小示例（双刚体法向冲量）
```cpp
float j = ComputeNormalImpulse(bodyA, bodyB, contact);
Vec3 J = j * contact.normal;

bodyA.v -= J * bodyA.invMass;
bodyB.v += J * bodyB.invMass;

bodyA.w -= bodyA.IworldInv * Cross(ra, J);
bodyB.w += bodyB.IworldInv * Cross(rb, J);
```

- 小白理解（一步步）：
- 1) 力像“持续推车”，车会慢慢加速。
- 2) 冲量像“啪地拍一下球”，速度会立刻改变。
- 3) 碰撞发生非常快（一个很短瞬间），直接用冲量描述最合适。
- 4) 所以游戏里碰撞求解不会去细抠这 0.000x 秒内每个细节，而是一次性修正速度。

- 小白数字例子：
- 质量 `m=2kg` 的球，原速度 `v=1m/s`。
- 给它一个冲量 `J=4N·s`，则 `Δv = J/m = 2m/s`。
- 新速度约 `v'=3m/s`（方向按冲量方向）。

- 常见坑：
- 只修正线速度不修正角速度，接触会不真实。
- 反弹和摩擦同一帧处理顺序不当，导致能量异常。
- 质量比差异大但迭代次数太少，接触抖动明显。
## 3. 数值积分与稳定性
### 3.1 Explicit Euler
- 简单，但稳定性差，能量容易发散。

### 3.2 Semi-Implicit Euler（游戏常用）
- **定义（先记公式）**
- Semi-Implicit Euler（半隐式欧拉 / Symplectic Euler）常见线性更新：
- 1) `v_{t+dt} = v_t + a_t * dt`
- 2) `x_{t+dt} = x_t + v_{t+dt} * dt`
- 关键点：位置更新使用“新速度”而不是旧速度。

- **和 Explicit Euler 的区别**
- Explicit Euler：
- `v_{t+dt} = v_t + a_t * dt`
- `x_{t+dt} = x_t + v_t * dt`（位置用旧速度）
- Semi-Implicit Euler：
- `x` 更新时用 `v_{t+dt}`，所以在很多动力学系统中稳定性更好。

- **为什么游戏里常用它**
- 相比显式欧拉，长期能量发散通常更可控。
- 比更高阶积分器（如 RK4）成本低，适合实时帧预算。
- 在“稳定性/性能”上是常见工程折中。

- **小白理解（一步步）**
- 1) 先根据受力把“速度表”更新到最新。
- 2) 再用这个最新速度去走位置。
- 3) 这样比“拿旧速度走位置”更接近真实过程，长期更不容易炸。

- **小白数字例子**
- 假设初始：`x=0, v=10, a=-9.8, dt=0.1`。
- Semi-Implicit：
- `v_new = 10 + (-9.8)*0.1 = 9.02`
- `x_new = 0 + 9.02*0.1 = 0.902`
- Explicit：
- `x_new = 0 + 10*0.1 = 1.0`
- 两者都是近似，但 Semi-Implicit 在物理系统里通常更稳。

- **线性与角运动伪代码**
```cpp
// 线性部分
v += a * dt;
x += v * dt; // 注意是新v

// 角运动部分（简化）
omega += alpha * dt;
q = Normalize(IntegrateQuaternion(q, omega, dt));
```

- **常见坑**
- 1) 即使用 Semi-Implicit，也不要直接用大幅波动的可变 dt。
- 2) 约束很硬但迭代次数太少，仍会抖动。
- 3) 角速度更新姿态后不归一化四元数，会漂移。

- **面试可复述模板**
- Semi-Implicit Euler 是“先更新速度、再用新速度更新位置”的积分法。它比显式欧拉在实时物理里通常更稳定，同时计算成本低，因此是游戏引擎常见的基础积分方案。
### 3.3 Fixed Timestep
- Fixed Timestep 是指“物理以固定 dt 更新”，渲染帧率可变。
- 目标是稳定、可复现、可调参，而不是单纯追求每帧只更新一次。

- 为什么固定步长更稳：
- 数值积分误差与 dt 强相关，可变 dt 会让同一输入在不同帧率下出现不同结果。
- 固定 dt 能让碰撞阈值、约束参数、阻尼参数更可控。

- 经典累积器（accumulator）流程：
- 每帧把 `frameDelta` 累加到 `accumulator`。
- 只要 `accumulator >= fixedDt` 就执行一次 `PhysicsStep(fixedDt)`。
- 渲染阶段可用插值让画面平滑。

- 最小示例（推荐写法）
```cpp
const double fixedDt = 1.0 / 60.0;
const int maxSubSteps = 4;

accumulator += Clamp(frameDelta, 0.0, 0.25); // 防止极端跳变

int steps = 0;
while (accumulator >= fixedDt && steps < maxSubSteps) {
    PhysicsStep(fixedDt);
    accumulator -= fixedDt;
    ++steps;
}

// 可选：渲染插值 alpha
double alpha = accumulator / fixedDt;
RenderInterpolate(alpha);
```

- “死亡螺旋”（spiral of death）是什么：
- 某帧过慢 -> 需要补很多物理步 -> 更慢 -> 再补更多，系统持续恶化。
- 常见防护：
- 限制每帧最大补步数（`maxSubSteps`）。
- 限制 `frameDelta` 上限（clamp）。

- dt 如何选：
- 1/60：成本更低，稳定性一般够用。
- 1/120：接触更细、穿透更少，但 CPU 成本更高。
- 实务按平台预算选择，并用压测验证稳定性。

- 小白理解（一步步）：
- 1) 物理像按节拍器走路，节拍固定就稳定。
- 2) 如果节拍忽快忽慢（可变 dt），同样动作每次结果都可能不一样。
- 3) 固定步长让“同样输入 -> 近似同样结果”，调参也更靠谱。

- 小白数字例子：
- 假设某一秒：
- 机器A用 60 帧（每帧约 16.7ms），机器B用 30 帧（每帧约 33.3ms）。
- 若直接用可变 dt 积分，A/B 误差积累路径不同，结果可能出现位置偏差。
- 若固定物理步长（如统一 1/60），A/B 只是在渲染频率不同，物理轨迹更一致。

- 常见坑：
- 用可变 dt 直接驱动物理，导致帧率依赖。
- 不做补步上限，卡顿时触发死亡螺旋。
- 物理和渲染都用同一状态不插值，视觉抖动明显。
## 4. 刚体数据结构与状态管理
### 4.1 刚体最小状态
- **为什么要定义“最小状态”**
- 物理引擎每个固定步都会更新刚体，最小状态就是“能推进刚体运动并参与碰撞求解”所必需的字段集合。
- 字段太少会算不完整，字段太多会拖慢性能和占内存。

- **核心字段与用途（必会）**
- 1) `position`（位置）
- 表示质心在世界空间的位置。
- 2) `orientation`（姿态，常用四元数）
- 表示物体朝向，决定局部坐标轴在世界中的方向。
- 3) `linearVelocity`（线速度）
- 决定位置每步怎么变化。
- 4) `angularVelocity`（角速度）
- 决定姿态每步怎么变化。
- 5) `mass` / `invMass`（质量与逆质量）
- 质量决定“推起来有多难”。
- 工程里常存 `invMass` 是因为求解器中常用乘法，避免频繁除法。
- 6) `inertiaTensorBody` / `invInertiaTensorBody`（局部惯量与逆惯量）
- 决定“转起来有多难”，是旋转版“质量”。
- 常在局部空间存常量，运行时转到世界空间参与求解。
- 7) （常见附加但非常实用）
- `forceAccum` / `torqueAccum`：本步累积外力和扭矩。
- `sleepFlag` / `isAwake`：休眠控制，减少无效计算。

- **为什么常同时存“值”和“逆值”**
- 求解器里大量公式是 `delta = something * invMass`。
- 若每次都做 `1/mass` 或矩阵求逆，开销高且不稳定。
- 所以初始化时预计算逆值，热路径只做乘法。

- **最小结构示例（简化）**
```cpp
struct RigidBody {
    Vec3  position;              // 质心位置
    Quat  orientation;           // 姿态
    Vec3  linearVelocity;        // 线速度
    Vec3  angularVelocity;       // 角速度

    float mass;                  // 质量
    float invMass;               // 逆质量

    Mat3  inertiaBody;           // 局部惯量张量
    Mat3  invInertiaBody;        // 局部逆惯量

    Vec3  forceAccum;            // 力累积
    Vec3  torqueAccum;           // 扭矩累积

    bool  isAwake;               // 休眠状态
};
```

- **小白理解（一步步）**
- 你可以把刚体看成“会动的盒子”，最少要知道：
- 它在哪（position）、朝哪（orientation）、移动多快（v）、转多快（ω）、有多重（mass）、有多难转（inertia）。
- 这些量合起来，才足够让引擎每步算出“下一刻在哪里、朝哪里”。

- **常见面试坑**
- 只存质量不存逆质量，求解热路径性能差。
- 只考虑线速度，不考虑角速度，碰撞看起来“像滑块而不是刚体”。
- 姿态不用单位四元数维护，长期漂移。
### 4.2 Static / Dynamic / Kinematic
- **三类刚体先给定义**
- 1) `Static`（静态体）
- 不动，通常不积分，不受力影响；常用于地面、墙体、建筑。
- 2) `Dynamic`（动态体）
- 完整参与物理：受力、重力、碰撞、约束、求解器迭代。
- 3) `Kinematic`（运动学体）
- 由代码/动画直接驱动位姿，不受外力积分反推；但可参与碰撞检测。

- **它们在引擎里的典型行为差异**
- Static：
- 可与 Dynamic 碰撞，通常不会被推动。
- Dynamic：
- 与 Static / Dynamic /（多数引擎中）Kinematic 产生接触与响应。
- Kinematic：
- 自己按“目标位姿”移动，常用于角色控制器、电梯、机关门。
- 与 Dynamic 接触时，Dynamic 会被推开；Kinematic 通常不被反推。

- **为什么要分这三类（工程原因）**
- 性能：Static 不需要每步积分，成本低。
- 可控性：Kinematic 适合“必须按设计轨迹移动”的对象。
- 真实性：Dynamic 用于可交互、可受力物体。

- **常见使用策略**
- 地图静态几何 -> Static。
- 箱子、球、可破坏碎片 -> Dynamic。
- 主角胶囊体（很多项目）/平台机关 -> Kinematic（或混合方案）。

- **小白理解（类比）**
- Static：像“钉在地上的墙”。
- Dynamic：像“踢一下会滚的足球”。
- Kinematic：像“按脚本移动的电梯”，它自己照时间表走，不因为被撞就偏航。

- **行为对比表（面试好用）**

| 类型 | 受力积分 | 被碰撞推动 | 可主动移动 | 常见场景 |
|---|---|---|---|---|
| Static | 否 | 否 | 通常否 | 地面/墙体/场景网格 |
| Dynamic | 是 | 是 | 可施力推动 | 箱子/球/碎片 |
| Kinematic | 否（通常） | 通常否 | 是（代码设位姿） | 角色控制/机关平台 |

- **常见坑（高频）**
- 把角色做成纯 Dynamic，手感和可控性难稳定。
- Kinematic 直接“瞬移”穿过物体，未做 sweep/shape cast 导致穿透。
- 把大量可不动对象设成 Dynamic，浪费性能预算。

- **面试可复述模板**
- Static 用于不动环境，Dynamic 用于受力交互对象，Kinematic 用于脚本/输入驱动且强调可控的对象。三者划分是物理真实性、控制需求和性能预算之间的工程取舍。
### 4.3 Sleeping/Wake
- 睡眠机制用于减少无效计算。
- 常见策略：速度低于阈值持续若干帧进入 sleep；受碰撞/外力唤醒。
