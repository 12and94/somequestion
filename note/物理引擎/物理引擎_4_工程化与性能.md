# 物理引擎 4：工程化与性能

## 1. 主循环与时间管理
### 1.1 FixedUpdate 设计
- 渲染帧率可变，物理步长固定（例如 60Hz / 120Hz）。
- 典型做法：累积时间 + 多次物理步 + 渲染插值。

### 1.2 Spiral of Death 防护
- **先解释两个核心词**
- `物理积压`：
- 某一帧太慢，导致“本应执行的固定物理步”没来得及执行，欠下了更新债务。
- 这些没执行的固定步会留在 accumulator（时间累积器）里，等待后续帧补上。
- `补帧（补物理步）`：
- 后续帧通过 `while(accumulator >= fixedDt)` 多跑几次 `PhysicsStep(fixedDt)`，把欠下的物理步补回来。

- **Spiral of Death 是怎么发生的**
- 1) 某帧卡顿（例如 40ms） -> accumulator 突然变大。
- 2) 下一帧为了追进度需要补很多物理步（例如 2~3 步）。
- 3) 多跑物理步让下一帧更慢。
- 4) 更慢又导致新的积压，继续补更多步。
- 5) 系统进入“越补越慢”的恶性循环，这就是 Death Spiral（死亡螺旋）。

- **为什么这是实时系统的大问题**
- 帧时间会持续恶化，出现卡顿尖峰。
- 玩家体感是“突然一顿一顿”，比平均帧率下降更难受。
- 即使 CPU 理论性能够，偶发抖动也可能触发连锁反应。

- **核心防护策略（工程必做）**
- 1) 限制每帧最大补步数 `maxSubSteps`：
- 超过上限就停止补，优先保住当前帧时间上限。
- 2) 限制输入帧时长 `frameDelta`（clamp）：
- 防止单次超长卡顿把 accumulator 瞬间冲爆。
- 3) 做降级策略：
- 当物理压力超预算时，优先降低非关键对象频率或精度。
- 4) 监控指标：
- 实时记录 accumulator 深度、每帧补步数、物理耗时。

- **典型伪代码（可直接用来讲）**
```cpp
const double fixedDt = 1.0 / 60.0;
const int maxSubSteps = 4;

// 1) 限制单帧输入，防止一次性积压过大
frameDelta = Clamp(frameDelta, 0.0, 0.25);
accumulator += frameDelta;

int subSteps = 0;
while (accumulator >= fixedDt && subSteps < maxSubSteps) {
    PhysicsStep(fixedDt);
    accumulator -= fixedDt;
    ++subSteps;
}

// 2) 若还有积压，允许“欠一点账”而不是把当前帧拖死
if (subSteps == maxSubSteps && accumulator >= fixedDt) {
    // 可选：打点告警 / 降级策略 / 丢弃部分远端物理更新
}
```

- **小白理解（通俗）**
- 物理更新像“每天要完成的作业”。
- 某天没做完（积压），第二天要补作业（补帧）。
- 如果第二天还要上新课，补作业又太多，就会一直跟不上（死亡螺旋）。
- 防护就是“每天最多补几页，先保证不彻底崩盘”。

- **面试可复述模板**
- Spiral of Death 是固定步长物理中的积压连锁问题：卡顿导致 accumulator 增大，后续帧补步过多又进一步拉长帧时间，形成恶性循环。常见防护是限制 maxSubSteps、clamp frameDelta，并配合物理降级策略与实时监控指标。

## 2. 数据结构与内存
### 2.1 SoA 优于 AoS 的场景
- **先定义两个布局**
- `AoS（Array of Structs）`：
- 数据按“对象”连续存储。例：`[Body0{pos,vel,mass}, Body1{...}, ...]`。
- `SoA（Struct of Arrays）`：
- 数据按“字段”连续存储。例：`pos[]`, `vel[]`, `mass[]` 分开存。

- **为什么 SoA 在很多物理热路径更快**
- 1) 缓存友好：
- 当你只处理某个字段（如只更新速度），SoA 只拉取需要的数据，不会把无关字段一起搬进 cache。
- 2) SIMD 友好：
- `vel[]` 连续内存更容易做向量化（一次处理 4/8 个元素）。
- 3) 预取效率更高：
- 连续数组更符合 CPU 预取器习惯。

- **典型“ SoA 明显占优”的场景**
- 批量积分：一次循环更新几千个刚体的 `v/x`。
- broad phase 中 AABB 批处理：只读 min/max 数组。
- 休眠判定：只扫描速度阈值。
- 碰撞预筛：只用位置/包围体，不用完整对象字段。

- **AoS 何时仍有价值**
- 对象级逻辑分散、随机访问强时，AoS 代码可读性更好。
- 调试/序列化/脚本绑定时，AoS 更直观。
- 小规模对象（数量很少）时，SoA 的收益可能不明显。

- **实战建议：混合布局而不是极端二选一**
- 热路径（批量数值计算）用 SoA。
- 管理层（生命周期、事件、编辑器）保留 AoS 视图。
- 常见做法：
- 逻辑层 AoS（易维护）
- 求解器镜像 SoA（高性能）

- **小白理解（通俗）**
- AoS 像“每个人一个文件夹，里边啥都有”。
- SoA 像“全体人的同一类信息放一摞”。
- 如果今天只查“所有人的身高”，SoA 一摞翻到底最快；AoS 需要每个文件夹都翻一遍。

- **最小示例（对比）**
```cpp
// AoS
struct Body { Vec3 pos; Vec3 vel; float invMass; int state; };
std::vector<Body> bodies;
for (auto& b : bodies) {
    b.vel += gravity * dt; // 访问vel时也会带入pos/state等邻近字段
}

// SoA
struct BodiesSoA {
    std::vector<Vec3> pos;
    std::vector<Vec3> vel;
    std::vector<float> invMass;
    std::vector<int> state;
};
BodiesSoA B;
for (size_t i = 0; i < B.vel.size(); ++i) {
    B.vel[i] += gravity * dt; // 紧凑连续，适合SIMD/cache
}
```

- **面试可复述模板**
- SoA 在物理引擎批量热路径中常优于 AoS，因为它减少无关字段带宽、提升 cache 命中并更利于 SIMD。工程上通常采用“热路径 SoA + 管理层 AoS”的混合方案，以平衡性能和可维护性。

### 2.2 内存池与对象生命周期
- 避免频繁分配释放导致碎片与抖动。
- 热路径尽量无分配。

## 3. 性能预算与降级策略
- 预算项：Broad phase、Narrow phase、求解器、CCD、约束。
- 降级手段：
  - 限制活跃刚体数
  - 限制每帧碰撞对
  - 远处对象降低物理精度
  - 只给关键对象开 CCD

## 4. 多线程与任务图
- 常见拆分：粗检测、窄相、求解、后处理。
- 注意同步点：接触缓存、岛屿构建、写回阶段。

## 5. 调试与可视化
- 必备可视化：碰撞体、接触点、法线、AABB、休眠状态。
- 必备日志：每帧 pair 数、求解迭代次数、耗时分布。
- 推荐能力：录制回放（deterministic replay）快速复现偶现 bug。
