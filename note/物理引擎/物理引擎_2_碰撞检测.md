# 物理引擎 2：碰撞检测

## 1. 碰撞检测分层
### 1.1 Broad Phase（粗检测）
- 目标：快速排除不可能碰撞的对象对。
- 常见结构：
  - Sweep and Prune（轴排序）
  - 动态 AABB 树
  - 网格分桶 / 哈希网格
- 重点：降低 pair 数量，减少后续窄相成本。

### 1.2 Narrow Phase（窄检测）
- 对候选 pair 进行精确求交，输出接触点、法线、穿透深度。
- 形状类型：Sphere/AABB/OBB/Capsule/Convex/Mesh。

### 1.3 为什么 Broad/Narrow 要分层？
- **先给结论**
- 不分层会接近两两全检，复杂度接近 `O(N^2)`，对象一多就跑不动。
- 分层的核心是“用便宜计算过滤大多数无关对，再把昂贵算法留给少量候选”。

- **直观数量级**
- 假设场景有 5000 个物体：
- 两两组合约 1250 万对。
- 若直接用 GJK/SAT 全检，CPU 成本通常不可接受。
- 用 Broad phase 后，候选对可能降到几千或几万，再做 Narrow 才现实。

- **工程价值**
- 提升性能上限：让物体规模可扩展。
- 提升算法可维护性：粗筛和精检职责分离。
- 便于并行：Broad 与 Narrow 都可做任务化批处理。

- **小白理解**
- 先“粗筛简历”再“精面试”。
- 如果不粗筛，相当于让所有人都进终面，面试官（CPU）会直接崩。

- **常见坑**
- Broad phase 太粗导致候选对爆炸，Narrow 仍被拖垮。
- Broad phase 更新不及时导致漏对（漏检）或冗余对（性能抖动）。

## 2. 常见算法
### 2.1 SAT（分离轴定理）
- **适用范围**
- 主要用于凸多边形（2D）与凸多面体（3D）的相交检测。
- 非凸体通常先做凸分解，再用 SAT。

- **核心思想（先记一句）**
- 只要能找到一根“分离轴”，让 A、B 两个物体在这根轴上的投影区间不重叠，则二者一定不碰撞。
- 反过来，如果所有候选轴上投影都重叠，则判定碰撞。

- **实战步骤（2D 最常见）**
- 1) 收集候选轴：通常取两个多边形所有边的法线方向。
- 2) 把两个多边形顶点分别投影到每根轴（点积）。
- 3) 得到两个投影区间 `[minA, maxA]`、`[minB, maxB]`。
- 4) 若存在任意轴满足 `maxA < minB` 或 `maxB < minA`，直接判不碰撞。
- 5) 若全部轴都重叠，则碰撞；最小重叠轴常用作 MTV（最小平移向量）方向参考。

- **小白数字例子（1 根轴）**
- 在某轴上：A 投影 `[1, 3]`，B 投影 `[3.5, 5]`。
- 因为 `3 < 3.5`，区间分离 -> 这根轴已能证明“不碰撞”。

- **优点**
- 原理直观，易调试。
- 2D 凸多边形性能与稳定性都不错。

- **缺点/坑点**
- 3D 候选轴增多，轴集管理复杂（尤其 OBB 需考虑叉积轴）。
- 浮点误差导致“边界抖动”，常需 epsilon。
- 非凸体不能直接套 SAT。

- **最小伪代码**
```cpp
bool IntersectSAT(const Poly& A, const Poly& B) {
    axes = BuildCandidateAxes(A, B);
    for (axis in axes) {
        Interval ia = Project(A, axis);
        Interval ib = Project(B, axis);
        if (ia.max < ib.min || ib.max < ia.min) {
            return false; // 找到分离轴
        }
    }
    return true; // 所有轴都重叠
}
```

### 2.2 GJK
- **适用范围**
- 主要用于凸体（2D/3D）相交检测。
- 非凸体同样需要先凸分解。

- **核心思想（几何本质）**
- 构造 Minkowski Difference：`A - B`。
- 若原点在 `A-B` 内部，则 A 与 B 相交。
- GJK 不显式构造整个差集，而是通过 Support Function 迭代逼近。

- **Support Function 是什么**
- 给定方向 `d`，返回形体在该方向上“最远点”。
- 对差集支持点可写成：
- `support(A-B, d) = support(A, d) - support(B, -d)`。

- **迭代流程（简化）**
- 1) 选初始方向 `d`。
- 2) 取支持点加入 simplex（点/线段/三角形/四面体）。
- 3) 判断 simplex 是否包围原点。
- 4) 若未包围，更新方向继续迭代；若包围则相交。
- 5) 若新支持点在方向上的推进不足（`dot(newPoint,d)<=0`），可判不相交。

- **优点**
- 对凸体通用，维度扩展性好。
- 与 EPA 组合后可得到穿透信息，工程上很常见。

- **缺点/坑点**
- 实现细节多，尤其 simplex 更新逻辑容易写错。
- 退化情况（共线/共面）需要鲁棒处理。
- 结果布尔稳定性依赖 epsilon 与迭代上限设置。

- **最小伪代码（布尔检测）**
```cpp
bool IntersectGJK(const Convex& A, const Convex& B) {
    Vec3 d = InitialDirection(A, B);
    Simplex s;
    s.add(Support(A, B, d));
    d = -s.last();

    for (int iter = 0; iter < kMaxIters; ++iter) {
        Vec3 p = Support(A, B, d);
        if (Dot(p, d) <= 0.0f) return false;
        s.add(p);
        if (SimplexContainsOriginAndUpdateDirection(s, d)) {
            return true;
        }
    }
    return false;
}
```

### 2.3 EPA
- **EPA 解决什么问题**
- GJK 只告诉你“碰了没”。
- EPA 在“已相交”前提下，进一步求：
- 穿透法线（normal）
- 穿透深度（depth）
- 可用于后续碰撞响应（推开方向与修正量）。

- **核心思想**
- 从 GJK 的终止 simplex 出发，扩展为一个包围原点的多面体。
- 每轮找“离原点最近的面”，沿该面法线做 support 扩展。
- 当扩展增量足够小（收敛），该面法线与距离近似就是穿透法线与深度。

- **流程（简化）**
- 1) 输入：GJK 的相交 simplex。
- 2) 构建初始 polytope。
- 3) 找最近面 `face`（距离原点最小）。
- 4) 新点 `p = support(A-B, face.normal)`。
- 5) 若 `dot(p, n) - dist < eps`，收敛并输出。
- 6) 否则把可见面替换，继续迭代。

- **常见坑**
- 面法线朝向不一致导致符号错。
- 多面体拓扑维护复杂，易出现“洞边”连接错误。
- 退化与浮点误差处理不当会卡迭代。

- **最小伪代码**
```cpp
EPAResult SolveEPA(const Convex& A, const Convex& B, const Simplex& gjkSimplex) {
    Polytope poly = BuildFromSimplex(gjkSimplex);
    for (int iter = 0; iter < kMaxIters; ++iter) {
        Face f = ClosestFaceToOrigin(poly);
        Vec3 p = Support(A, B, f.normal);
        float delta = Dot(p, f.normal) - f.distance;
        if (delta < kEps) {
            return {f.normal, f.distance};
        }
        ExpandPolytope(poly, p);
    }
    return FallbackResult(poly);
}
```

### 2.4 SAT 与 GJK/EPA 如何选
- **2D 凸多边形为主、实现追求直观**：SAT 通常更容易落地。
- **3D 凸体统一框架、后续还要穿透信息**：GJK + EPA 常见。
- **工程实践**
- 很多引擎会“按形体类型混用算法”，而不是单算法覆盖所有情形。

- **小白总结**
- SAT：像“找一根尺子，看两物体影子能不能分开”。
- GJK：像“判断原点是不是被差集包住”。
- EPA：像“既然已经包住了，再量出包住多深、往哪推开”。
## 3. 接触流形（Contact Manifold）
- 不只要“碰了没”，还要“接触在哪里、方向是什么、深度多少”。
- 稳定接触需要流形持续性（缓存与 warm start 相关）。

## 4. 连续碰撞检测 CCD
### 4.1 为什么需要 CCD
- 离散检测只看离散时刻，快速运动物体可能穿透（tunneling）。

### 4.2 常见方案
- **A. Sweep Test（扫掠检测）**
- 含义：把物体从“当前位置到下一帧位置”的运动路径看成连续过程，沿路径检测首次碰撞。
- 适合：高速关键对象（子弹、冲刺角色、关键机关）。
- 优点：比纯离散检测更不易穿透。
- 成本：高于普通离散检测。

- **B. TOI（Time of Impact，首次碰撞时刻）**
- 含义：求本步 `dt` 内第一次发生碰撞的时间 `t_hit`。
- 常见流程：
- 1) 先推进到 `t_hit`。
- 2) 处理碰撞响应。
- 3) 再处理剩余时间 `dt - t_hit`。
- Sweep test 常被用来求 TOI。

- **C. 子步进（Substep）**
- 含义：把一个大时间步拆成多个小步，每个小步都做离散碰撞与求解。
- 优点：实现简单，容易接入现有离散框架。
- 缺点：成本近似按子步数线性增长，速度极高时仍可能漏检。

- **怎么选（实战建议）**
- 默认对象：离散检测。
- 高速关键对象：启用 Sweep/TOI。
- 预算允许时：对局部热点场景加子步进。
- 常见做法是混合策略，而不是全场景一刀切。

- **小白理解**
- Sweep/TOI 像“看视频找第一次撞上的时刻”。
- 子步进像“把大步走改成碎步走”，降低一脚跨过障碍的概率。
### 4.3 工程取舍
- CCD 成本高，通常只对高速/关键对象启用。

## 5. 角色系统常见形体
- **最常用：Capsule（胶囊体）**
- 原因 1：没有尖锐边角，贴墙和过门时不容易卡住。
- 原因 2：上下端是圆弧，处理台阶与斜坡更稳定。
- 原因 3：做 sweep（角色移动探测）时鲁棒性和手感通常更好。

- **其他常见形体与对比**
- Sphere：
- 最便宜最稳定，但“站立角色”表现不自然，贴墙/台阶控制较差。
- Box：
- 表达简单，但边角容易卡碰撞，角色手感调参难度较高。
- Cylinder：
- 几何语义接近角色，但许多引擎的原生支持和稳定性不如 Capsule。
- Mesh：
- 精度高但成本高、鲁棒性差，不适合做角色主碰撞体。

- **角色系统实战建议**
- 主碰撞体：Capsule。
- 头/脚/手等细节：用附加查询（射线、球扫、局部体积）补充。
- 不要把角色主碰撞体直接做成复杂网格。

- **小白理解**
- Capsule 像“药丸”，滚动和滑动都顺，不容易被角落“勾住”。
- Box 像“带棱角的纸盒”，拐角处更容易卡。
