## 题目：固定时间步（Fixed Timestep）为什么重要？
>[!NOTE]- 标准准确的说法
>- 物理系统是数值积分系统，对时间步长高度敏感。
>- 固定 dt 能提高稳定性、可复现性与调参一致性。
>- 可变 dt 易导致同一场景在不同帧率下结果漂移。

>[!tip]- 代码示例
>```cpp
>accumulator += frameDelta;
>while (accumulator >= fixedDt) {
>    PhysicsStep(fixedDt);
>    accumulator -= fixedDt;
>}
>```

>[!quote]- 通俗易懂的理解
>- 物理像“按固定拍子走路”，拍子忽快忽慢就容易摔。

>[!example]- 面试回答简版模板
>`固定时间步让物理更新在不同帧率下更稳定和可复现，是实时物理引擎的基础工程约束。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_4_工程化与性能.md`

## 题目：为什么会发生穿透（tunneling）？
>[!NOTE]- 标准准确的说法
>- 离散碰撞检测只检查离散时刻位置。
>- 高速物体可能在两帧之间跨过障碍，导致未检测到接触。
>- 典型解决：CCD（TOI/sweep）或子步进。

>[!tip]- 代码示例
>```cpp
>// 关键对象启用CCD
>if (body.speed > ccdThreshold) {
>    SolveTOI(body);
>}
>```

>[!quote]- 通俗易懂的理解
>- 相机每秒只拍几张，子弹在两张照片之间已经穿过去了。

>[!example]- 面试回答简版模板
>`穿透本质是离散采样漏检。解决思路是对高速关键对象启用连续碰撞检测，而不是全量对象都开 CCD。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_2_碰撞检测.md`

## 题目：Broad Phase 和 Narrow Phase 为什么要分层？
>[!NOTE]- 标准准确的说法
>- 粗检测先快速过滤不可能碰撞对，降低候选规模。
>- 窄相对少量候选做精确求交，输出接触信息。
>- 分层可显著降低总体复杂度。

>[!tip]- 代码示例
>```cpp
>pairs = BroadPhase(worldAABBs);
>for (auto& p : pairs) {
>    contacts += NarrowPhase(p);
>}
>```

>[!quote]- 通俗易懂的理解
>- 先用粗筛把明显无关的人排掉，再做精细面试。

>[!example]- 面试回答简版模板
>`碰撞检测分层是复杂度优化核心：Broad Phase 负责规模，Narrow Phase 负责精度。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_2_碰撞检测.md`

## 题目：GJK 和 EPA 分别解决什么问题？
>[!NOTE]- 标准准确的说法
>- GJK：判断两个凸体是否相交。
>- EPA：在已相交前提下估计穿透深度与分离方向。
>- 二者常配合用于凸体窄相。

>[!tip]- 代码示例
>```cpp
>if (GJKIntersect(a, b)) {
>    mtv = EPA(a, b); // normal + depth
>}
>```

>[!quote]- 通俗易懂的理解
>- GJK 回答“碰没碰”，EPA 回答“碰了多深、往哪推开”。

>[!example]- 面试回答简版模板
>`GJK做相交判定，EPA做穿透信息恢复，是凸体窄相中的常见组合。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_2_碰撞检测.md`

## 题目：冲量（Impulse）和力（Force）有什么区别？
>[!NOTE]- 标准准确的说法
>- 力是持续作用量，影响加速度并经积分改变速度。
>- 冲量是瞬时量，直接改变速度，常用于碰撞响应。

>[!tip]- 代码示例
>```cpp
>v += impulse * invMass;
>```

>[!quote]- 通俗易懂的理解
>- 力像“持续推着走”，冲量像“瞬间拍一下”。

>[!example]- 面试回答简版模板
>`力用于连续动力学，冲量用于离散碰撞校正，二者作用时间尺度不同。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_1_数学与刚体基础.md`

## 题目：为什么约束求解常用迭代法（PGS/Sequential Impulse）？
>[!NOTE]- 标准准确的说法
>- 全局精确求解成本高，不适合实时。
>- 迭代法可在有限预算内给出可接受近似解。
>- 可通过 warm start、迭代次数等手段平衡质量与性能。

>[!tip]- 代码示例
>```cpp
>for (int it=0; it<iterations; ++it) {
>    SolveAllConstraints();
>}
>```

>[!quote]- 通俗易懂的理解
>- 不是一次算到最完美，而是快速多次修正到“够稳够像”。

>[!example]- 面试回答简版模板
>`实时物理更强调预算内稳定收敛，迭代求解是工程上最常见的质量/性能折中。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_3_碰撞响应与约束求解.md`

## 题目：Warm Start 有什么作用？
>[!NOTE]- 标准准确的说法
>- 使用上一帧已收敛的冲量作为初始值。
>- 能加快当前帧收敛，降低抖动。
>- 对持续接触（堆叠、静摩擦）收益明显。

>[!tip]- 代码示例
>```cpp
>constraint.lambda = cachedLambdaFromLastFrame;
>```

>[!quote]- 通俗易懂的理解
>- 上一帧“解过的题”先拿来当参考答案，当前帧少走弯路。

>[!example]- 面试回答简版模板
>`Warm Start 利用时序连续性提升收敛速度与稳定性，是接触求解中的常用技巧。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_3_碰撞响应与约束求解.md`

## 题目：为什么角色控制常用 Kinematic 而不是纯 Dynamic 刚体？
>[!NOTE]- 标准准确的说法
>- 角色控制需要强可控性（输入响应、台阶、坡面、滑墙）。
>- 纯动力学刚体手感与可预测性较难稳定。
>- 常见做法：Kinematic + Sweep + 自定义接地逻辑。

>[!tip]- 代码示例
>```cpp
>desiredPos = currentPos + inputMove;
>actualPos  = SweepAndSlide(currentPos, desiredPos);
>```

>[!quote]- 通俗易懂的理解
>- 主角要“听手柄话”，不能完全交给物理自己漂。

>[!example]- 面试回答简版模板
>`角色系统优先可控与一致性，因此常用 kinematic 方案配合碰撞查询，而不是完全依赖动态刚体求解。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_6_动画联动与特殊模拟.md`

## 题目：多人联机里物理为什么难保持一致？
>[!NOTE]- 标准准确的说法
>- 浮点误差、执行顺序、平台差异都会导致状态分叉。
>- 网络延迟和丢包使同 tick 状态对齐更困难。
>- 工程上多用“预测 + 校正”而非追求绝对一致。

>[!tip]- 代码示例
>```cpp
>PredictLocalInput();
>if (ServerStateArrived()) {
>    ReconcileAndReplay();
>}
>```

>[!quote]- 通俗易懂的理解
>- 两台机器像两个人边看边猜同一场球，细节一定会跑偏，只能不断对表校正。

>[!example]- 面试回答简版模板
>`联机物理难点是确定性与网络时延叠加，常见解法是客户端预测+服务器权威+回滚重演校正。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_5_网络同步与确定性.md`

## 题目：如何定位“偶现穿透”这类疑难物理 bug？
>[!NOTE]- 标准准确的说法
>- 先建立可复现：录制输入/随机种子/关键状态回放。
>- 打开可视化：碰撞体、接触点、法线、TOI 轨迹。
>- 分层排查：Broad phase 丢对？Narrow phase 精度？求解器迭代不足？
>- 最后做回归：加入自动化用例，防止回归。

>[!tip]- 代码示例
>```cpp
>RecordPhysicsFrame(state, contacts, pairs, dt);
>ReplayFromFrame(targetFrame);
>```

>[!quote]- 通俗易懂的理解
>- 先把“事故现场录像”拿到，再逐层追查，不要靠猜。

>[!example]- 面试回答简版模板
>`偶现物理 bug 的核心是先可复现，再分层定位。常用手段是回放系统+可视化+分阶段日志，最终配自动化回归。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_4_工程化与性能.md`

## 题目：移动端物理性能怎么做预算？
>[!NOTE]- 标准准确的说法
>- 明确每帧物理预算（ms）。
>- 控制活跃对象数、碰撞对数、CCD对象比例。
>- 分级更新：远处对象降低频率或简化碰撞。
>- 持续 profile，做场景峰值压测。

>[!tip]- 代码示例
>```cpp
>if (distanceToCamera > LODThreshold) {
>    body.simulationRate = Low;
>    body.collider = SimplifiedShape;
>}
>```

>[!quote]- 通俗易懂的理解
>- 移动端预算像“每帧只有固定零花钱”，必须把钱花在镜头前最关键对象上。

>[!example]- 面试回答简版模板
>`移动端物理预算管理重点是限制活跃规模、关键对象优先、分级模拟与持续压测，不追求所有对象同精度。`

>[!info]- 对应示例
>- `note/物理引擎/物理引擎_4_工程化与性能.md`
