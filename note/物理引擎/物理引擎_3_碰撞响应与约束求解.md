# 物理引擎 3：碰撞响应与约束求解

## 1. 碰撞响应基础
### 1.1 法向冲量
- 用于阻止物体继续互相穿透。
- 依赖相对速度在接触法线方向上的分量。

### 1.2 反弹系数（Restitution）
- 控制“弹性程度”。
- 0 近似不弹，1 接近理想弹性。

### 1.3 摩擦（Friction）
- 切线方向约束，常见库仑摩擦近似。
- 静摩擦与动摩擦在引擎里通常做近似切换或统一约束。

## 2. 约束系统
### 2.1 约束本质
- 用方程限制自由度（位置/速度层面）。
- 常见：距离约束、铰链、点到点、滑轨等。

### 2.2 速度约束与位置校正
- 实时引擎多在速度层做迭代求解，再配合位置纠正减少穿透。

## 3. 迭代求解器（Sequential Impulse / PGS）
### 3.1 为什么用迭代
- **先给结论**
- 物理约束求解本质是一个大规模、稀疏、带不等式约束（接触/摩擦）的系统。
- 在实时游戏里，“全局精确解”通常代价过高且鲁棒性不稳定；迭代解法（PGS/Sequential Impulse）更符合实时预算。

- **为什么全局精确解昂贵（数学层面）**
- 1) 约束规模会随接触点快速增长：
- 角色、箱子、地形、关节共同形成上千到上万约束并不罕见。
- 2) 系统是稀疏且动态变化：
- 每帧接触关系都在变，矩阵结构和有效约束集（active set）持续变化。
- 3) 接触与摩擦是互补/不等式问题：
- 不是简单线性方程组，直接精确求解复杂度和实现难度都高。
- 4) 病态场景多：
- 质量比悬殊、长链约束、深度堆叠会导致条件数恶化，精确法也未必稳定。

- **为什么迭代法更适合实时（工程层面）**
- 1) Anytime 特性：
- 迭代次数越多质量越好；预算紧时可提前停止，保证帧时间。
- 2) 局部修正易并行：
- 按 island 或批次迭代，容易任务化。
- 3) 与 warm start 搭配效果好：
- 连续帧接触相似，复用上一帧冲量可加速收敛。
- 4) 可控可调：
- 通过“迭代次数、误差阈值、偏置项”直接调稳定性与性能平衡。

- **复杂度直觉（便于面试表达）**
- 全局直接法在大规模约束下常出现高阶代价与内存带宽压力。
- 迭代法每轮近似线性于约束数：
- `Cost ≈ O(iterations * constraints)`，更容易做预算管理。

- **小白理解（通俗）**
- 全局精确解像“把全城交通一次性算到绝对最优”，很慢。
- 迭代法像“红绿灯一轮轮调整拥堵路口”，每轮更好一点，且能随时停在可接受结果。

- **小白数字例子**
- 假设一帧有 6000 个约束：
- 迭代 8 轮 -> 约做 `48000` 次局部更新。
- 若预算紧改成 6 轮，质量略降但帧时间更稳。
- 这就是迭代法“可控”的工程价值。

- **最小伪代码**
```cpp
for (int it = 0; it < solverIterations; ++it) {
    for (Constraint& c : constraints) {
        SolveConstraintPGS(c); // 局部修正冲量
    }
}
```

- **常见面试追问与回答点**
- 问：既然是近似，为什么还能用？
- 答：实时系统更重视“帧内稳定收敛 + 可预测耗时”，不是数学最优。
- 问：什么时候要加迭代次数？
- 答：堆叠、质量比悬殊、摩擦主导场景。
- 问：什么时候减迭代？
- 答：移动端或 CPU 紧张场景，配合分级模拟与关键对象优先策略。
### 3.2 常见稳定性技巧
- Warm Start：复用上一帧冲量，加快收敛。
- Baumgarte：位置误差转速度偏置。
- Clamp：限制冲量范围，避免不稳定。
- Split Impulse：降低位置修正引入的能量伪增。

## 4. 岛屿（Island）与批处理
- 把相互关联的刚体/约束分组求解。
- 好处：并行友好，减少无关对象干扰。

## 5. 典型问题与调参方向
- 抖动：迭代次数不足、约束冲突、质量比过大。
- 爆炸：时间步过大、冲量限制不当、约束配置错误。
- 残余穿透：位置修正不足或接触法线不稳定。

## 6. 常见面试追问
- 为什么不做一次性精确求解？
- Warm Start 的作用是什么？
- 大质量比场景怎么稳住？
- 摩擦为什么要在切线方向处理？
