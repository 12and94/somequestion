## 3. PBD算法中的约束是怎么实现的，会推导吗？

### 标准回答
PBD（Position-Based Dynamics）不是先解力再积分位置，而是直接在“位置层”迭代满足约束。

给定约束函数：
- `C(x) = 0`（等式约束）
- 或 `C(x) >= 0`（不等式约束，如防穿透）

设粒子/节点位置为 `x_i`，逆质量为 `w_i`，约束梯度为 `∇_{x_i}C`，
PBD 一次投影修正可写为：

- `Δx_i = - w_i * λ * ∇_{x_i}C`
- `λ = C(x) / (Σ_j w_j ||∇_{x_j}C||^2 + eps)`

组合后：
- `Δx_i = - w_i * C(x) / (Σ_j w_j ||∇_{x_j}C||^2 + eps) * ∇_{x_i}C`

更新位置：
- `x_i <- x_i + Δx_i`

然后对所有约束做 Gauss-Seidel 式迭代（通常多轮）。

### 常见约束例子：距离约束
两点 `x1, x2` 期望距离 `d`：
- `C(x1,x2) = ||x1-x2|| - d = 0`

梯度：
- `∇_{x1}C = (x1-x2)/||x1-x2||`
- `∇_{x2}C = -∇_{x1}C`

代入上式即可得到两点修正量。

### XPBD（面试加分）
XPBD 在 PBD 上加入 compliance（柔顺度）和拉格朗日乘子累计，减少“时间步依赖”和“越迭代越硬”的问题：
- `Δλ = (-C - α~ λ) / (Σ w_i ||∇C_i||^2 + α~)`
- `α~ = α / dt^2`
- `Δx_i = w_i ∇C_i Δλ`
- `λ <- λ + Δλ`

### 通俗理解
PBD 就像“先把物体放到新位置，再把不合理的位置一点点掰正”，
约束就是“尺子规则”（长度不能变、不能穿透、体积要保留），
每轮都往满足规则的方向改一点，迭代几轮后看起来就稳定了。

### 面试一句话
PBD 通过约束投影直接修正位置，核心是按约束梯度分配修正量；XPBD 通过 compliance 和 λ 累计提升物理一致性与时间步鲁棒性。

## 4. 你的蒙皮动画是怎么做的？BVH是指什么？

### 蒙皮动画标准流程（LBS）
1. 导入骨架（joint hierarchy）、绑定姿态（bind pose）、权重（weights）。
2. 每帧根据动画曲线求每个骨骼的全局矩阵 `M_i`。
3. 计算骨骼蒙皮矩阵：
   - `Skin_i = M_i * InvBind_i`
4. 顶点变换（线性混合蒙皮，LBS）：
   - `v' = Σ_i w_i * (Skin_i * v)`

法线一般用对应线性部分或逆转置处理。

### 常见改进
- DQS（Dual Quaternion Skinning）可减轻 LBS 在关节处的“糖纸扭曲”。
- GPU 中通常在 Vertex Shader 做蒙皮；大规模角色可做 GPU Skinning + 动画缓存。

### BVH 是什么
“BVH”在面试里常有两种含义，要先确认语境：

1. **Bounding Volume Hierarchy**（包围体层次结构）
- 渲染/碰撞加速结构，用树组织 AABB/OBB，快速裁剪求交。

2. **Biovision Hierarchy (.bvh)**（动作捕捉文件格式）
- 动画语境下常指 BVH 文件，包含骨架层级 + 每帧关节通道数据。

### 通俗理解
蒙皮就是“骨骼在动，网格顶点按权重跟着几根骨头一起动”。
BVH 要么是“加速树”，要么是“动作文件格式”，看上下文判断。

### 面试一句话
我一般用 LBS（必要时 DQS）做蒙皮：先算骨骼全局矩阵，再乘 inverse bind 得到 skin matrix，顶点按权重线性混合；BVH 在图形里可能指包围体层级，也可能指 mocap 的 Biovision Hierarchy 文件格式。

## 5. 了解 Vulkan 管线吗？layout、descriptor 是什么？

### Vulkan 图形管线（高层）
典型阶段：
- Vertex Input
- Vertex Shader
- (可选 Tessellation / Geometry)
- Rasterization
- Fragment Shader
- Depth/Stencil
- Color Blend

Vulkan 把大部分状态做成显式对象（pipeline object），便于驱动优化和多线程录制命令。

### Pipeline Layout 是什么
`VkPipelineLayout` 定义“shader 能访问哪些资源槽位”的总布局，主要包含：
1. Descriptor Set Layout 列表
2. Push Constant Range 列表

可以理解为：管线与资源绑定的“接口契约”。

### Descriptor 是什么
Descriptor 是 shader 访问资源（buffer/image/sampler）的绑定描述。

常见 descriptor type：
- Uniform Buffer
- Storage Buffer
- Combined Image Sampler
- Storage Image
- Uniform Texel Buffer 等

资源通常按 Descriptor Set 分组，绘制前通过 `vkCmdBindDescriptorSets` 绑定。

### 通俗理解
- `pipeline layout`：先约定“插座长什么样”。
- `descriptor`：把“具体资源（纹理/缓冲）插到插座上”。

### 面试一句话
Vulkan 的 pipeline layout 描述 shader 资源接口（set layout + push constants），descriptor 则是具体资源绑定描述；二者一个定义“接口形状”，一个提供“实际数据”。

## 6. 当两个 kernel 想同时读取一个 buffer，会有什么问题？要怎么做？

### 标准回答
如果两个 kernel 都是**只读同一个 buffer**，通常没有数据竞争（data race）问题。

但要关注三类风险：
1. **带宽竞争**：并发读会争抢内存带宽，可能降性能。
2. **可见性前置条件**：如果该 buffer 之前由 GPU/CPU 写过，需要先保证写入可见。
3. **队列/家族问题**：跨 queue family 时可能需要 ownership transfer。

### 真正危险的情况
- 一个 kernel 写、另一个读（RAW）
- 两个 kernel 同时写（WAW）
- 读写交错（WAR）
这些都需要同步原语与 barrier 保证顺序和可见性。

### 实战做法
- 同一队列只读-只读：通常不需要额外 barrier（前提写入已完成并可见）。
- 写后读：插入 pipeline barrier（或同步2 barrier）标明 stage/access。
- 跨队列：信号/等待 semaphore，必要时做 ownership transfer。

### 通俗理解
“两个读者同时读同一本书”没冲突；
问题出在“有人还在改书内容”，这时必须先同步再读。

### 面试一句话
只读并发通常是安全的，核心关注写后读可见性和队列同步；真正的数据竞争来自并发写或读写交错，需要 barrier/semaphore 正确约束。

## 7. 知道 Vulkan 同步 API 吗？各自有什么区别？

### 常见同步原语与用途
1. **Fence（栅栏）**
- GPU -> CPU 同步。
- CPU 等待某批 GPU 工作完成。

2. **Semaphore（信号量）**
- GPU -> GPU 跨提交同步。
- Binary semaphore：一次信号一次等待。
- Timeline semaphore：单对象递增计数，适合复杂帧图和多队列。

3. **Pipeline Barrier（vkCmdPipelineBarrier / vkCmdPipelineBarrier2）**
- 命令缓冲内部同步，控制执行顺序 + 内存可见性。
- 通过 stage mask + access mask + memory/image/buffer barrier 指明依赖。

4. **Event（VkEvent）**
- 命令流中的轻量同步点（使用场景较少，通常 barrier/semaphore 更主流）。

5. **Subpass Dependency（RenderPass 内）**
- 传统 render pass 子通道间依赖描述。

### 核心区别（面试常问）
- Fence：给 CPU 等。
- Semaphore：给 GPU 提交之间等。
- Barrier：同一命令流内控制“什么时候看见什么数据”。

### 同步2（推荐口径）
- 新接口 `vkCmdPipelineBarrier2` / `VkDependencyInfo` 可读性更强，粒度更清晰，建议新项目优先用同步2。

### 面试一句话
Vulkan 同步分三层：Fence 解决 CPU-GPU，Semaphore 解决提交间 GPU-GPU，Barrier 解决命令流内执行与内存可见性；同步2是更现代、可维护性更高的接口体系。

## 8. GPU 编程题两道（口述思路）

### 8.1 用 compute shader 做碰撞检测后，如何把有效结果紧密存到新 buffer？

#### 推荐思路：`Flag + Prefix Sum + Scatter`（流压缩）
1. 每个线程先写 `flag[i]`：有效=1，无效=0。
2. 对 `flag` 做并行前缀和（exclusive scan），得到每个有效元素的目标下标 `idx`。
3. 若 `flag[i]==1`，把结果写到 `out[idx]`。
4. 有效总数 `count = scan[n-1] + flag[n-1]`。

#### 为什么必须是这三步
- 目标是把“稀疏有效数据”压成“连续紧密数据”，方便后续 pass 只处理有效元素。
- 在并行里，每个线程不知道自己是“第几个有效元素”。
- Prefix Sum 的作用就是并行分配唯一且连续的输出下标。

#### 手算例子（建议背下来）
输入：
```txt
in   = [A, B, C, D, E]
flag = [1, 0, 1, 1, 0]   // A/C/D 有效
```

做排他前缀和：
```txt
scan = [0, 1, 1, 2, 3]
```

解释：
- `scan[i]` 表示 i 前面有多少个有效元素。
- 所以有效项写入 `out[scan[i]]`。

Scatter 结果：
```txt
out[0] = A
out[1] = C
out[2] = D
```

最终：
```txt
out   = [A, C, D]
count = scan[n-1] + flag[n-1] = 3
```

#### 为什么通常优先于 `atomicAdd`
- `atomicAdd(counter)` 直接 append 的确实现简单。
- 但高并发下全体线程争同一个原子计数器，热点严重，吞吐下降明显。
- `flag+scan+scatter` 更可扩展，且更容易保持结果顺序稳定。

#### 工程注意点
- `exclusive` 和 `inclusive` 不要混：如果是 inclusive scan，目标下标通常要 `-1`。
- 三个 pass 之间要有正确同步（barrier / semaphore），保证写入对下一步可见。
- 数据量大时做分层 scan：workgroup 内扫描 + block 总和扫描 + 回填。

#### 通俗说法
- `Flag`：先贴标签，留下谁。
- `Prefix Sum`：给留下的人发座位号。
- `Scatter`：按座位号入座，队伍就变紧凑了。

### 8.2 用 compute shader 计算一个 buffer 的总和，且 compute shader 可调用多次

#### 推荐思路：分层归约（hierarchical reduction）
1. 每个 workgroup 先在 shared memory 内做局部归约，写出 partial sum。
2. 对 partial buffer 再次 dispatch 归约。
3. 重复直到剩一个值。

#### 多次调用注意点
- 每一轮 dispatch 后要有正确 barrier，保证写入对下一轮可见。
- 可用 ping-pong buffer 存 partial，避免读写冲突。
- 若跨队列/跨提交，补 semaphore/fence 同步。

#### 复杂场景补充
- 如果是“多次增量求和”，可维护持久化累加 buffer（注意重置与精度问题）。

### 面试一句话
- 流压缩用 `flag+scan+scatter` 最通用；
- 总和计算用多轮并行归约最稳；
- 关键不是公式，而是每轮读写可见性与同步边界。
