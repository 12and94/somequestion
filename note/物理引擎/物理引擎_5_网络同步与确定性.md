# 物理引擎 5：网络同步与确定性

## 1. 为什么网络物理难
- 浮点误差、执行顺序、平台差异会导致状态分叉。
- 延迟和丢包让“同一时刻状态一致”更难。

## 2. 常见同步架构
### 2.1 服务器权威（Server Authoritative）
- 服务器真值，客户端预测后等待校正。

### 2.2 客户端预测 + 回滚
- 客户端先本地模拟，提高手感。
- 收到服务器状态后做 Reconciliation（重播输入纠偏）。

### 2.3 快照同步
- 周期下发关键状态，客户端插值/外推。

## 3. Rollback 核心流程
- 保存历史状态快照。
- 收到纠正包后回滚到对应 tick。
- 重放输入到当前 tick。

## 4. 确定性问题来源
- 浮点舍入差异。
- 线程执行顺序不确定。
- 容器遍历顺序不稳定。
- 第三方库平台行为差异。

## 5. 工程策略
- 强确定性（成本高）：固定点/严格顺序/平台统一。
- 近似一致 + 校正（常用）：允许小误差，依赖周期校正与平滑。

## 6. 常见面试追问
- 为什么同一输入不同机器结果会偏？
- 回滚系统最难的点是什么？
- 你如何平衡手感与一致性？
