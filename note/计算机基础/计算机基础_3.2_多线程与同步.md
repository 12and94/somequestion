## 题目：多线程有哪些锁？各自的使用场景？

### 标准准确的说法
- 先给总框架：并发同步原语可以分为
1. 独占锁：同一时刻只允许一个线程进入临界区。
2. 共享锁（读写锁）：允许多个读者并发，但写者独占。
3. 自旋类锁：抢不到就忙等，不立即阻塞睡眠。
4. 计数型同步：如 semaphore（信号量），它不是“互斥锁”，而是“配额/令牌”模型。

- 常见锁逐个说明：

1. `std::mutex`（最常用的独占互斥锁）
- 机制：`lock()` 成功后线程独占，其他线程阻塞等待。
- 优点：语义最清晰、可维护性最好、适用范围最广。
- 缺点：高竞争下会产生上下文切换开销；临界区设计不好会拖慢吞吐。
- 适用场景：保护共享 map/vector、对象状态更新、非极限性能路径。
- 实践建议：默认从 `mutex + RAII` 开始，不要先上复杂锁。

2. `std::timed_mutex`（支持超时的互斥锁）
- 机制：`try_lock_for / try_lock_until`，拿不到锁可超时返回。
- 优点：可做超时降级、避免无限等待、便于故障隔离。
- 缺点：超时失败路径会让业务逻辑复杂，重试策略不当会抖动。
- 适用场景：需要 SLA（Service Level Agreement，服务等级目标）兜底的接口、可降级业务。

3. `std::recursive_mutex`（可重入互斥锁）
- 机制：同一线程可重复加锁，内部维护持有计数；解锁次数要匹配。
- 优点：可兼容“函数调用链重入同一把锁”的遗留结构。
- 缺点：容易掩盖设计问题，锁层级变复杂，调试难度上升。
- 适用场景：历史代码改造过渡期、递归代码暂时无法重构时。
- 工程态度：能不用就不用，优先重构为非递归锁设计。

4. `std::shared_mutex`（读写锁，C++17）
- 机制：读线程可 `shared_lock` 并发进入，写线程 `unique_lock` 独占。
- 优点：读多写少时并发度明显提升。
- 缺点：
- 写者可能饥饿（实现相关）；读写切换频繁时性能未必优于 `mutex`。
- 临界区仍要尽量短，否则“读并发”也会被长读拖垮。
- 适用场景：配置中心快照、字典查询、缓存元数据等“读远多于写”的数据结构。

5. `std::shared_timed_mutex`（带超时的读写锁，C++14）
- 机制：在读写锁基础上增加超时尝试。
- 适用场景：既要读写并发，又要超时降级控制。
- 备注：新项目常优先 `shared_mutex`，除非确实需要 timed 能力。

6. 自旋锁（spin lock，标准库无统一高级类型，常基于 `std::atomic_flag` 或平台原语）
- 机制：抢不到锁时循环 CAS（Compare-And-Swap，比较并交换）/test-and-set，不睡眠。
- 优点：避免线程睡眠唤醒，极短临界区下可能低延迟。
- 缺点：竞争时会持续耗 CPU；单核或高竞争下可能非常差。
- 适用场景：临界区极短、线程不会被长时间抢占、对尾延迟敏感的热路径。
- 边界：若持锁代码里可能 I/O、系统调用、阻塞等待，几乎不该用自旋。

7. `std::scoped_lock` / `std::lock_guard` / `std::unique_lock`（不是锁类型，而是加锁策略封装）
- `lock_guard`：最轻量 RAII（Resource Acquisition Is Initialization，资源获取即初始化）封装，作用域退出自动解锁。
- `unique_lock`：更灵活，可延迟加锁、手动解锁、与条件变量配合。
- `scoped_lock`：可一次锁多把锁，内部做死锁规避（基于 `std::lock` 语义）。
- 面试要点：很多线程安全 bug 不是“选错锁”，而是“忘记在异常/早返回路径释放锁”，RAII 能系统性避免。

8. semaphore（信号量，`std::counting_semaphore`）与 mutex 的边界
- semaphore 管的是“可并发名额 N”，不是“谁拥有锁”。
- mutex 管的是“互斥访问 + 所有权”，同一时刻只允许一个线程进入。
- 典型组合：semaphore 做限流，mutex 保护共享结构。

- 选型决策顺序（实战比死记更重要）：
1. 先问是否真的需要共享可变数据。
2. 需要共享时，先用 `std::mutex + lock_guard/scoped_lock` 拿到正确性。
3. 确认“读远多于写”且锁是瓶颈，再考虑 `shared_mutex`。
4. 需要超时/降级时，再上 `timed_mutex` 或 timed 读写锁。
5. 仅在明确“临界区极短 + 低等待 + 可控竞争”时考虑自旋。
6. 用 profile（性能剖析）和 contention 指标验证收益，不凭感觉换锁。

- 常见误区（面试高频）：
1. “`recursive_mutex` 更强大，优先用它”：错误。它通常是结构问题的权宜之计。
2. “读写锁一定比互斥锁快”：错误。写频繁或竞争高时可能更慢。
3. “自旋锁更快”：错误。高竞争时会烧 CPU，吞吐和时延都可能变差。
4. “只要用了 atomic 就不需要锁”：错误。复合状态一致性仍常需要锁。

- 最小代码片段：
```cpp
// 1) 默认方案：mutex + RAII
std::mutex m;
int counter = 0;

void inc() {
    std::lock_guard<std::mutex> lk(m);
    ++counter;
}
```

```cpp
// 2) 读多写少：shared_mutex
#include <shared_mutex>
std::shared_mutex rw;
int value = 0;

int read_value() {
    std::shared_lock<std::shared_mutex> lk(rw);
    return value;
}

void write_value(int x) {
    std::unique_lock<std::shared_mutex> lk(rw);
    value = x;
}
```

```cpp
// 3) 超时控制：timed_mutex
#include <chrono>
#include <mutex>
std::timed_mutex tm;

bool try_update_with_timeout() {
    if (!tm.try_lock_for(std::chrono::milliseconds(5))) {
        return false; // 超时降级
    }
    tm.unlock();
    return true;
}
```

### 通俗易懂的理解
- 锁就像不同安检通道：
1. `mutex` 是“单通道，一次过一人”，最稳。
2. `shared_mutex` 是“读者走多人通道，写者清场独占”。
3. `timed_mutex` 是“排队超过 5ms 就放弃走降级流程”。
4. 自旋锁是“门口原地一直等”，省了挂号休眠，但会一直占 CPU。
- 所以不是“越高级越好”，而是“你的业务等待模型是什么”。

### 面试回答简版模板
`锁选型我会按负载模型分层：默认先用 mutex + RAII 保证正确性；读多写少再考虑 shared_mutex；需要超时降级用 timed_mutex；只有在临界区极短且竞争可控时才考虑自旋。recursive_mutex 通常仅用于遗留重入场景，不作为默认方案。`

### 对应示例
- 面试可直接给一个“1 句话决策树”：
- `先 mutex 保正确，再用 profile 看争用热点；读多写少上 shared_mutex，超时需求上 timed，极短临界区才尝试自旋。`

## 题目：自旋锁和互斥锁的区别？

### 标准准确的说法
- 自旋锁：抢不到锁时循环忙等，不让出 CPU。
- 互斥锁：抢不到锁时阻塞休眠，交给调度器。
- 代价对比：
- 自旋锁避免上下文切换，但会占用 CPU 周期。
- 互斥锁节省 CPU，但有睡眠/唤醒与调度开销。
- 使用边界：
- 临界区极短、核数充足、低竞争：自旋可能更优。
- 临界区不确定或偏长：互斥锁更稳。
- 常见误区：
- “自旋更快”不是普适结论；高竞争下自旋通常更差。

### 通俗易懂的理解
- 自旋是“站门口一直等”，互斥是“拿号去旁边坐着等叫号”。

### 面试回答简版模板
`自旋锁用 CPU 换切换成本，互斥锁用切换换 CPU 资源。短临界区可自旋，等待时间不确定时优先互斥。`

## 题目：死锁是什么？如何避免？

### 标准准确的说法
- 死锁：多个线程形成循环等待，系统进入永久阻塞。
- 四必要条件：互斥、持有并等待、不可抢占、循环等待。
- 常见触发场景：
- 锁顺序不一致（A->B 与 B->A）。
- 业务回调中再次请求同一批资源。
- 锁粒度过大 + 嵌套过深。
- 避免策略：
1. 全局锁顺序（lock hierarchy）。
2. 减少嵌套锁和锁持有时长。
3. 多锁同时获取用 `std::scoped_lock`。
4. 用 `try_lock` + 回退避免硬等待。
5. 加超时与监控，快速暴露问题。

### 通俗易懂的理解
- 两个线程各拿一把钥匙，都等对方先放手，就是死锁。

### 面试回答简版模板
`死锁的根因是循环等待。工程上最有效的是统一锁顺序、减少嵌套、多锁用 scoped_lock，并用超时和监控尽早发现。`

## 题目：进程间通信的方式有哪些？

### 标准准确的说法
- 先给一句总定义：
- IPC（Inter-Process Communication，进程间通信）是“彼此独立地址空间的进程之间交换数据和同步状态”的机制。

- 常见 IPC 方式逐个说明：

1. 管道（Pipe）  
- 匿名管道（Anonymous Pipe）：
- 本质是内核维护的字节流缓冲区（byte stream），通常单向。
- 经典场景是有亲缘关系进程（如 `fork` 后父子进程）通信。
- 优点：简单、系统调用接口直接、适合小到中等流式数据。
- 缺点：通常仅本机；无消息边界（读到的是字节流切片）；不天然支持多对多复杂拓扑。
- 命名管道（Named Pipe / FIFO）：
- 在文件系统中有名字（路径），无亲缘关系进程也可通信。
- 仍是字节流语义，仍需自己定义应用层消息边界。

2. 消息队列（Message Queue）
- 本质是内核维护“按消息为单位”的队列，天然有消息边界。
- 支持异步解耦：发送者和接收者不必同时在线。
- 常见能力：消息类型/优先级（实现相关）、阻塞或非阻塞收发。
- 优点：消息边界清晰，业务模型直观（任务分发、事件投递）。
- 缺点：内核队列容量有限；大消息传输不经济；系统接口和平台差异需关注。

3. 共享内存（Shared Memory）
- 本质是多个进程映射同一段物理页（同一块内存），数据不需要在内核和用户态间反复拷贝。
- 优点：吞吐和延迟通常最优，适合大数据量和高频交换。
- 缺点：它只解决“看见同一份数据”，不解决“并发一致性”。
- 必须配合同步原语（mutex/semaphore/atomic/futex 等）避免竞态。
- 常见风险：写覆盖、读到中间态、缓存行伪共享、版本一致性管理复杂。

4. 信号量（Semaphore）
- 本质是计数同步原语，不是主要数据通道。
- 作用是控制“可并发数量”或“先后顺序”，常用于生产者-消费者、资源池限流。
- 优点：语义明确、同步协调能力强。
- 缺点：不适合承载业务大数据，只表示“能不能做/还有几个名额”。

5. Socket（套接字）
- 最通用 IPC：既可本机也可跨主机。
- 本机常用 Unix Domain Socket（AF_UNIX），跨机常用 TCP/UDP（AF_INET/AF_INET6）。
- 优点：
- 跨主机能力强，协议生态成熟（重连、超时、TLS、多路复用等）。
- 可做请求-响应、流式传输、消息化协议（应用层封包）。
- 缺点：
- 协议栈和序列化成本更高，编程复杂度通常高于本机 IPC。

6. 信号（Signal）
- 本质是异步通知机制（如 `SIGTERM`、`SIGCHLD`），用于“事件告知”而非传输大数据。
- 优点：轻量、适合状态通知（退出、重载配置、子进程状态变化）。
- 缺点：可携带数据能力弱，处理函数受异步安全限制（async-signal-safe）。
- 典型用法：信号只做“唤醒/打断/通知”，真正数据走管道、队列或共享内存。

- 选型维度（实战顺序）：
1. 是否跨主机：跨主机通常首选 socket。
2. 数据规模与时延：大数据低延迟优先共享内存；控制消息可队列/管道。
3. 是否需要消息边界：需要边界优先消息队列或 datagram/socket 协议封包。
4. 是否需要强解耦与缓冲：消息队列更自然。
5. 同步复杂度能否接受：共享内存性能高，但同步和一致性成本最高。

- 典型场景映射：
1. 父子进程命令/日志流：匿名管道。
2. 无亲缘本机进程的简单流通信：命名管道（FIFO）。
3. 任务投递与异步消费：消息队列。
4. 高频行情/图像帧共享：共享内存 + 信号量/原子同步。
5. 跨机服务调用：socket（TCP/UDP/Unix Domain 按场景选）。
6. 进程控制通知（退出、重载）：signal。

### 通俗易懂的理解
- 你可以把 IPC 想成几种“沟通工具”，每种工具都适合不同场景：
1. 管道：像一根水管，数据按字节流过去，简单直接，但默认不知道“哪一段是一条完整消息”。
2. 消息队列：像快递柜，一格一格按“包裹”收发，天然分条，适合异步派单。
3. 共享内存：像共用白板，大家都能直接看到和改同一块内容，速度最快，但最容易互相写乱，必须配锁。
4. 信号量：像发门票，不传货物本身，只管“还有几个名额能进”。
5. Socket：像电话/网络通话，本机能用，跨城市也能用，最通用但协议处理更复杂。
6. Signal：像按门铃，只负责“提醒你有事”，不负责把大包裹送进来。

### 面试回答简版模板
`IPC 常见有管道、消息队列、共享内存、信号量、socket、signal。管道简单流式，消息队列有边界且异步解耦，共享内存性能最高但同步最复杂，信号量主要做同步配额，signal 主要做通知，跨主机通信通常选 socket。选型按“跨机需求、数据规模与时延、消息边界、复杂度”四步判断。`

### 对应示例
- 最小选型示例（可直接复述）：
```text
场景A：父进程把子进程 stdout 实时收集 -> 匿名管道
场景B：多个 worker 消费任务 -> 消息队列
场景C：高频大块数据共享 -> 共享内存 + 信号量/原子
场景D：服务A调用远端服务B -> socket(TCP)
场景E：通知进程重载配置 -> signal + 配置读取
```

## 题目：原子操作和原子变量是什么？

### 标准准确的说法
- 原子变量（`std::atomic<T>`）保证单次原子读写/更新不会被并发撕裂。
- 常见操作：`load/store/fetch_add/exchange/compare_exchange`。
- 核心不止“原子性”，还有“内存序”：
- `memory_order_relaxed`：只保原子，不建同步顺序。
- `release/acquire`：建立跨线程发布-订阅关系。
- `seq_cst`：最强全序语义，通常最易理解。
- 边界：
- 原子并不能让复杂业务逻辑自动正确。
- 多字段一致性或大临界区仍常需锁。

### 通俗易懂的理解
- 原子像“一次完整盖章动作”，但只保证这一章完整，不保证你整份流程都正确。

### 面试回答简版模板
`atomic 保证单操作原子，并通过内存序控制可见性与顺序。简单标志和计数适合 atomic，复杂复合状态仍需锁或更高层并发设计。`

## 题目：无锁队列的实现？

### 标准准确的说法
- 无锁队列核心：用 CAS 在并发下更新 head/tail，不用互斥阻塞。
- 模型分级：
- SPSC（单生产者单消费者）最简单，环形缓冲区常用。
- MPSC/SPMC 次之。
- MPMC 最复杂。
- 关键难点：
- ABA 问题。
- 节点内存回收安全（不能过早回收被其他线程读到的节点）。
- 缓存伪共享与重试风暴。
- 工程建议：
- 优先选简单模型（SPSC）与成熟库；
- MPMC 除非确有必要，且要配套 hazard pointer/epoch 回收策略。

### 通俗易懂的理解
- 无锁队列不是“没有同步”，而是用“原子重试”替代“上锁排队”。

### 面试回答简版模板
`无锁队列通常基于 CAS 维护指针/索引，重点难点是 ABA 和安全回收。工程上优先简单模型和成熟实现，避免手写高风险 MPMC。`

## 题目：信号量和互斥量的区别？

### 标准准确的说法
- 互斥量（mutex）：
- 二值独占，保护临界区“同一时刻仅一线程进入”。
- 信号量（semaphore）：
- 计数器语义，表示可用资源配额。
- 适用于生产者-消费者、限流、资源池并发数控制。
- 语义侧重点：
- mutex 重“所有权+互斥访问”。
- semaphore 重“资源数量协调”。

### 通俗易懂的理解
- mutex 是“单人钥匙”，semaphore 是“有 N 张门票”。

### 面试回答简版模板
`mutex 用于保护共享数据互斥访问，semaphore 用于计数型资源协调和限流。一个重互斥，一个重配额。`

## 题目：多线程数据竞争问题如何解决？

### 标准准确的说法
- 数据竞争定义：并发访问同一内存，至少一方写，且无同步。
- 解决策略分层：
1. 设计层：减少共享可变状态（消息传递、不可变数据）。
2. 同步层：`mutex`、`atomic`、读写锁、条件变量。
3. 架构层：分片（sharding）、线程本地存储、批处理归并。
- 调试手段：
- ThreadSanitizer、日志打点、竞争热点统计。
- 误区：
- `volatile` 不是并发同步手段，不能解决数据竞争。

### 通俗易懂的理解
- 最好的并发优化是“少抢同一份数据”，而不是“抢了再拼命加锁”。

### 面试回答简版模板
`数据竞争先从设计上减少共享，再用正确同步原语，最后配合工具验证。volatile 不能替代锁和 atomic。`

## 题目：线程池的实现？

### 标准准确的说法
- 线程池目标：复用线程，降低创建销毁成本，平滑并发负载。
- 基本组件：
- 任务队列（线程安全）。
- Worker 线程循环。
- 条件变量唤醒机制。
- 停机协议（优雅关闭）。
- 关键实现细节：
1. 任务提交策略：阻塞提交、拒绝策略、降级策略。
2. 异常隔离：任务异常不能崩掉 worker。
3. 背压与容量：队列上限与监控。
4. 停机语义：立即停机 vs drain（清空队列后停）。
- 性能指标：
- 队列长度、任务等待时延、worker 利用率、拒绝次数。

### 通俗易懂的理解
- 线程池是“固定工人 + 任务排队系统”，重点不是开线程，而是把调度与停机规则设计好。

### 面试回答简版模板
`线程池核心是任务队列、worker 循环、同步唤醒和停机协议。工程重点在背压、异常隔离和可观测性，而不只是“开 N 个线程取任务”。`
