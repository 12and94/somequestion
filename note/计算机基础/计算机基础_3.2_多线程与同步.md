## 题目：多线程有哪些锁？各自的使用场景？
>[!NOTE]- 标准准确的说法
>- 可以把 C++ 多线程“锁”分成两类来记：
>1. 锁本体（同步原语）：`std::mutex` 这类。
>2. 加锁工具（RAII 封装）：`std::lock_guard` / `std::unique_lock` 这类。
>
>- 常见锁本体 + 使用场景
>
>| 锁类型 | 特点 | 典型场景 | 注意点 |
>| --- | --- | --- | --- |
>| `std::mutex` | 最基础互斥锁，独占 | 保护一个共享结构（队列、map、状态） | 简单可靠，优先默认选它 |
>| `std::timed_mutex` | 支持超时尝试加锁 | 不想无限阻塞，需降级/超时返回 | 用 `try_lock_for/try_lock_until` |
>| `std::recursive_mutex` | 同一线程可重复加锁 | 旧代码/递归调用链里可能重复进入同一临界区 | 容易掩盖设计问题，能不用尽量不用 |
>| `std::recursive_timed_mutex` | 递归 + 超时 | 同上且还要超时控制 | 更少见，复杂度高 |
>| `std::shared_mutex` (C++17) | 读写锁：多读单写 | 读多写少的数据（配置、缓存、字典） | 写线程可能饥饿，注意公平性问题 |
>| `std::shared_timed_mutex` (C++14) | 读写锁 + 超时 | 读写锁场景且需要超时策略 | C++17 后常直接用 `shared_mutex` |
>| `std::once_flag` + `std::call_once` | 一次性初始化同步 | 单例初始化、懒加载全局资源 | 线程安全初始化首选，比手写双检锁稳 |
>
>- 常见“加锁工具” + 使用场景
>
>| 工具 | 适配锁 | 典型场景 | 注意点 |
>| --- | --- | --- | --- |
>| `std::lock_guard<Mutex>` | `mutex` 等 | 函数作用域内简单上锁解锁 | 最轻量，不能手动 `unlock` |
>| `std::unique_lock<Mutex>` | `mutex`/`timed_mutex` 等 | 需要延迟加锁、条件变量、手动解锁 | 功能强，开销略高于 `lock_guard` |
>| `std::scoped_lock` (C++17) | 多个互斥量 | 一次锁多个锁并避免死锁 | 多锁场景首选，内部用死锁避免策略 |
>| `std::shared_lock<shared_mutex>` | `shared_mutex` | 读路径共享锁 | 与写锁互斥，适合读多写少 |
>
>- 面试常问的“该怎么选”
>1. 默认先用 `std::mutex + std::lock_guard`。
>2. 需要配合条件变量时，用 `std::unique_lock`。
>3. 读多写少，用 `std::shared_mutex`（读用 `shared_lock`，写用 `unique_lock`/`lock_guard`）。
>4. 需要超时退避，用 `timed_mutex` 或 `shared_timed_mutex`。
>5. 多把锁一起拿，用 `std::scoped_lock` 或 `std::lock(...)`，避免手写顺序导致死锁。
>6. 初始化只做一次，用 `call_once`。
>
>- 高频坑点（实战很重要）
>- 不要手动 `lock()` 后忘记 `unlock()`，优先 RAII。
>- 临界区尽量短，锁内不要做 I/O、网络、复杂计算。
>- 多锁顺序不一致容易死锁。
>- `recursive_mutex` 不是“高级功能”，更多是“历史兼容工具”。
>- 性能优化优先级：先保证正确，再减少锁粒度，再考虑无锁结构/原子操作。
>[!tip]- 代码示例（SPSC 无锁环形队列，最容易上手）
>- 思路：一个生产者只写 `tail`，一个消费者只写 `head`，因此不需要互斥锁。
>- 判空：`head == tail`；判满：`(tail + 1) % N == head`。
>
>```cpp
>#include <array>
>#include <atomic>
>#include <cstddef>
>
>template <class T, std::size_t N>
>class SPSCQueue {
>    static_assert(N > 1, "N must be > 1");
>
>public:
>    bool push(const T& v) {
>        auto t = tail_.load(std::memory_order_relaxed);
>        auto next = (t + 1) % N;
>
>        if (next == head_.load(std::memory_order_acquire)) {
>            return false; // full
>        }
>
>        buffer_[t] = v; // 先写数据
>        tail_.store(next, std::memory_order_release); // 再发布 tail
>        return true;
>    }
>
>    bool pop(T& out) {
>        auto h = head_.load(std::memory_order_relaxed);
>
>        if (h == tail_.load(std::memory_order_acquire)) {
>            return false; // empty
>        }
>
>        out = buffer_[h]; // 先读数据
>        head_.store((h + 1) % N, std::memory_order_release); // 再推进 head
>        return true;
>    }
>
>private:
>    std::array<T, N> buffer_{};
>    std::atomic<std::size_t> head_{0};
>    std::atomic<std::size_t> tail_{0};
>};
>```

>[!quote]- 通俗易懂的理解
>- 把它想成“环形传送带”：
>1. `tail` 负责把新货放到下一个槽位。
>2. `head` 负责从下一个槽位取货。
>3. 在 SPSC 里，一个线程只写 `tail`、另一个只写 `head`，通常不需要 CAS。
>4. 到了 MPSC/MPMC，多个线程会竞争同一个位置，这时才需要 CAS 重试。
>- 所以“无锁”不是“没有同步”，而是“用原子语义保证正确并发”。

>[!example]- 面试回答简版模板
>`无锁队列是用原子操作而不是互斥锁来维护并发队列状态。最容易落地的是 SPSC 环形队列：生产者推进 tail、消费者推进 head，用 acquire/release 保证可见性。若升级到 MPSC/MPMC，通常需要 CAS 处理竞争，同时要解决 ABA 和内存回收问题。`
>[!info]- 对应示例
>- 面试可直接给一个“1 句话决策树”：
>- `先 mutex 保正确，再用 profile 看争用热点；读多写少上 shared_mutex，超时需求上 timed，极短临界区才尝试自旋。`

## 题目：自旋锁和互斥锁的区别？
>[!NOTE]- 标准准确的说法
>- 自旋锁：抢不到锁时循环忙等，不让出 CPU。
>- 互斥锁：抢不到锁时阻塞休眠，交给调度器。
>- 代价对比：
>- 自旋锁避免上下文切换，但会占用 CPU 周期。
>- 互斥锁节省 CPU，但有睡眠/唤醒与调度开销。
>- 使用边界：
>- 临界区极短、核数充足、低竞争：自旋可能更优。
>- 临界区不确定或偏长：互斥锁更稳。
>- 常见误区：
>- “自旋更快”不是普适结论；高竞争下自旋通常更差。

## 题目：死锁是什么？如何避免？
>[!NOTE]- 标准准确的说法
>- 死锁：多个线程形成循环等待，系统进入永久阻塞。
>- 四必要条件：互斥、持有并等待、不可抢占、循环等待。
>- 常见触发场景：
>- 锁顺序不一致（A->B 与 B->A）。
>- 业务回调中再次请求同一批资源。
>- 锁粒度过大 + 嵌套过深。
>- 避免策略：
>1. 全局锁顺序（lock hierarchy）。
>2. 减少嵌套锁和锁持有时长。
>3. 多锁同时获取用 `std::scoped_lock`。
>4. 用 `try_lock` + 回退避免硬等待。
>5. 加超时与监控，快速暴露问题。

## 题目：进程间通信的方式有哪些？
>[!NOTE]- 标准准确的说法
>- 先给一句总定义：
>- IPC（Inter-Process Communication，进程间通信）是“彼此独立地址空间的进程之间交换数据和同步状态”的机制。
>
>- 常见 IPC 方式逐个说明：
>
>1. 管道（Pipe）
>- 匿名管道（Anonymous Pipe）：
>- 本质是内核维护的字节流缓冲区（byte stream），通常单向。
>- 经典场景是有亲缘关系进程（如 `fork` 后父子进程）通信。
>- 优点：简单、系统调用接口直接、适合小到中等流式数据。
>- 缺点：通常仅本机；无消息边界（读到的是字节流切片）；不天然支持多对多复杂拓扑。
>- 命名管道（Named Pipe / FIFO）：
>- 在文件系统中有名字（路径），无亲缘关系进程也可通信。
>- 仍是字节流语义，仍需自己定义应用层消息边界。
>
>2. 消息队列（Message Queue）
>- 本质是内核维护“按消息为单位”的队列，天然有消息边界。
>- 支持异步解耦：发送者和接收者不必同时在线。
>- 常见能力：消息类型/优先级（实现相关）、阻塞或非阻塞收发。
>- 优点：消息边界清晰，业务模型直观（任务分发、事件投递）。
>- 缺点：内核队列容量有限；大消息传输不经济；系统接口和平台差异需关注。
>
>3. 共享内存（Shared Memory）
>- 本质是多个进程映射同一段物理页（同一块内存），数据不需要在内核和用户态间反复拷贝。
>- 优点：吞吐和延迟通常最优，适合大数据量和高频交换。
>- 缺点：它只解决“看见同一份数据”，不解决“并发一致性”。
>- 必须配合同步原语（mutex/semaphore/atomic/futex 等）避免竞态。
>- 常见风险：写覆盖、读到中间态、缓存行伪共享、版本一致性管理复杂。
>
>4. 信号量（Semaphore）
>- 本质是计数同步原语，不是主要数据通道。
>- 作用是控制“可并发数量”或“先后顺序”，常用于生产者-消费者、资源池限流。
>- 优点：语义明确、同步协调能力强。
>- 缺点：不适合承载业务大数据，只表示“能不能做/还有几个名额”。
>
>5. Socket（套接字）
>- 最通用 IPC：既可本机也可跨主机。
>- 本机常用 Unix Domain Socket（AF_UNIX），跨机常用 TCP/UDP（AF_INET/AF_INET6）。
>- 优点：
>- 跨主机能力强，协议生态成熟（重连、超时、TLS、多路复用等）。
>- 可做请求-响应、流式传输、消息化协议（应用层封包）。
>- 缺点：
>- 协议栈和序列化成本更高，编程复杂度通常高于本机 IPC。
>
>6. 信号（Signal）
>- 本质是异步通知机制（如 `SIGTERM`、`SIGCHLD`），用于“事件告知”而非传输大数据。
>- 优点：轻量、适合状态通知（退出、重载配置、子进程状态变化）。
>- 缺点：可携带数据能力弱，处理函数受异步安全限制（async-signal-safe）。
>- 典型用法：信号只做“唤醒/打断/通知”，真正数据走管道、队列或共享内存。
>
>- 选型维度（实战顺序）：
>1. 是否跨主机：跨主机通常首选 socket。
>2. 数据规模与时延：大数据低延迟优先共享内存；控制消息可队列/管道。
>3. 是否需要消息边界：需要边界优先消息队列或 datagram/socket 协议封包。
>4. 是否需要强解耦与缓冲：消息队列更自然。
>5. 同步复杂度能否接受：共享内存性能高，但同步和一致性成本最高。
>
>- 典型场景映射：
>1. 父子进程命令/日志流：匿名管道。
>2. 无亲缘本机进程的简单流通信：命名管道（FIFO）。
>3. 任务投递与异步消费：消息队列。
>4. 高频行情/图像帧共享：共享内存 + 信号量/原子同步。
>5. 跨机服务调用：socket（TCP/UDP/Unix Domain 按场景选）。
>6. 进程控制通知（退出、重载）：signal。

## 题目：原子操作和原子变量是什么？
>[!NOTE]- 标准准确的说法
>- 先给一句完整定义：
>- 原子变量（`std::atomic<T>`）保证对该变量的单次操作不可分割，并在指定内存序（memory order）下约束跨线程可见性与顺序。
>
>1. 三个核心概念必须区分
>- 原子性（atomicity）：
>- 单次读/写/读改写（RMW, Read-Modify-Write）不会被并发“撕裂”。
>- 可见性（visibility）：
>- 一个线程写入后，另一个线程什么时候能看到。
>- 有序性（ordering）：
>- 编译器和 CPU 是否允许把指令重排；跨线程观察顺序是否受约束。
>
>2. 常见原子操作分别是什么
>- `load(order)`：原子读取。
>- `store(v, order)`：原子写入。
>- `exchange(v, order)`：原子交换，返回旧值。
>- `fetch_add/sub/and/or/xor(order)`：原子读改写并返回旧值，常用于计数器。
>- `compare_exchange_weak/strong(expected, desired, order)`：
>- CAS（Compare-And-Swap，比较并交换）。若当前值等于 `expected` 则写入 `desired` 并成功；否则失败并把当前值回填给 `expected`。
>- `weak` 可能“伪失败（spurious failure）”，适合循环重试；`strong` 更少伪失败，单次尝试语义更直观。
>
>3. 内存序（memory order）要点
>- `memory_order_relaxed`：
>- 只保证该原子变量本身操作原子，不提供跨线程先后同步关系。
>- 适合纯计数统计（不依赖其他共享数据时序）。
>- `memory_order_release`（写端） + `memory_order_acquire`（读端）：
>- 建立“发布-订阅（publish-subscribe）”关系。
>- 读端一旦看到发布标志，就能看到发布前写入的数据（形成 happens-before）。
>- `memory_order_acq_rel`：
>- 用于读改写操作，兼具 acquire 与 release 语义。
>- `memory_order_seq_cst`（sequentially consistent）：
>- 最强语义：所有 `seq_cst` 原子操作看起来像处在一个全局总顺序中。
>- 最易推理，但在部分平台上代价更高。
>- `memory_order_consume`：
>- 标准层面长期实现不一致，工程中通常按 acquire 对待。
>
>4. happens-before（先行发生）与正确性
>- 并发正确性核心是“是否建立了同步边”。
>- 若线程 A 的写通过 release，线程 B 通过 acquire 读到对应标志，则 A 的先前写对 B 可见（happens-before 成立）。
>- 没有同步边时，即使用了原子，也可能读到旧状态或中间状态。
>
>5. 原子不是万能替代锁
>- 原子擅长：
>- 标志位、状态位、计数器、简单无锁结构的索引更新。
>- 原子不擅长：
>- 多变量不变量维护（例如 `balance` 和 `version` 必须一致）。
>- 大临界区事务语义、复杂条件等待。
>- 这类场景通常仍要 `mutex` 或更高层并发结构。
>
>6. lock-free / wait-free 需要会说
>- lock-free：系统整体保证“总有线程前进”，不保证每个线程都不饿死。
>- wait-free：每个线程都能在有限步内完成，要求更强，实现更难。
>- C++ 可通过 `atomic.is_lock_free()` 观测目标类型在当前平台是否 lock-free。
>
>7. 常见误区
>- 误区 1：`volatile` 可以做线程同步。错误，`volatile` 不是并发同步原语。
>- 误区 2：用了原子就没有数据竞争。错误，复合状态仍可竞争。
>- 误区 3：`relaxed` 一定更快且总可用。错误，若需要可见性顺序，必须用 acquire/release 或更强语义。
>- 误区 4：CAS 一次失败就代表逻辑错误。错误，`weak` 在循环里失败是正常现象。

>[!tip]- 代码示例（SPSC 无锁环形队列，最容易上手）
>- 思路：一个生产者只写 `tail`，一个消费者只写 `head`，因此不需要互斥锁。
>- 判空：`head == tail`；判满：`(tail + 1) % N == head`。
>
>```cpp
>#include <array>
>#include <atomic>
>#include <cstddef>
>
>template <class T, std::size_t N>
>class SPSCQueue {
>    static_assert(N > 1, "N must be > 1");
>
>public:
>    bool push(const T& v) {
>        auto t = tail_.load(std::memory_order_relaxed);
>        auto next = (t + 1) % N;
>
>        if (next == head_.load(std::memory_order_acquire)) {
>            return false; // full
>        }
>
>        buffer_[t] = v; // 先写数据
>        tail_.store(next, std::memory_order_release); // 再发布 tail
>        return true;
>    }
>
>    bool pop(T& out) {
>        auto h = head_.load(std::memory_order_relaxed);
>
>        if (h == tail_.load(std::memory_order_acquire)) {
>            return false; // empty
>        }
>
>        out = buffer_[h]; // 先读数据
>        head_.store((h + 1) % N, std::memory_order_release); // 再推进 head
>        return true;
>    }
>
>private:
>    std::array<T, N> buffer_{};
>    std::atomic<std::size_t> head_{0};
>    std::atomic<std::size_t> tail_{0};
>};
>```

>[!quote]- 通俗易懂的理解
>- 把它想成“环形传送带”：
>1. `tail` 负责把新货放到下一个槽位。
>2. `head` 负责从下一个槽位取货。
>3. 在 SPSC 里，一个线程只写 `tail`、另一个只写 `head`，通常不需要 CAS。
>4. 到了 MPSC/MPMC，多个线程会竞争同一个位置，这时才需要 CAS 重试。
>- 所以“无锁”不是“没有同步”，而是“用原子语义保证正确并发”。

>[!example]- 面试回答简版模板
>`无锁队列是用原子操作而不是互斥锁来维护并发队列状态。最容易落地的是 SPSC 环形队列：生产者推进 tail、消费者推进 head，用 acquire/release 保证可见性。若升级到 MPSC/MPMC，通常需要 CAS 处理竞争，同时要解决 ABA 和内存回收问题。`
>[!info]- 对应示例
>- 示例 1：`relaxed` 计数（仅统计，不同步其他数据）
>```cpp
>#include <atomic>
>
>std::atomic<int> cnt{0};
>
>void on_event() {
>   cnt.fetch_add(1, std::memory_order_relaxed);
>}
>```
>
>- 示例 2：`release/acquire` 发布订阅
>```cpp
>#include <atomic>
>#include <string>
>
>std::string payload;
>std::atomic<bool> ready{false};
>
>void producer() {
>   payload = "hello";
>   ready.store(true, std::memory_order_release); // 发布
>}
>
>void consumer() {
>   if (ready.load(std::memory_order_acquire)) {  // 订阅
>       // 看到 ready=true 后，能看到 producer 在 release 前写入的 payload
>       auto x = payload;
>       (void)x;
>   }
>}
>```
>
>- 示例 3：CAS 循环（`weak` 可伪失败，循环是常态）
>```cpp
>#include <atomic>
>
>std::atomic<int> value{0};
>
>void add10() {
>   int old = value.load(std::memory_order_relaxed);
>   while (!value.compare_exchange_weak(
>       old, old + 10,
>       std::memory_order_acq_rel,
>       std::memory_order_relaxed)) {
>       // 失败时 old 已被回填为最新值，继续重试
>   }
>}
>```

## 题目：无锁队列的实现？
>[!NOTE]- 标准准确的说法
>- 先给定义：
>- 无锁队列（lock-free queue）是指“队列并发操作不依赖全局互斥锁阻塞”，通常用原子操作（尤其 CAS）维护队列状态。
>- 这里的“无锁”不是“没有同步”，而是“用原子重试替代互斥阻塞”。
>
>1. CAS 是什么（你需要会展开全称）
>- CAS = Compare-And-Swap（比较并交换）。
>- 语义：只有当当前值仍等于 `expected` 时，才把它改成 `desired`，否则失败并返回当前值。
>- 为什么用于无锁队列：
>- 多线程并发更新同一个指针/索引时，CAS 能保证“只有一个线程抢到更新权”，失败者重试。
>
>2. head / tail 是什么
>- `head`：队头位置（或队头节点指针），出队（dequeue）时主要更新它。
>- `tail`：队尾位置（或队尾节点指针），入队（enqueue）时主要更新它。
>- 在线性结构里，你可以把它理解为：
>- `head` 指向“下一次要被取走的位置”；
>- `tail` 指向“下一次要被写入的位置”。
>- 在环形数组实现中通常是“索引”；在链表实现中通常是“节点指针”。
>
>3. 无锁队列常见实现族
>- 有界环形队列（ring buffer）：
>- 固定容量，`head/tail` 多为递增索引（对容量取模）。
>- 优点：无动态分配，缓存局部性好，性能稳定。
>- 缺点：容量固定，需要处理“满/空判定”。
>- 链表队列（linked queue）：
>- `head/tail` 是节点指针，容量可扩展。
>- 优点：弹性容量。
>- 缺点：内存分配和安全回收复杂，是高风险点。
>
>4. 模型缩写逐个解释（必须会）
>- SPSC = Single Producer Single Consumer（单生产者单消费者）：
>- 最简单，常用环形缓冲区即可做到高性能且易验证正确性。
>- MPSC = Multiple Producer Single Consumer（多生产者单消费者）：
>- 生产端竞争同一个 `tail`，消费者单线程处理。
>- SPMC = Single Producer Multiple Consumer（单生产者多消费者）：
>- 消费端竞争同一个 `head`，生产端单线程写入。
>- MPMC = Multiple Producer Multiple Consumer（多生产者多消费者）：
>- 最复杂，`head` 和 `tail` 两端都可能高并发竞争。
>
>5. 进度保证术语（面试加分）
>- lock-free：系统整体保证“总有线程能推进”，但某个线程可能反复失败重试。
>- wait-free：每个线程都保证在有限步内完成，约束更强、实现更难。
>- obstruction-free：无竞争时可推进，有竞争时不保证。
>
>6. 无锁队列为什么难
>- ABA 问题：
>- 变量从 A 变 B 又变回 A，CAS 看起来“没变”，但实际语义已经变了。
>- 常见缓解：版本号/tagged pointer、双宽 CAS、hazard pointer 等。
>- 安全回收（memory reclamation）：
>- 一个线程刚读到节点地址，另一个线程就把该节点回收了，会出现悬垂访问。
>- 常见策略：Hazard Pointer（危险指针）、Epoch-Based Reclamation（按代回收，EBR）。
>- 缓存与重试风暴：
>- 热点 `head/tail` 在多核上会产生缓存一致性抖动；
>- 高竞争下 CAS 失败率高，空转重试会放大延迟。
>
>7. 内存序（memory order）不能忽略
>- 生产者写入元素后，再发布 `tail`，通常需要 release 语义。
>- 消费者读取 `head/tail` 并取元素，通常需要 acquire 语义。
>- 内存序设置错误时，可能出现“看见索引更新，却没看见对应数据”。
>
>8. 极简伪代码（以 CAS 更新为核心）
>```text
>enqueue(x):
> loop:
>   old_tail = tail.load()
>   next = old_tail + 1
>   if queue is full: return false
>   if CAS(tail, old_tail, next) succeeds:
>      buffer[old_tail] = x
>      publish with proper memory order
>      return true
>
>dequeue():
> loop:
>   old_head = head.load()
>   if queue is empty: return false
>   next = old_head + 1
>   if CAS(head, old_head, next) succeeds:
>      x = buffer[old_head]
>      return x
>```
>
>9. 工程建议（比“手写算法”更重要）
>- 先按模型降维：优先 SPSC，再考虑 MPSC/SPMC，最后才是 MPMC。
>- 优先成熟实现（如工业库）而不是手写高并发 MPMC。
>- 若必须链表化无锁结构，必须把“安全回收策略”当一等公民设计。
>- 先做正确性验证（压力测试 + sanitizer + 线性化检查），再做极限优化。

## 题目：多线程数据竞争问题如何解决？

>- ThreadSanitizer、日志打点、竞争热点统计。

>- `volatile` 不是并发同步手段，不能解决数据竞争。

>4. 到了 MPSC/MPMC，多个线程会竞争同一个位置，这时才需要 CAS 重试。
>- 所以“无锁”不是“没有同步”，而是“用原子语义保证正确并发”。

## 题目：线程池的实现？
>[!NOTE]- 标准准确的说法
>- 先给定义：
>- 线程池（Thread Pool）是“固定或可调数量的 worker 线程 + 任务队列 + 调度/停机协议”的执行框架。
>- 它解决的是“任务调度与资源复用”，不只是“少创建线程”。
>
>1. 为什么线程池常比“每次开新线程”更好
>- 线程创建/销毁有系统调用和调度开销；线程池把这部分摊薄。
>- 限制并发度，避免任务风暴把 CPU、内存和上下文切换打爆。
>- 便于统一治理：背压、超时、重试、拒绝策略、监控。
>
>2. 基本组件（每个都要会解释）
>- 任务队列（thread-safe queue）：
>- 存放待执行任务（函数对象、可调用对象、封装的作业单元）。
>- 常见实现：`mutex + condition_variable + deque`（简单稳定）。
>- Worker 集合：
>- N 个后台线程，循环“取任务 -> 执行 -> 继续取任务”。
>- 唤醒机制：
>- 队列为空时 worker 阻塞等待条件变量；有新任务时 `notify_one/notify_all`。
>- 池状态机：
>- `Running`（正常接收）/`Stopping`（不再接收）/`Stopped`（全部退出）。
>- 停机协议：
>- `stop_now`（立即停机，可能丢未执行任务）；
>- `drain`（不再接收新任务，执行完队列后退出）。
>
>3. 线程池生命周期（建议按这个流程讲）
>```text
>构造 -> 创建 N 个 worker 并进入等待
>提交任务 -> 入队 -> 唤醒 worker
>worker 取任务并执行 -> 回到等待
>收到停机信号:
> - stop_now: 尽快退出循环
> - drain: 队列清空后退出
>析构/close -> join 所有 worker -> 资源释放
>```
>
>4. worker 主循环核心逻辑（伪代码）
>```text
>worker_loop():
> while true:
>   lock(queue_mutex)
>   wait(cv, 条件: stopping 或 队列非空)
>   if stopping 且 (stop_now 或 队列为空):
>      unlock; break
>   task = 从队列取一个任务
>   unlock
>   try:
>      执行 task
>   catch (...):
>      记录异常，不能让 worker 线程崩掉
>```
>
>5. 任务提交（submit）需要明确的策略
>- 入队成功后返回句柄（常见是 `future`）让调用方拿结果/异常。
>- 队列满时策略必须提前定清楚：
>1. 阻塞提交（block）：提交线程等待有空位。
>2. 超时提交（timeout）：等一段时间失败返回。
>3. 拒绝策略（reject）：直接失败并计数告警。
>4. 调用者执行（caller-runs）：提交线程自己执行任务，形成自然背压。
>- 没有明确策略会导致隐式丢任务或系统雪崩。
>
>6. 异常与结果传播（最容易漏）
>- 任务内部异常必须被捕获并写入 `promise/future` 或日志系统。
>- 不可让异常逃出 worker 顶层导致线程终止，否则池容量悄悄下降。
>- 生产系统要有“worker 存活数”健康检查与自动补偿策略。
>
>7. 背压与容量治理（工程重点）
>- 队列不能无限长：无界队列会把内存吃爆并放大排队时延。
>- 必须给出上限：`max_queue_size`、最大并发、最大等待时长。
>- 关键指标：
>- 队列长度（queue depth）
>- 任务排队时延（queue wait latency）
>- 任务执行时延（run latency）
>- worker 利用率（worker utilization）
>- 拒绝次数（rejection count）
>- 超时/取消次数（timeout/cancel count）
>
>8. 常见架构变体
>- 固定大小线程池：实现简单，行为稳定。
>- 动态伸缩线程池：按负载增减 worker，需防抖和上下限控制。
>- 分层/分类线程池：CPU 密集与 I/O 密集任务分池，避免互相拖垮。
>- work-stealing（工作窃取）池：每个 worker 本地队列，空闲线程窃取他人任务提升负载均衡。
>
>9. 常见坑（面试高频）
>- 池内任务再同步等待池内任务（nested wait）导致自锁/饥饿死锁。
>- 长阻塞 I/O 任务放在 CPU 池，导致吞吐骤降。
>- 忘记停机时 `notify_all`，worker 永远睡眠导致 join 卡死。
>- 停机语义不清：调用方不知道任务是“必执行”还是“可能被丢弃”。
>
>10. 一个最小可落地骨架（C++ 风格）
>```cpp
>class ThreadPool {
>public:
>   explicit ThreadPool(size_t n);
>   ~ThreadPool(); // 调用 close + join
>
>   template<class F>
>   auto submit(F&& f) -> std::future<std::invoke_result_t<F>>;
>
>   void shutdown_now(); // 立即停机
>   void shutdown_drain(); // 清队列后停机
>
>private:
>   std::mutex m_;
>   std::condition_variable cv_;
>   std::deque<std::function<void()>> q_;
>   std::vector<std::thread> workers_;
>   bool stopping_ = false;
>   bool drain_ = true;
>};
>```

>[!quote]- 通俗易懂的理解
>- 把它想成“环形传送带”：
>1. `tail` 负责把新货放到下一个槽位。
>2. `head` 负责从下一个槽位取货。
>3. 在 SPSC 里，一个线程只写 `tail`、另一个只写 `head`，通常不需要 CAS。
>4. 到了 MPSC/MPMC，多个线程会竞争同一个位置，这时才需要 CAS 重试。
>- 所以“无锁”不是“没有同步”，而是“用原子语义保证正确并发”。
