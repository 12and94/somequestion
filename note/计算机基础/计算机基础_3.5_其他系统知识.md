## 题目：CPU缓存/预读机制？

### 标准准确的说法
- 缓存层级：L1/L2/L3，容量递增、延迟递增。
- 目标：利用局部性减少主存访问。
- 局部性：
- 时间局部性：刚访问的数据很快还会访问。
- 空间局部性：访问某地址后，附近地址更可能被访问。
- 硬件预取：CPU 根据访问模式预测并提前加载 cache line。
- 常见性能陷阱：
- 随机访存导致 cache miss 增多。
- 伪共享（false sharing）：不同线程改同一 cache line 不同字段也会频繁失效。
- 工程优化方向：
- 数据连续布局（AoS/SoA 按场景选）。
- 批处理访问与循环重排。
- 热字段聚合，避免冷热数据混放。

### 通俗易懂的理解
- 缓存像办公桌，主存像仓库；预读像助手猜你下一页要看什么提前翻好。

### 面试回答简版模板
`CPU 缓存靠局部性提速，预读会提前加载可能访问的数据。优化重点是访问模式和数据布局，尤其避免随机访存与伪共享。`

## 题目：浮点数的编码原则？精度问题？

### 标准准确的说法
- IEEE 754 浮点核心结构：符号位 + 指数位 + 尾数位。
- 精度本质：有限位表示无限小数，很多十进制小数无法精确表示（如 0.1）。
- 典型问题：
- 累计舍入误差。
- 直接相等比较不稳定。
- NaN、Inf、上溢/下溢。
- 工程建议：
- 比较使用 epsilon（`abs(a-b) < eps`）。
- 金额等强精度场景用定点/整数缩放或高精度十进制方案。
- 数值迭代关注算法稳定性（避免灾难性相消）。

### 通俗易懂的理解
- 浮点数不是“精确小数”，而是“有限位近似”。

### 面试回答简版模板
`浮点按 IEEE754 编码，有限精度决定了很多小数只能近似。工程中要避免直接相等比较，并按业务精度要求选对数值表示。`

## 题目：游戏引擎的GC是怎么做的？

### 标准准确的说法
- 引擎 GC 目标不是单纯“回收干净”，而是“回收正确 + 帧时间稳定”。
- 常见策略：
- 引用计数：实时回收，循环引用要额外处理。
- 标记-清除：可处理环，但停顿明显。
- 分代 GC：新生代高频回收，老年代低频回收。
- 增量/并发 GC：把停顿拆碎，降低单帧尖峰。
- 游戏场景关键点：
- 回收预算（每帧限制 GC 时间）。
- 对象生命周期与场景加载卸载节奏协同。
- 需要和资源系统（纹理/网格/句柄）配套，避免“对象回收但资源还泄漏”。

### 通俗易懂的理解
- 游戏 GC 像边开车边清理后备箱：不能一次停太久，否则就“掉帧卡顿”。

### 面试回答简版模板
`游戏引擎 GC 重点是可控停顿。常用分代、增量或并发策略把回收成本分摊到多帧，并结合对象生命周期和资源系统做整体管理。`

## 题目：Windows/Linux的内存管理机制？

### 标准准确的说法
- 共性：
- 都基于虚拟内存 + 分页 + 缺页中断机制。
- 都有进程隔离、页缓存、映射共享等能力。
- 差异主要在接口与策略：
- Linux：用户态常见 `mmap`、历史 `brk/sbrk`，配套 `/proc` 可观测体系。
- Windows：常见 `VirtualAlloc/VirtualFree`（保留 reserve / 提交 commit 分离语义）。
- 实务层面：
- 性能问题往往不是“谁机制更先进”，而是具体访问模式、分配策略和观测手段。
- 面试建议：
- 先讲共性机制，再讲 API 差异和调优工具差异。

### 通俗易懂的理解
- 两个系统都在做“虚拟地址到账物理页”的映射，只是操作入口和内核策略不同。

### 面试回答简版模板
`Windows 和 Linux 都是分页虚拟内存体系。共性是地址隔离与按页管理，差异主要在系统调用接口和工具链；回答时先共性后差异更稳。`
