## 题目：源代码到可执行文件的过程？
>[!NOTE]- 标准准确的说法
>- 先给总流程（工程视角）：
>```text
>源文件(.cpp/.h)
> -> 预处理(Preprocess)
> -> 编译(Compile)
> -> 汇编(Assemble)
> -> 链接(Link)
> -> 可执行文件(ELF/PE/Mach-O)
> -> 进程装载与启动(Loader + Runtime -> main)
>```
>- 严格说“产出可执行文件”常指前四步；但面试高分回答通常会顺带补一句“运行前还有装载阶段”。
>
>1. 预处理（Preprocess）
>- 做什么：
>- 展开 `#include`，把头文件内容并入当前翻译单元（Translation Unit, TU）。
>- 宏替换（`#define`）、条件编译（`#if/#ifdef`）、去注释等文本级处理。
>- 输出形态：预处理后的文本（常见后缀 `.i`）。
>- 典型命令（GCC/Clang）：
>```bash
>g++ -E main.cpp -o main.i
>```
>- 通俗解释：
>- 预处理就像“先把配方里的引用章节全贴进来、把缩写词全部替换成原文”，还没开始真正“理解语义”。
>- 常见坑：
>- 宏是文本替换，可能产生副作用与优先级陷阱（例如 `#define SQR(x) x*x`）。
>
>2. 编译（Compile）
>- 做什么：
>- 词法/语法/语义分析，类型检查，模板实例化，生成中间表示（IR）并做优化，最后产出汇编（`.s`）。
>- 输出形态：汇编代码（`.s`）。
>- 典型命令：
>```bash
>g++ -S main.i -o main.s
>```
>- 通俗解释：
>- 编译像“把人类语言的代码真正读懂并翻译成低级指令草稿（汇编）”，同时做一些“整理优化”。
>- 常见坑：
>- 语法错误、类型不匹配、模板实例化失败都属于编译阶段错误。
>
>3. 汇编（Assemble）
>- 做什么：
>- 把汇编文本转为机器码，生成目标文件（`.o/.obj`）。
>- 目标文件里包含：机器指令、符号表（symbol table）、重定位信息（relocation entries）、各段信息（`.text/.data/.bss`）。
>- 输出形态：目标文件（`.o/.obj`）。
>- 典型命令：
>```bash
>g++ -c main.s -o main.o
>```
>- 通俗解释：
>- 汇编像“把指令草稿压成可拼装的标准零件包”，每个零件包里会写清“我还依赖谁”。
>- 常见坑：
>- 目标文件本身通常还不能直接运行，因为外部符号地址还没最终确定。
>
>4. 链接（Link）
>- 做什么：
>- 把多个目标文件和库（静态库/动态库）合并，完成符号解析与重定位，产出可执行文件或动态库。
>- 核心任务 A：符号解析（Symbol Resolution）
>- 例如在 `main.o` 里引用了 `foo()`，链接器要在其他目标文件/库里找到 `foo` 的定义。
>- 核心任务 B：重定位（Relocation）
>- 把“未定地址引用”修正为最终地址或可运行时修正的引用。
>- 输出形态：`a.out/.exe` 或 `.so/.dll/.dylib`。
>- 典型命令：
>```bash
>g++ main.o util.o -o app
>```
>- 通俗解释：
>- 链接像“总装车间”：把零件包拼成整车，并把每根线最终接到正确位置。
>- 常见坑：
>- `undefined reference`（只声明未定义）、`multiple definition`（重复定义）、ABI 不兼容（编译选项/编译器差异）等。
>
>5. 装载与启动（Load + Runtime Startup，面试补充项）
>- 做什么：
>- OS 加载器把可执行文件映射进内存，装入依赖动态库，处理动态重定位，然后跳到程序入口（如 `_start`），再由运行时初始化后进入 `main`。
>- 通俗解释：
>- 可执行文件像“装好的车”，装载阶段是“点火前检查和挂接外设”，完成后才真正“上路运行 main”。
>- 常见坑：
>- 运行时报“找不到动态库”或“符号版本冲突”，属于装载/运行时链接问题，不是编译期问题。
>
>- 一个最小多文件示例（观察链接错误）
>```cpp
>// main.cpp
>extern int add(int, int);
>int main() { return add(1, 2); }
>```
>```cpp
>// add.cpp
>int add(int a, int b) { return a + b; }
>```
>```bash
># 正常
>g++ -c main.cpp -o main.o
>g++ -c add.cpp -o add.o
>g++ main.o add.o -o app
>
># 若漏掉 add.o，会在链接阶段报 undefined reference to add
>g++ main.o -o app
>```
>
>- 错误分类（高频面试点）
>1. 预处理/编译错误：语法、类型、模板、头文件依赖问题。
>2. 链接错误：未定义符号、重复定义、库顺序问题、ABI 不兼容。
>3. 装载错误：找不到 `.so/.dll`、符号版本不匹配、运行时路径配置问题。

>[!tip]- 代码示例
>```text
>源文件(.cpp/.h)
> -> 预处理(Preprocess)
> -> 编译(Compile)
> -> 汇编(Assemble)
> -> 链接(Link)
> -> 可执行文件(ELF/PE/Mach-O)
> -> 进程装载与启动(Loader + Runtime -> main)
>```

>[!quote]- 通俗易懂的理解
>- 可以把它理解成“做一台机器”的流水线：
>1. 预处理：把外部说明书页（`#include`）都贴进总文档，宏先替换掉。
>2. 编译：工程师读懂文档并画出机器的低级施工图（汇编）。
>3. 汇编：把施工图变成标准零件包（目标文件）。
>4. 链接：总装车间把所有零件包和外购件（库）拼成整机（可执行文件）。
>5. 装载启动：开机前由系统把整机搬进内存、接好依赖，最后启动 `main`。
>- 所以“编得过但跑不起来”并不矛盾，可能卡在链接或装载阶段。

>[!example]- 面试回答简版模板
>`源码到可执行核心是预处理、编译、汇编、链接：预处理做文本展开，编译做语义分析和优化，汇编生成目标文件，链接做符号解析和重定位。工程上还要补一句装载阶段：程序运行前加载器会映射可执行和动态库并完成运行时重定位。很多“编译通过但运行失败”问题实际属于链接或装载。`

>[!info]- 对应示例
>- 观察四阶段最小命令链：
>```bash
>g++ -E main.cpp -o main.i   # 预处理
>g++ -S main.i -o main.s     # 编译
>g++ -c main.s -o main.o     # 汇编
>g++ main.o -o app           # 链接
>```
>- 观察符号与重定位（Linux 常用）：
>```bash
>nm main.o
>readelf -r main.o
>```

## 题目：静态链接库和动态链接库的区别？
>[!NOTE]- 标准准确的说法
>- 静态库（`.a/.lib`）：
>- 链接时把所需代码拷入可执行文件。
>- 优点：部署简单，单文件可运行性好。
>- 缺点：体积大、代码重复、升级需重发程序。
>- 动态库（`.so/.dll/.dylib`）：
>- 运行时由加载器装入并解析。
>- 优点：多进程共享库代码页、升级灵活。
>- 缺点：部署和版本管理复杂，可能有运行时依赖问题。
>- 工程取舍：
>- 客户端发行常关注部署复杂度与热更新策略。
>- 服务端常关注内存复用、模块化与灰度发布。
>- ABI 兼容性是动态库升级中的核心风险点。

>[!quote]- 通俗易懂的理解
>- 静态链接像“把零件焊死到整机”，动态链接像“运行时插模块”。

>[!example]- 面试回答简版模板
>`静态库在构建期并入程序，部署简单但体积大；动态库在运行期装载，复用好但依赖管理更难。实际选型要看部署策略和 ABI 兼容成本。`

## 题目：编译型语言和解释型语言的区别？
>[!NOTE]- 标准准确的说法
>- 编译型：
>- 先把源码编译成机器码/中间产物，再执行。
>- 优势：运行性能通常更高，很多错误可前置到编译期。
>- 解释型：
>- 通过解释器逐步执行源码/字节码。
>- 优势：开发迭代快，动态能力强。
>- 现代现实：
>- 许多语言是混合模式（解释 + JIT + AOT 组合），二分法只适合入门抽象。
>- 面试表达建议：
>- 讲“执行链路”与“工程权衡”（性能、迭代效率、部署与调试体验）。

>```cpp

>```

>[!quote]- 通俗易懂的理解
>- 编译型像“先翻译整本书再阅读”，解释型像“边翻译边读”。

>[!example]- 面试回答简版模板
>`编译型偏性能和前置检查，解释型偏开发效率和灵活性。现代语言常采用混合执行模式，实际效果取决于具体实现与工具链。`
