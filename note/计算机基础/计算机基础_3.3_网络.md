## 题目：TCP和UDP的区别？

### 标准准确的说法
- 连接语义：
- TCP 面向连接，建立连接后传输。
- UDP 无连接，直接发送报文。
- 传输语义：
- TCP 是字节流（无消息边界）。
- UDP 是报文（保留消息边界）。
- 可靠性：
- TCP 内建重传、按序、流量控制、拥塞控制。
- UDP 默认尽力而为，不保证到达、顺序、去重。
- 开销：
- TCP 头与状态机更重，时延与带宽开销更高。
- UDP 头小、路径短，时延低。
- 典型选型：
- 强一致/可靠业务（登录、配置、交易）偏 TCP。
- 实时交互（位置、输入、语音）偏 UDP 或 UDP+可靠层。

### 通俗易懂的理解
- TCP 像“签收快递”，UDP 像“普通投递”。

### 面试回答简版模板
`TCP 可靠有序但更重，UDP 轻量低时延但不可靠。实时业务常用 UDP 并在应用层补可靠，强一致业务通常用 TCP。`

## 题目：三次握手和四次挥手的过程？

### 标准准确的说法
- 三次握手（建连）：
1. Client -> SYN（发送初始序号）。
2. Server -> SYN+ACK（确认并发送自己的初始序号）。
3. Client -> ACK（确认 server 序号），连接建立。
- 为什么三次：
- 双方都要确认“我能发、我能收、对方也能收发”。
- 四次挥手（断连）：
1. 主动方发 FIN。
2. 被动方回 ACK（表示我知道你不再发数据）。
3. 被动方处理完剩余数据后再发 FIN。
4. 主动方回 ACK。
- TIME_WAIT（主动关闭方常见状态）意义：
- 保证最后 ACK 丢失时可重发确认。
- 让旧连接迟到包自然过期，避免污染新连接。

### 通俗易懂的理解
- 建连像双方互报“我准备好了”；断连像双方分别确认“我不再发了”。

### 面试回答简版模板
`三次握手用于确认双向通信能力和序列号；四次挥手是半关闭流程，TIME_WAIT 用于保证最后确认可达并隔离旧报文。`

## 题目：游戏中用TCP还是UDP？UDP如何保证可靠？

### 标准准确的说法
- 实务上多为混合协议：
- 关键业务（账号、匹配、支付）走 TCP/可靠通道。
- 高频实时状态（移动、朝向、技能输入）走 UDP。
- UDP 可靠化组件常见包括：
1. 序列号：检测乱序和丢包。
2. ACK/NACK：确认收到哪些包。
3. 重传机制：超时重传、选择重传。
4. 去重与重排：按序交付上层。
5. 分片重组：大包拆分与超时丢弃。
6. 拥塞与带宽控制：限速、退避、发送窗口。
- 工程常做“多通道语义”：
- 可靠有序、可靠无序、不可靠有序、不可靠无序。

### 通俗易懂的理解
- 游戏不是“全 TCP 或全 UDP”，而是按消息价值分层运输。

### 面试回答简版模板
`游戏网络通常混合：关键消息走可靠通道，实时状态走 UDP。UDP 可靠化通过序号、ACK、重传、去重重排、分片和拥塞控制来实现。`

## 题目：帧同步和状态同步的区别？

### 标准准确的说法
- 帧同步（输入同步）：
- 同步玩家输入，各端本地运行同一仿真。
- 优点：带宽低、一致性强、便于回放。
- 缺点：强依赖确定性，延迟敏感，卡顿联动明显。
- 状态同步（状态广播）：
- 服务端周期发送世界状态（快照或增量），客户端插值预测。
- 优点：实现直观，对非确定性容忍度高。
- 缺点：带宽更高，插值/预测/回滚复杂。
- 常见映射：
- RTS/MOBA 偏帧同步。
- FPS/MMO 偏状态同步或混合。

### 通俗易懂的理解
- 帧同步是“发操作自己算结果”，状态同步是“发结果你来显示并修正”。

### 面试回答简版模板
`帧同步同步输入，要求确定性且延迟敏感；状态同步同步结果，依赖预测与修正。选型取决于游戏类型、网络条件和一致性要求。`

## 题目：OSI七层模型？

### 标准准确的说法
- OSI 七层（低到高）：
- 物理层
- 数据链路层
- 网络层
- 传输层
- 会话层
- 表示层
- 应用层
- 工程中更常用 TCP/IP 分层：
- 链路层、网络层（IP）、传输层（TCP/UDP）、应用层（HTTP/DNS/自定义协议）。
- 面试建议：
- 先准确给 OSI 七层，再映射到 TCP/IP，体现“理论 + 实践”。

### 通俗易懂的理解
- 分层是为了“职责解耦”：下层管怎么送，上层管送什么。

### 面试回答简版模板
`OSI 是理论七层模型，工程更常按 TCP/IP 四层落地。面试可先背七层，再说明 TCP/UDP/IP 所在层和协议映射。`

## 题目：大端和小端？网络字节序是哪个？

### 标准准确的说法
- 大端：高位字节放低地址。
- 小端：低位字节放低地址。
- 网络字节序规定为大端。
- 跨平台网络通信要做字节序转换（如 `htonl/htons/ntohl/ntohs`）。
- 典型风险：
- 本地直接序列化结构体后发网，可能因端序/对齐差异导致协议不兼容。

### 通俗易懂的理解
- 端序就是“多字节数字在内存里的排队顺序”，网络要求统一队形是大端。

### 面试回答简版模板
`大端高位在前、小端低位在前；网络字节序统一大端。跨平台协议传输要显式做主机序和网络序转换。`
