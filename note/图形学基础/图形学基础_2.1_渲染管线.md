## 题目：介绍一下GPU渲染管线
>[!NOTE]- 标准准确的说法
>- 可以把实时渲染管线理解为“把场景数据逐步变成屏幕像素”的工业流水线。典型顺序是：
>- `CPU应用阶段 -> 输入装配(IA) -> 顶点着色(VS) -> (可选)曲面细分(Tessellation) -> (可选)几何着色(GS) -> 图元装配/裁剪/剔除 -> 光栅化 -> 片元着色(FS/PS) -> 深度/模板测试 -> 混合 -> 帧缓冲写回 -> 后处理/显示`。
>
>1. CPU应用阶段（Application Stage）
>- 标准解释：CPU 更新相机/动画/材质参数，准备顶点缓冲、索引缓冲、纹理、常量缓冲，并提交 DrawCall 与管线状态给 GPU。
>- 通俗解释：CPU 是“导演”，先把演员、道具、机位和拍摄清单都准备好，再让 GPU 开工。
>
>2. 输入装配（Input Assembler, IA）
>- 标准解释：按顶点布局从 Vertex Buffer 取数据，按 Index Buffer 与拓扑（点/线/三角形）组装 Primitive。
>- 通俗解释：把一堆散零件按说明书拼成“可加工的三角形骨架”。
>
>3. 顶点着色（Vertex Shader, VS）
>- 标准解释：逐顶点执行。核心是坐标变换（模型空间->世界空间->观察空间->裁剪空间），并输出后续插值要用的属性（法线、UV、颜色等）。
>- 通俗解释：给每个顶点算“它在镜头里应该出现在哪”，再贴上后面上色要用的标签。
>
>4. 曲面细分（Tessellation，常为可选）
>- 标准解释：把粗三角形在 GPU 上细分成更多小三角形，常用于地形、曲面细节与 LOD。细分级别由控制因子决定。
>- 通俗解释：原来网格太粗，近看会糙；这一步像“自动加密网格”，让表面更细腻。
>
>5. 几何着色（Geometry Shader, GS，常为可选）
>- 标准解释：输入一个图元，输出零个或多个图元；可做图元扩增、裁剪、特殊可视化。实务中因吞吐开销常谨慎使用。
>- 通俗解释：像图元加工站，一块三角形进来，能改造后吐出更多形状，但加工费通常不低。
>
>6. 图元装配 + 裁剪 + 剔除
>- 标准解释：
>- 图元装配：把顶点按拓扑关系组织成完整图元。
>- 裁剪（Clipping）：裁掉视锥外部分。
>- 面剔除（Face Culling）：常见背面剔除，减少无效片元。
>- 通俗解释：看不到的先剪掉，背对镜头的先扔掉，避免后面白算。
>
>7. 光栅化（Rasterization）
>- 标准解释：把连续几何图元离散成屏幕上的片元（fragment），并对顶点输出属性做插值（含透视校正插值）。
>- 通俗解释：把“数学三角形”翻译成“屏幕像素候选清单”，并给每个候选像素补齐属性。
>
>8. 片元/像素着色（Fragment/Pixel Shader, FS/PS）
>- 标准解释：逐片元执行，完成纹理采样、法线映射、BRDF/PBR 光照、阴影查询等，输出颜色或多渲染目标。
>- 通俗解释：真正上色的车间，决定每个像素看起来是金属、塑料、粗糙还是光滑。
>
>9. 深度/模板测试（Depth/Stencil Test）
>- 标准解释：
>- 深度测试决定“谁挡住谁”；
>- 模板测试用于遮罩/轮廓等规则控制；
>- Early-Z 能在片元着色前尽早淘汰被遮挡片元，降低像素着色开销。
>- 通俗解释：先做“可见性安检”，被挡住的像素直接淘汰，省下大量着色时间。
>
>10. 混合（Blending）与写回（ROP/Framebuffer）
>- 标准解释：将当前片元颜色与目标缓冲已有颜色按混合方程合成（透明常用 `SrcAlpha/OneMinusSrcAlpha`），再写入颜色/深度缓冲。
>- 通俗解释：不是每次都覆盖旧颜色，透明物体会按比例“叠色”。
>
>11. 后处理与显示（Post Process + Present）
>- 标准解释：在最终图像上做 Bloom、色调映射、TAA、调色等，再通过交换链提交到屏幕显示。
>- 通俗解释：成片后再做调色和特效，最后把这一帧送上屏幕。
>
>- 面试补充要点：
>- 可编程阶段主要是 VS/TS/GS/FS；光栅化和 ROP 等偏固定功能。
>- 性能瓶颈常在“高开销片元着色 + 过度绘制”，Early-Z、剔除与批次组织都很关键。

>[!quote]- 通俗易懂的理解
>- 一句话：GPU 渲染管线就是“把3D场景压成2D像素”的装配线。
>- 前半段（IA/VS/装配裁剪）更像“搭骨架、定位置”；中段（光栅化）是“把形状映射到像素网格”；后半段（FS/测试/混合）是“给像素上妆并决定最终显示”。
>- 你可以把它想成电影制作：
>- CPU 是制片组（排计划）。
>- VS 是摄影测量（定机位和点位）。
>- 光栅化是把场景切成可处理的像素片。
>- FS 是调色师和灯光师。
>- 深度测试是遮挡判定员。
>- 混合是后期合成。

>[!example]- 面试回答简版模板
>`GPU 渲染管线是把场景数据变成屏幕像素的流水线：CPU 先准备资源和命令，GPU 依次经过 IA、VS、可选 TS/GS、裁剪剔除、光栅化、FS、深度模板测试和混合写回，最后后处理并显示。每个阶段各司其职：前段决定几何位置，中段把图元离散成片元，后段决定像素外观与可见性。性能优化重点通常在剔除、Early-Z 和片元阶段成本控制。`

## 题目：前向渲染和延迟渲染的区别？各自的优劣？
>[!NOTE]- 标准准确的说法
>- 先给一句话结论：
>- 前向渲染（Forward）= 画物体时直接算光照。
>- 延迟渲染（Deferred）= 先存几何/材质信息（G-Buffer），再统一算光照。
>
>1. 前向渲染完整流程（按一帧看）
>- CPU 更新场景、相机、材质参数、灯光列表并提交 DrawCall。
>- GPU 执行几何阶段（VS/可选TS/可选GS），再光栅化生成片元。
>- 片元阶段直接做材质采样 + 光照累计（受影响的灯逐个计算）。
>- 通过深度/模板测试后混合并写入颜色缓冲。
>- 最后做后处理并 Present。
>
>2. 延迟渲染完整流程（按一帧看）
>- CPU 同样先提交几何、材质、灯光数据。
>- Pass 1（Geometry Pass）：
>- 只写 G-Buffer，不做完整光照。
>- 常见写入：法线、深度、反照率、粗糙度、金属度、AO、材质ID（按项目选）。
>- Pass 2（Lighting Pass）：
>- 在屏幕空间读取 G-Buffer，统一计算光照并输出结果。
>- Pass 3（常见混合策略）：
>- 透明物体单独走 Forward pass。
>- 然后再进行后处理与 Present。
>
>3. 真正不同点（核心对照）
>- 光照计算时机不同：
>- Forward：在“画物体时”就算光。
>- Deferred：在“几何完成后”统一算光。
>- 材质与光照耦合度不同：
>- Forward：耦合更强（一个 pass 内同时处理材质与光照）。
>- Deferred：先记录材质，再做光照，解耦更明显。
>- 中间数据流不同：
>- Forward：中间缓冲相对轻。
>- Deferred：引入 G-Buffer 的大规模写入/读取。
>- 透明处理难度不同：
>- Forward：天然适配排序混合。
>- Deferred：透明通常需要额外前向通道。
>- MSAA 适配成本不同：
>- Forward：相对直接。
>- Deferred：与 G-Buffer 结合成本更高。
>
>4. 为什么会有这些优劣（代价来源）
>- Forward 的代价：
>- 当动态光源很多时，片元阶段会出现大量重复光照计算。
>- Deferred 的代价：
>- G-Buffer 的写入与读取带来显存带宽与存储压力。
>- 因此不是“谁绝对更快”，而是“瓶颈位置不同”：
>- Forward 更容易在多灯片元阶段吃紧。
>- Deferred 更容易在带宽与内存系统吃紧。
>
>5. 各自优点
>- Forward 优点：
>- 流程简单，工程实现与调试成本较低。
>- 透明支持自然。
>- 带宽压力通常低于重型 G-Buffer 方案。
>- Deferred 优点：
>- 多光源扩展性好。
>- 光照组织清晰，便于做屏幕空间光照策略。
>
>6. 各自缺点
>- Forward 缺点：
>- 多光源场景下像素着色成本容易上升。
>- Deferred 缺点：
>- G-Buffer 带来显存和带宽开销。
>- 透明和 MSAA 处理复杂度更高。
>
>7. 平台与工程常见选型
>- 移动端（带宽/功耗敏感）：常见 `Forward/Forward+`。
>- 桌面端（大量动态光）：常见 `Deferred` 或 `Tiled/Clustered Deferred`。
>- 真实项目常见混合：
>- 不透明走 Deferred，透明走 Forward。
>
>8. 常见误区澄清
>- 误区：Deferred 一定比 Forward 快。
>- 更准确：性能取决于光源密度、分辨率、透明比例、带宽上限与硬件架构。
>- 误区：Forward 不适合复杂画面。
>- 更准确：可做复杂画面，但多光源下成本管理难度通常更高。
>- 误区：Deferred 能天然处理所有透明。
>- 更准确：透明常需单独前向路径或 OIT 方案。

>}

>[!quote]- 通俗易懂的理解
>- 前向渲染像“边上菜边结账”：每上一道菜（每个物体像素）就现场算税费（灯光），流程直观，但菜和桌都多时前台会很忙。
>- 延迟渲染像“先记账再统一结算”：先把每桌点单录系统（G-Buffer），最后统一结账（Lighting Pass），多桌多菜时更容易统一管理。
>- 为什么延迟多光源常更稳：
>- 灯光从“每个物体上重复算”转成“按屏幕统一算”。
>- 为什么延迟对透明麻烦：
>- 透明要按前后顺序一层层叠，而延迟是“先存后统一结算”，思路不一致。
>- 工程里最常见不是二选一，而是混搭：
>- 不透明吃延迟优势，透明回前向处理。

>[!example]- 面试回答简版模板
>`前向渲染是在绘制物体时直接做光照，优点是流程简单、透明友好、带宽压力小，缺点是多光源下片元成本高。延迟渲染先写 G-Buffer 再统一做光照，多光源扩展性好，但显存带宽开销大，透明和 MSAA 更复杂。项目里常用混合管线：不透明 Deferred，透明 Forward，再按平台和光源复杂度做取舍。`

## 题目：延迟渲染怎么处理透明物体？
>[!NOTE]- 标准准确的说法
>- 先说结论：
>- 纯 Deferred 对透明天然不友好，因为透明依赖“按前后顺序逐层混合”，而 Deferred 是“先记 G-Buffer 再统一光照”，两者数据流模型不一致。
>- 工程上通常采用“多方案并存”，按画质/性能/复杂度选型。下面按统一维度对比。
>
>1. 方案 A：混合管线（不透明 Deferred + 透明 Forward）
>- 核心原理：
>- 不透明几何用 Deferred（享受多光源优势）；透明物体单独走 Forward，并按从远到近排序混合。
>- 实现流程：
>- Pass 1：不透明写 G-Buffer + Deferred Lighting。
>- Pass 2：透明物体排序后 Forward 渲染，使用深度测试与 alpha blending。
>- 优点：
>- 工程最成熟，兼容性好，调试成本低。
>- 透明结果可控，易与现有材质系统对接。
>- 缺点：
>- 透明层多、排序不稳定时会出现视觉伪影（排序错误）。
>- 透明物体仍承受前向成本。
>- 适用场景：
>- 大多数商业实时渲染项目（游戏/引擎）默认方案。
>- 面试一句话：
>- `这是工业默认解：不透明吃 Deferred 优势，透明回 Forward 解决顺序混合。`
>
>2. 方案 B：Weighted Blended OIT（加权混合 OIT）
>- 核心原理：
>- 不做严格排序，而是用“加权累计颜色 + 透射率”近似多层透明叠加，达到 Order-Independent Transparency（近似）。
>- 实现流程：
>- 透明片元写入两个累计缓冲（如 accum/revealage）。
>- 末尾做一次合成 pass 得到最终透明颜色。
>- 优点：
>- 不需要精确排序，性能通常比链表类 OIT 更稳。
>- 易并行，适合实时场景的大量半透明粒子。
>- 缺点：
>- 物理正确性是近似，厚玻璃/高对比叠层可能偏差明显。
>- 参数权重调不好会发灰或过亮。
>- 适用场景：
>- 粒子、烟雾、体积感半透明等“可近似”的效果。
>- 面试一句话：
>- `Weighted OIT 用近似替代排序，换来更稳定性能，但不是严格物理正确。`
>
>3. 方案 C：A-Buffer / Per-Pixel Linked List（PPLL）
>- 核心原理：
>- 每个像素保存一组透明片元（颜色、深度、alpha 等），再按深度排序后精确混合。
>- 实现流程：
>- 透明 pass 把片元 append 到像素链表或片元数组。
>- resolve pass 对每像素片元排序并前后合成。
>- 优点：
>- 透明层次表达更准确，复杂重叠场景质量高。
>- 对排序依赖小，理论上更“正确”。
>- 缺点：
>- 显存占用和带宽压力大，原子操作与同步开销高。
>- 实现复杂，性能波动明显（场景相关）。
>- 适用场景：
>- 高质量需求、透明层复杂且可接受较高成本的场景。
>- 面试一句话：
>- `A-Buffer 类方案质量高但昂贵，本质是用内存和复杂度换精确透明。`
>
>4. 方案 D：Depth Peeling（深度剥离，或 Dual Depth Peeling）
>- 核心原理：
>- 多次 pass 每次剥离一层前/后表面，逐层重建透明叠加结果。
>- 实现流程：
>- 第 1 次 pass 得到最近层；后续 pass 逐层剥离剩余片元；最终合成。
>- 优点：
>- 透明顺序正确性高，概念直接。
>- 缺点：
>- pass 数多，代价高，实时项目常难承受。
>- 适用场景：
>- 科学可视化、离线预览、透明层较少但要求顺序正确的场合。
>- 面试一句话：
>- `Depth Peeling 质量可靠但非常吃 pass，实时里常是高成本选项。`
>
>5. 如何选型（工程决策）
>- 实时游戏默认：方案 A（混合管线）。
>- 半透明粒子很多：A + B（Forward 透明中引入 Weighted OIT）。
>- 高质量透明重叠：评估 C（A-Buffer/PPLL），必要时局部启用。
>- 对精确层顺序有强要求且预算高：评估 D（Depth Peeling）。
>
>6. 常见问题与误区
>- 误区 1：Deferred 不能做透明。
>- 更准确：可以做，但通常不是“纯 Deferred 一把梭”，而是混合/近似/高成本 OIT。
>- 误区 2：OIT 一定更好。
>- 更准确：OIT 有多种，质量与成本差异巨大，必须按项目预算选。
>- 误区 3：只要排序就能解决全部透明问题。
>- 更准确：自相交、粒子云、大量细碎透明会让排序不稳定，需 OIT 或特化策略。

>// 伪代码：混合管线 + 可选 OIT
>RenderOpaqueDeferred();
>
>if (useWeightedOIT) {
>    RenderTransparentWeightedOIT();
>    ResolveWeightedOIT();
>} else if (usePPLL) {
>    BuildPerPixelLists();
>    ResolvePerPixelLists();
>} else {
>    SortTransparentBackToFront();
>    RenderTransparentForward();
>}

>[!quote]- 通俗易懂的理解
>- 延迟渲染像“先记账后结算”，透明却要求“按层一层层叠加”。
>- 所以工程上常见做法是“分治”：
>- 大头（不透明）走 Deferred，提高多光源效率。
>- 难点（透明）单独处理，按成本选普通前向或 OIT。
>- 你可以把几种方案理解成：
>- 方案 A：最实用的日常通勤车。
>- 方案 B：速度快、精度近似的折中方案。
>- 方案 C：高精度但昂贵的高配方案。
>- 方案 D：最“教科书正确”但实时成本高的方案。

>[!example]- 面试回答简版模板

## 题目：光栅化的定义与作用？算法有哪些？
>[!NOTE]- 标准准确的说法
>**1. 定义（Rasterization 是什么）**
>- 光栅化是把连续几何图元（通常是三角形）离散成屏幕采样网格上片元（fragment）的过程。
>- 它属于“几何 -> 像素候选”的过渡阶段，输出不是最终像素颜色，而是待着色与待测试的片元记录。
>
>**2. 输入（光栅化吃什么数据）**
>- 主要输入是“已完成顶点处理后”的图元数据，而不是原始模型数据。
>- 常见输入内容包括：
>- 图元拓扑：点/线/三角形（最常见三角形）。
>- 顶点位置：已可用于屏幕覆盖判定的位置（经历过投影/视口相关变换）。
>- 顶点属性（varyings）：如 `uv`、法线、颜色、深度相关值等，供后续插值。
>
>**3. 输出（光栅化产出什么）**
>- 输出是一组片元（fragment）候选记录。
>- 每个片元通常包含：
>- 屏幕坐标（或采样点坐标）。
>- 深度值（`z`）。
>- 由顶点属性插值得到的数据（`uv/normal/color` 等）。
>- 覆盖信息（coverage，MSAA 下尤其重要）。
>
>**4. 片元（Fragment）到底是什么形式**
>- 片元可以理解为“像素候选条目”，而不是最终帧缓冲像素。
>- 片元在后续会经过：片元着色 -> 深度/模板测试 -> 混合。
>- 只有通过这些阶段，才可能成为最终显示像素。
>
>**5. 作用（为什么必须有光栅化）**
>- 作用 A：覆盖判定
>- 判断屏幕上哪些采样点被图元覆盖。
>- 作用 B：属性插值
>- 把顶点属性连续地传播到图元内部每个片元，为片元着色器提供输入。
>- 作用 C：桥接几何与像素
>- 把“几何世界（图元）”转换为“像素世界（片元）”，是实时渲染管线核心关口。
>
>**6. 常见算法与实现思路**
>- 边函数（Edge Function）+ 重心坐标（Barycentric）：
>- 现代 GPU 主流数学基础；先判覆盖，再做插值。
>- 扫描线（Scanline）：
>- 经典软件光栅化思路，按行填充图元。
>- 分块/瓦片（Tiled/Binned Rasterization）：
>- 先按屏幕 tile 分桶再光栅化，提高局部性与并行效率（移动 GPU 常见）。
>- 保守光栅化（Conservative Rasterization）：
>- 只要像素与图元有交就算覆盖，常用于可见性与碰撞相关场景。
>
>**7. 易错点（面试高频）**
>- 误区 1：光栅化输出的是最终像素颜色。
>- 更准确：输出是片元候选，最终颜色要后续阶段决定。
>- 误区 2：片元等于像素。
>- 更准确：片元是“像素候选记录”，可能被测试丢弃，MSAA 下还涉及多采样覆盖。

>[!tip]- 代码示例
>```cpp
>// 用结构体帮助理解“光栅化输入/输出”
>struct VertexOut {
>    float4 clipPos;   // 顶点阶段输出位置（后续会用于覆盖判定相关流程）
>    float2 uv;
>    float3 normal;
>    float4 color;
>};
>
>struct Fragment {
>    int x, y;         // 屏幕位置（像素/采样点）
>    float depth;      // 深度值
>    float2 uv;        // 插值属性
>    float3 normal;
>    float4 color;
>    uint coverage;    // 覆盖信息（MSAA 常见）
>};
>
>// 逻辑上可以理解为：Triangle(VertexOut x3) -> 多个 Fragment
>```
>
>```mermaid
>flowchart LR
>  A[图元: 三角形 + 顶点属性] --> B[覆盖判定]

>[!quote]- 通俗易懂的理解
>- 光栅化可以想成“把数学上的三角形，盖到像素方格纸上”。
>- 被三角形盖住的格子，会生成一条“待处理记录”（片元），里面写着这个格子的位置、深度和纹理坐标等信息。
>- 这些记录先交给片元着色器上色，再过深度和混合检查，最后才可能变成你看到的像素。

>[!example]- 面试回答简版模板

## 题目：顶点着色器和片元着色器各自的作用？
>[!NOTE]- 标准准确的说法
>**1. 顶点着色器（Vertex Shader, VS）**
>- 执行粒度：逐顶点执行，每个输入顶点运行一次。
>- 输入：顶点属性（位置、法线、UV、颜色、骨骼权重等）+ 常量参数（MVP 矩阵、骨骼矩阵、相机参数）。
>- 主要职责：
>- 坐标变换（模型空间 -> 世界空间 -> 观察空间 -> 裁剪空间）。
>- 骨骼蒙皮、顶点形变、顶点级动画。
>- 输出后续需要插值的属性（`uv/normal/worldPos/color` 等）。
>- 必要输出：裁剪空间位置（如 `gl_Position`），否则图元无法进入后续光栅化。
>
>**2. 片元着色器（Fragment/Pixel Shader, FS/PS）**
>- 执行粒度：逐片元执行，片元由光栅化生成，数量通常远大于顶点数。
>- 输入：
>- 来自 VS 的插值结果（如 `uv/normal/worldPos`）。
>- 纹理与采样器。
>- 光照、相机、阴影等统一参数。
>- 主要职责：
>- 材质采样（albedo/normal/roughness/metallic 等）。
>- 光照计算（Phong/Blinn/PBR/BRDF）。
>- 局部像素效果计算（透明度、发光、屏幕空间局部逻辑）。
>- 输出：颜色（可多渲染目标 MRT）；可选改写深度（但会影响早期深度优化机会）。
>
>**3. 两者核心区别（面试高频）**
>- 执行对象不同：VS 处理“顶点”，FS 处理“片元”。
>- 目标不同：VS 主要决定“几何点在哪里”；FS 主要决定“像素看起来怎样”。
>- 成本来源不同：VS 更受顶点数影响，FS 更受分辨率、过度绘制、纹理采样与光照复杂度影响。
>- 常见性能瓶颈：现代实时渲染里 FS 更容易成为瓶颈（特别是高分辨率+复杂材质）。
>
>**4. 纹理（Texture）和采样器（Sampler）是什么**
>- 纹理（Texture）：
>- 本质是 GPU 上的“数据图”，不只可存颜色，也可存法线、深度、查找表等任意按格式组织的数据。
>- 常见类型：`Texture2D`、`TextureCube`、`Texture2DArray` 等。
>- 采样器（Sampler）：
>- 本质是“如何读取纹理”的规则集合，不存图像数据本体。
>- 典型采样状态：
>- 过滤方式（最近点/双线性/三线性/各向异性）。
>- 地址模式（Repeat/Clamp/Mirror）。
>- Mipmap 级别选择与 LOD 偏置。
>- 为什么要分开：
>- 同一张纹理可搭配不同采样器读取（比如一处用线性过滤，一处用点采样），提高复用与灵活性。
>
>**5. 常见误区澄清**
>- 误区 1：FS 输出就一定是最终屏幕像素。
>- 更准确：FS 输出还要经过深度/模板测试和混合，可能被丢弃或与目标颜色合成。
>- 误区 2：VS 只负责把模型“乘矩阵”。
>- 更准确：VS 还常负责顶点级动画、蒙皮、位移和向 FS 提供关键插值数据。
>- 误区 3：纹理就是图片，采样器就是函数。
>- 更准确：纹理是数据资源，采样器是读取策略状态，二者是不同资源对象。

>[!tip]- 代码示例
>```glsl
>// Vertex Shader（简化）
>layout(location=0) in vec3 aPos;
>layout(location=1) in vec3 aNormal;
>layout(location=2) in vec2 aUV;
>
>uniform mat4 M, V, P;
>
>out vec3 vNormalWS;
>out vec2 vUV;
>out vec3 vPosWS;
>
>void main() {
>    vec4 posWS = M * vec4(aPos, 1.0);
>    vPosWS = posWS.xyz;
>    vNormalWS = mat3(transpose(inverse(M))) * aNormal;
>    vUV = aUV;
>    gl_Position = P * V * posWS; // VS 必须给出裁剪空间位置
>}
>```
>
>```glsl
>// Fragment Shader（简化）
>in vec3 vNormalWS;
>in vec2 vUV;
>in vec3 vPosWS;
>
>uniform sampler2D albedoTex; // 纹理+采样器绑定后的采样入口（GLSL 语义）
>uniform vec3 lightPos;
>
>out vec4 FragColor;
>
>void main() {
>    vec3 albedo = texture(albedoTex, vUV).rgb; // 按采样规则读取纹理
>    vec3 N = normalize(vNormalWS);
>    vec3 L = normalize(lightPos - vPosWS);
>    float ndotl = max(dot(N, L), 0.0);
>    FragColor = vec4(albedo * ndotl, 1.0);
>}
>```
>
>```cpp
>// API 视角（伪代码）：纹理对象与采样器对象分离
>Texture2D albedo = LoadTexture("albedo.png");
>Sampler linearWrap = CreateSampler(Filter::Linear, Address::Repeat);
>BindTexture(0, albedo);
>BindSampler(0, linearWrap);
>Draw();
>```

>[!quote]- 通俗易懂的理解
>- VS 像“布景师+定位师”：先把模型顶点放到镜头该出现的位置，并给每个点贴上标签（UV、法线等）。
>- FS 像“上色师+打光师”：拿到每个像素候选的标签，去查纹理、算光照，决定这个像素应该长什么样。
>- 纹理是“素材库里的图/数据”，采样器是“读取素材的规则”（怎么缩放、到边界怎么处理、用哪级 Mipmap）。

>[!example]- 面试回答简版模板
>`顶点着色器逐顶点执行，负责坐标变换与插值数据准备；片元着色器逐片元执行，负责纹理采样与光照材质计算。纹理是数据资源本体，采样器是读取规则状态，二者配合决定最终像素外观与性能表现。`

## 题目：G-Buffer存储什么内容？格式是什么？
>[!NOTE]- 标准准确的说法
>**1. G-Buffer 是什么**
>- G-Buffer（Geometry Buffer）是延迟渲染里“按像素存储几何与材质信息”的缓冲集合。
>- 目标是让光照 pass 在不回看原始几何的情况下，直接按屏幕像素完成光照计算。
>
>**2. 常见存储内容 + 具体用途 + 常见格式**
>- `Depth（深度）`：
>- 用途：重建位置（世界/视空间）、做屏幕空间效果（SSAO/SSR）与可见性相关计算。
>- 常见格式：`D24S8`、`D32F`、或线性深度 `R32F`。
>- `Normal（法线）`：
>- 用途：决定受光方向关系（`N·L`、反射方向、BRDF 几何项）。
>- 常见格式：`RGB10A2`、`RG16_SNORM`（配合法线压缩/重建）、`RGBA8_SNORM`。
>- `Albedo / BaseColor（反照率/底色）`：
>- 用途：材质本色输入，参与漫反射与金属工作流中的基色定义。
>- 常见格式：`RGBA8_UNORM`（常见），高动态需求可用 `RGBA16F`。
>- `Roughness（粗糙度）`：
>- 用途：控制高光锐利程度与反射模糊程度。
>- 常见格式：`R8_UNORM`（常与其他参数打包）。
>- `Metallic（金属度）`：
>- 用途：区分金属/非金属，影响 F0 与能量分配。
>- 常见格式：`R8_UNORM`。
>- `AO（环境遮蔽）`：
>- 用途：压低缝隙与遮蔽区域的间接光，增强接触层次。
>- 常见格式：`R8_UNORM`。
>- `Emissive（自发光）`：
>- 用途：表示材质自身发光，不依赖外部灯光。
>- 常见格式：`RGB8`、`R11G11B10F`、`RGBA16F`（看亮度范围）。
>- `Material ID / Flags（材质ID/标记）`：
>- 用途：后续 pass 分支控制（例如不同材质模型、特效掩码、调试分类）。
>- 常见格式：`R8_UINT`、`R16_UINT`。
>
>**3. 这些字段在延迟光照里怎么被用到**
>- 光照阶段通常会：
>- 从 `Depth` 重建像素位置 `P`。
>- 从 `Normal` 取法线 `N`。
>- 从 `BaseColor/Roughness/Metallic/AO` 构建材质参数。
>- 结合灯光与视角，代入 BRDF/PBR 公式得到结果颜色。
>- 所以 G-Buffer 不是“存图而已”，而是“光照计算输入表”。
>
>**4. 格式选型原则（为什么这么选）**
>- 原则 A：精度够用即可，避免过度位宽。
>- 原则 B：优先减少带宽（延迟渲染瓶颈常在读写 G-Buffer）。
>- 原则 C：尽量参数打包与压缩（如法线编码、MRA 打包）。
>- 原则 D：能由 `Depth` 重建的位置就不直接存 `Position`，省显存与带宽。
>
>**5. 常见布局示例（PBR，示意）**
>```text
>Depth: D32F
>
>RT0 (RGBA8): BaseColor.rgb + AO.a
>RT1 (RGBA8): Normal.xy(编码) + Roughness + Metallic
>RT2 (RGBA8): Emissive.rgb + MaterialID
>```
>- 说明：这是“示意布局”，真实项目会按平台、目标帧率和画质要求调整。
>
>**6. 常见误区（面试易错）**
>- 误区 1：G-Buffer 存得越全越好。
>- 更准确：字段越多越吃带宽，应该按需求最小化。
>- 误区 2：直接存世界坐标最方便。
>- 更准确：通常用 `Depth` 重建更省资源。
>- 误区 3：延迟渲染瓶颈主要在算术。
>- 更准确：很多场景下瓶颈在 G-Buffer 的读写与内存系统。

>[!tip]- 代码示例
>```cpp
>// 延迟光照 pass（伪代码）：展示 G-Buffer 字段如何被使用
>for each pixel {
>    float depth      = gDepth[pixel];
>    float3 normal    = DecodeNormal(gNormal[pixel]);
>    float3 baseColor = gBaseColor[pixel].rgb;
>    float roughness  = gParam[pixel].r;
>    float metallic   = gParam[pixel].g;
>    float ao         = gParam[pixel].b;
>
>    float3 P = ReconstructPosition(depth, pixel, invVP);
>    float3 V = normalize(cameraPos - P);
>
>    float3 Lo = EvaluatePBR(normal, V, baseColor, roughness, metallic, lights);
>    float3 color = Lo * ao + gEmissive[pixel].rgb;
>    outColor[pixel] = float4(color, 1.0);
>}
>```
>
>```mermaid
>flowchart LR
>  A[G-Buffer: depth/normal/baseColor/params] --> B[重建位置与法线]
>  B --> C[BRDF/PBR光照计算]
>  C --> D[叠加AO/Emissive]
>  D --> E[输出最终颜色]
>```

>[!quote]- 通俗易懂的理解
>- G-Buffer 可以理解为“每个像素的病例档案”。
>- 光照 pass 像医生查档案：先看位置和法线，再看材质参数，最后开出该像素的明暗与颜色结果。
>- 档案太少会算不准，档案太多又会拖慢系统，所以要做“够用且精简”的平衡。

>[!example]- 面试回答简版模板
>`G-Buffer 是延迟渲染的像素级几何材质档案，常存深度、法线、底色、粗糙度、金属度、AO、发光和材质ID。光照 pass 读取这些字段重建位置并做 BRDF 计算。格式设计本质是精度、带宽和显存三者权衡，工程上常用参数打包和法线压缩来降成本。`

## 题目：deferred shading和deferred lighting的区别？
>[!NOTE]- 标准准确的说法
>**1. 先给一句结论**
>- `Deferred Shading`：G-Buffer 里通常直接存几何 + 材质参数，光照 pass 里可直接完成“光照+材质”求值。
>- `Deferred Lighting`（Light Pre-Pass）：先用较瘦的几何信息算光照项，再在后续材质 pass 结合 albedo/spec 等得到最终颜色。
>
>**2. Deferred Shading 的典型流程**
>- Pass 1（Geometry Pass）：
>- 写入 G-Buffer：常见有 `depth/normal/albedo/roughness/metallic/AO...`。
>- Pass 2（Lighting Pass）：
>- 读取 G-Buffer，直接做 BRDF/PBR 计算并输出结果。
>- 核心特征：
>- 材质信息在光照 pass 已可用，流程直观。
>
>**3. Deferred Lighting 的典型流程（Light Pre-Pass）**
>- Pass 1（Geometry Pass）：
>- 写“瘦 G-Buffer”：通常以 `depth/normal` 为主，尽量不放完整材质。
>- Pass 2（Light Pass）：
>- 先计算光照项（例如 diffuse/spec lighting term）。
>- Pass 3（Material Pass）：
>- 再读取材质（albedo/spec/roughness 等）与光照项合成最终颜色。
>- 核心特征：
>- 光照与材质进一步拆分，G-Buffer 可减压，但 pass 更复杂。
>
>**4. 两者关键区别（逐点对照）**
>- 区别 A：材质参与时机
>- Deferred Shading：光照时就参与。
>- Deferred Lighting：先光照、后材质。
>- 区别 B：G-Buffer 内容规模
>- Deferred Shading：通常更“胖”。
>- Deferred Lighting：通常更“瘦”。
>- 区别 C：Pass 数量与管线复杂度
>- Deferred Shading：常见 2 大步（几何+光照）。
>- Deferred Lighting：常见 3 大步（几何+光照+材质）。
>- 区别 D：代价位置
>- Deferred Shading：更偏 G-Buffer 带宽压力。
>- Deferred Lighting：更偏流程复杂度与额外 pass 开销。
>
>**5. 为什么会有 Deferred Lighting**
>- 历史动机是减轻“胖 G-Buffer”的存储与带宽压力。
>- 通过把材质参与后移，换取 G-Buffer 压缩空间。
>- 代价是管线复杂、调试难度和维护成本上升。
>
>**6. 现代项目中的实践倾向**
>- 在 PBR 工作流下，很多引擎更常见 Deferred Shading（或其 tiled/clustered 变体）。
>- Deferred Lighting 仍有价值，但更多是特定约束下的取舍方案。
>
>**7. 常见误区**
>- 误区 1：Deferred Lighting 一定更快。
>- 更准确：要看目标硬件、带宽瓶颈、pass 开销和材质复杂度。
>- 误区 2：两者只是命名不同。
>- 更准确：它们在“材质参与时机”和“数据流组织”上有实质差别。

>[!tip]- 代码示例
>```mermaid
>flowchart LR
>  subgraph DS[Deferred Shading]
>    A1[Geometry Pass\n写 depth/normal/albedo/roughness/metallic...] --> A2[Lighting Pass\n直接算完整BRDF/PBR]
>    A2 --> A3[Final Color]
>  end
>
>  subgraph DL[Deferred Lighting / Light Pre-Pass]
>    B1[Geometry Pass\n写 depth/normal(瘦G-Buffer)] --> B2[Light Pass\n先算光照项]
>    B2 --> B3[Material Pass\n再和材质合成]
>    B3 --> B4[Final Color]
>  end
>```
>
>```cpp

>// Deferred Shading

>LightingPass_FullPBRFromGBuffer();
>
>// Deferred Lighting

>LightPrePass_ComputeLightingTerms();

>[!quote]- 通俗易懂的理解
>- Deferred Shading 像“手里已有完整菜谱+食材信息，结账时一次算总价”。
>- Deferred Lighting 像“先算服务费和税费，再回头套每桌菜谱细项”。
>- 前者更直接，后者更拆分；前者偏带宽压力，后者偏流程复杂度。

## 题目：几何着色器的作用？
>[!NOTE]- 标准准确的说法
>**1. GS 在管线中的位置**
>- 几何着色器（Geometry Shader, GS）位于“顶点处理之后、光栅化之前”。
>- 它处理的单位是“图元（point/line/triangle + adjacency）”，不是单个顶点、也不是片元。
>
>**2. GS 的输入与输出**
>- 输入：一个图元（包含该图元的多个顶点及其插值属性）。
>- 输出：零个、一个或多个图元（可改变图元数量、拓扑类型）。
>- GS 能做的事包括：
>- 丢弃图元（输出 0 个）。
>- 扩增图元（一个输入生成多个输出）。
>- 附加/改写顶点属性（为后续光栅化与片元阶段准备数据）。
>
>**3. GS 典型用途（为什么会用）**
>- 图元级可视化：法线线段显示、线框辅助、调试渲染。
>- 屏幕朝向面片（billboard）生成：例如把一个点扩展成面片。
>- 某些阴影体/轮廓体构建（历史项目中出现较多）。
>- Layered Rendering：向立方体贴图或多层渲染目标输出。
>
>**4. GS 的主要问题（为什么很多项目少用）**
>- 吞吐瓶颈：
>- GS 位于几何到光栅化关键路径，图元扩增会快速放大后续成本。
>- 并行效率受限：
>- 相比 VS/FS，GS 在很多硬件上的扩展性和并行利用率不理想。
>- 性能波动大：
>- 场景相关性强，复杂图元放大时容易掉帧。
>
>**5. 工程替代方案（高频面试点）**
>- `实例化（Instancing）`：常用于可重复几何扩展。
>- `Compute Shader`：更灵活做几何预处理、剔除、粒子生成。
>- `Tessellation`：适用于曲面细分类需求。
>- `Mesh/Task Shader（新管线）`：在支持的平台上更现代的几何组织方式。
>
>**6. 结论（什么时候该用 GS）**
>- 小规模图元级效果、调试可视化、平台验证通过时可用。
>- 大规模几何放大与性能敏感路径通常优先评估 Instancing/Compute 等替代。

>[!tip]- 代码示例
>```glsl
>// 简化示例：输入一个三角形，原样输出
>layout(triangles) in;
>layout(triangle_strip, max_vertices = 3) out;
>
>in vec3 vColor[];
>out vec3 gColor;
>
>void main() {
>    for (int i = 0; i < 3; ++i) {
>        gl_Position = gl_in[i].gl_Position;
>        gColor = vColor[i];
>        EmitVertex();
>    }
>    EndPrimitive();
>}
>```
>
>```cpp
>// 伪代码：何时谨慎启用 GS
>if (featureNeedPrimitiveAmplification && perfBudgetOK) {
>    EnableGeometryShader();
>} else {
>    UseInstancingOrComputePath();
>}
>```

>[!quote]- 通俗易懂的理解
>- GS 像“图元加工站”：前面产线把三角形送来，这一站可以删掉、复制、改造后再送下去。
>- 它的优点是灵活，缺点是这站常常不够快；一旦在这里放大图元，后面所有工序都会更重。
>- 所以工程里通常把 GS 当“特种工具”，而不是默认主路径。

>[!example]- 面试回答简版模板
>`几何着色器位于顶点后、光栅化前，按图元工作，可丢弃或扩增图元并改写顶点属性，适合图元级特效和可视化。但它在很多平台吞吐与扩展性一般，图元放大时成本高。工程上常优先评估实例化、compute 或 mesh shader 替代。`

## 题目：可编程管线分几个阶段？
>[!NOTE]- 标准准确的说法
>**1. 先给标准答案（经典光栅图形管线）**
>- 常见可编程着色阶段是 5 类：
>- VS（Vertex Shader）
>- TCS/Hull（Tessellation Control Shader，可选）
>- TES/Domain（Tessellation Evaluation Shader，可选）
>- GS（Geometry Shader，可选）
>- FS/PS（Fragment/Pixel Shader）
>- 其中“最小必备”通常是 VS + FS；细分与几何阶段按需求启用。
>
>**2. 每个可编程阶段在做什么**
>- VS：逐顶点处理。坐标变换、蒙皮、顶点属性准备。
>- TCS（Hull）：控制细分因子，决定图元细分密度。
>- TES（Domain）：在细分后生成新顶点并求值位置/属性。
>- GS：逐图元处理，可丢弃或扩增图元。
>- FS（Pixel）：逐片元处理，做纹理采样、材质与光照计算。
>
>**3. 可编程阶段与固定功能阶段的关系**
>- 可编程阶段之间穿插固定功能步骤：
>- 输入装配（IA）、图元装配、裁剪、光栅化、深度/模板测试、混合等。
>- 因此“可编程管线”不等于“全流程都可编程”。
>
>**4. API 术语对照（面试常问）**
>- OpenGL：VS / TCS / TES / GS / FS。
>- DirectX：Vertex / Hull / Domain / Geometry / Pixel。
>- Vulkan：与 OpenGL 命名相近（并支持更多现代扩展）。
>- 名称不同但语义基本对应。
>
>**5. Compute Shader 算不算可编程管线阶段**
>- Compute Shader 是 GPU 可编程阶段，但通常不属于“经典图形光栅管线的串行阶段”。
>- 它是并行计算旁路，可与图形管线协同（如剔除、后处理、粒子/可见性预处理）。
>
>**6. 现代扩展（可选补充）**
>- 一些现代平台支持 Task/Mesh Shader 路径，用来替代传统 VS+可选GS 的部分几何组织流程。
>- 面试中说“经典 5 类 + 现代可扩展路径”是最稳妥表述。
>
>**7. 常见误区**
>- 误区 1：可编程管线永远是 5 个阶段都启用。
>- 更准确：通常只启 VS+FS，其他按需求打开。
>- 误区 2：Compute Shader 属于 VS->FS 中的某一步。
>- 更准确：Compute 是并行计算管线，不是经典图形阶段链路中的固定一环。
>- 误区 3：API 名字不同就是机制不同。
>- 更准确：多数只是术语差异，核心职责相同。

>[!tip]- 代码示例
>```mermaid
>flowchart LR
>  A[Vertex Shader] --> B[Tess Control 可选]
>  B --> C[Tess Eval 可选]
>  C --> D[Geometry 可选]
>  D --> E[Rasterization 固定功能]
>  E --> F[Fragment/Pixel Shader]
>  F --> G[Depth/Stencil/Blend 固定功能]
>```
>
>```cpp
>// 伪代码：按特性选择可编程阶段
>EnableVS();
>EnableFS();
>
>if (needTessellation) {
>    EnableTCS();
>    EnableTES();
>}
>
>if (needPrimitiveAmplification) {
>    EnableGS();
>}
>
>// 可选旁路计算
>if (needGPUPreprocess) {
>    DispatchCompute();
>}
>```

>[!quote]- 通俗易懂的理解
>- 可编程管线可以理解成“你能写 shader 的工位”。
>- 经典必经的核心工位是 VS 和 FS；细分和几何是“可插拔工位”。
>- Compute 像“旁边的并行加工车间”，不一定在这条主装配线里，但会经常配合主线使用。

>[!example]- 面试回答简版模板
