## 题目：HDR和LDR是什么？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说人话**
>- LDR（Low Dynamic Range）像“小水杯”，亮度范围有限。
>- HDR（High Dynamic Range）像“大水桶”，可以容纳更亮的高光和更暗的细节。
>- 
>- **LDR 和 HDR 的本质差异**
>- LDR 常见是 8bit 颜色表示，范围大致在显示可见亮度区间。
>- HDR 常用 `FP16/FP32` 渲染目标，允许远超 1.0 的线性亮度值。
>- 
>- **为什么实时渲染更偏 HDR 流程**
>- 物理光照累加时会出现很大动态范围：
>- 太阳高光、金属反射可远高于暗部。
>- 如果直接在 LDR 累加，容易“高光死白、暗部丢细节”。
>- 
>- **现代管线典型流程**
>- 1. 在线性 HDR 空间做光照。
>- 2. 后处理（Bloom、曝光、体积光等）。
>- 3. Tone Mapping 压到显示范围。
>- 4. 再做 gamma/sRGB 输出。
>- 
>- **常见坑点**
>- 把 HDR 和“看起来更亮”混为一谈。
>- HDR 是“数据和计算范围更大”，不等于一定过曝。

>[!tip]- 代码示例
>```text
>Pipeline: Linear HDR Lighting -> PostFX -> Tone Mapping -> Gamma Encode -> Display
>```

>[!quote]- 通俗易懂的理解
>- HDR 不是特效，而是“先把真实亮度算对”的基础条件。

>[!example]- 面试回答简版模板
>`HDR和LDR的核心区别是可表示亮度范围。实时渲染通常先在HDR线性空间完成光照累加，再经过Tone Mapping和Gamma输出到显示设备。这样能同时保留高光和暗部层次。`

## 题目：色调映射(Tone Mapping)是什么？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说人话**
>- Tone Mapping 是把“超大亮度范围的 HDR”压缩到“屏幕能显示的范围”，同时尽量保持观感自然。
>- 
>- **为什么必须做 Tone Mapping**
>- 显示器动态范围有限。
>- HDR 里很多值可能远大于 1，不能直接显示。
>- 不压缩会出现严重裁剪（clamp），画面死白。
>- 
>- **典型数学形式（简化）**
>- 先曝光：
> $$
> C_{exp} = exposure \cdot C_{hdr}
> $$
>- 再应用 tone curve（例如 Reinhard）：
> $$
> C_{ldr} = \frac{C_{exp}}{1 + C_{exp}}
> $$
>- 最后 gamma/sRGB 编码输出。
>- 
>- **常见曲线**
>- Reinhard：简单稳定，易实现。
>- ACES 近似：电影感更强，工业项目常见。
>- Filmic：高光 roll-off 更自然，避免硬截断。
>- 
>- **曝光（Auto Exposure）与 Tone Mapping 的关系**
>- 曝光决定“整体亮度基准”。
>- Tone Mapping 决定“压缩形状和高光风格”。
>- 两者常配合使用。
>- 
>- **常见坑**
>- 把 Bloom 当 Tone Mapping。
>- Bloom 是光晕特效；Tone Mapping 是亮度映射基础流程。

>[!tip]- 代码示例
>```mermaid
>flowchart LR
>  A[HDR Color] --> B[Exposure]
>  B --> C[Tone Curve]
>  C --> D[Gamma/sRGB Encode]
>  D --> E[LDR Display]
>```

>[!quote]- 通俗易懂的理解
>- Tone Mapping 就像“压缩动态范围的音量旋钮”，让最亮和最暗都还能听见细节。

>[!example]- 面试回答简版模板
>`Tone Mapping是HDR到显示范围的非线性压缩过程，通常先曝光再套曲线（Reinhard/ACES/Filmic），最后做Gamma输出。它决定高光保留和整体观感风格。`

## 题目：Gamma校正是什么？为什么需要？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说结论**
>- 光照计算必须在“线性空间”做，显示输出才做 gamma/sRGB 编码。
>- 
>- **为什么需要**
>- 显示设备亮度响应是非线性的。
>- 人眼感知也接近非线性。
>- 如果在非线性颜色上直接做加法/插值，结果会偏暗或不自然。
>- 
>- **核心关系（近似）**
>- 线性到显示：
> $$
> C_{srgb} = C_{linear}^{1/\gamma},\quad \gamma \approx 2.2
> $$
>- 显示到线性：
> $$
> C_{linear} = C_{srgb}^{\gamma}
> $$
>- 
>- **工程正确流程**
>- 纹理读取：
>- 颜色纹理（albedo）走 sRGB 采样自动解码到线性。
>- 数据纹理（normal/roughness/metallic）不能走 sRGB。
>- 光照计算：在线性空间。
>- 最终输出：线性 -> sRGB 编码。
>- 
>- **常见坑**
>- 重复 gamma（双重编码）。
>- normal/roughness 误标 sRGB 导致材质全错。

>[!tip]- 代码示例
>```cpp
>// 概念伪代码
>float3 albedoLinear = SRGBToLinear(albedoTexSample);
>float3 litLinear = DoLighting(albedoLinear, ...);
>float3 outSRGB = LinearToSRGB(litLinear);
>```

>[!quote]- 通俗易懂的理解
>- 计算用“物理直尺”（线性），显示给人看再转“人眼习惯尺”（sRGB）。

>[!example]- 面试回答简版模板
>`Gamma校正的本质是空间管理：线性空间做光照和混合，输出前编码到sRGB。否则会出现亮度和混合失真。颜色纹理与数据纹理的sRGB配置要分开。`

## 题目：Bloom效果的实现？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说人话**
>- Bloom 是“亮部溢光”效果，让非常亮的区域向周围扩散，增强亮度氛围。
>- 
>- **典型实现流程**
>- 1. Bright Pass：提取高亮区域（threshold/knee）。
>- 2. Downsample：构建多级低分辨率金字塔。
>- 3. Blur：每级做可分离高斯或 Kawase 模糊。
>- 4. Upsample + Combine：逐级回叠。
>- 5. 与原图按强度混合。
>- 
>- **为什么要多级金字塔**
>- 大半径模糊更便宜。
>- 同时可形成自然的多尺度光晕。
>- 
>- **常见参数**
>- threshold：多高亮度开始溢光。
>- intensity：Bloom 强度。
>- knee：高亮过渡柔和度。
>- 
>- **与 HDR 的关系**
>- HDR 下 Bloom 更自然（可区分“真正高亮”）。
>- 纯 LDR 下容易整图发灰。
>- 
>- **常见坑**
>- 阈值过低导致全屏泛白。
>- 直接在 gamma 空间做 blur 观感不对。

>[!tip]- 代码示例
>```mermaid
>flowchart TD
>  A[HDR Scene] --> B[Bright Pass]
>  B --> C[Downsample Pyramid]
>  C --> D[Blur Each Level]
>  D --> E[Upsample + Add]
>  E --> F[Composite with Scene]
>```

>[!quote]- 通俗易懂的理解
>- 只让“特别亮”的地方发出柔和光晕，不是给整张图套雾滤镜。

>[!example]- 面试回答简版模板
>`Bloom通常是高亮提取、金字塔降采样模糊、再上采样回叠。和HDR配合可得到更自然的亮部扩散，关键是threshold与强度控制避免画面发灰。`

## 题目：双缓冲和垂直同步？
>[!NOTE]- 标准准确的说法（详细版）
>- **双缓冲是什么**
>- 前缓冲：正在显示。
>- 后缓冲：GPU 正在渲染下一帧。
>- 渲染完成后交换，避免“边画边显示”引发撕裂。
>- 
>- **VSync（垂直同步）是什么**
>- 让缓冲交换对齐显示器垂直刷新时刻。
>- 目标是减少 tearing（画面撕裂）。
>- 
>- **为什么会撕裂**
>- 显示器在扫描当前帧时，GPU 已开始写下一帧。
>- 同一屏幕同时出现两帧内容的拼接。
>- 
>- **VSync 的代价**
>- 输入延迟可能上升。
>- 帧率达不到刷新率时，帧时节奏可能抖动（卡顿感）。
>- 
>- **常见折中方案**
>- Triple Buffering：减轻 VSync 等待阻塞。
>- VRR（G-Sync/FreeSync）：动态匹配刷新率，兼顾撕裂和延迟。

>[!tip]- 代码示例
>```text
>Swapchain Present Modes (Vulkan):
>FIFO  -> 类似VSync，稳定不撕裂，延迟较高
>MAILBOX -> 低延迟、少撕裂，常用于高帧率
>IMMEDIATE -> 最低延迟但可能撕裂
>```

>[!quote]- 通俗易懂的理解
>- 双缓冲解决“画和看同时发生”的冲突；VSync 让换帧踩点更整齐。

>[!example]- 面试回答简版模板
>`双缓冲通过前后缓冲分离显示与渲染，VSync将换帧对齐刷新周期以减少撕裂。代价是可能增加输入延迟，工程上常结合三缓冲或VRR做折中。`

## 题目：蒙特卡洛积分的原理？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说核心公式**
>- 对积分
> $$
> I = \int f(x)\,dx
> $$
>- 蒙特卡洛估计：
> $$
> I \approx \frac{1}{N}\sum_{i=1}^{N}\frac{f(x_i)}{p(x_i)},\quad x_i\sim p(x)
> $$
>- 这里 `p(x)` 是采样概率密度函数（PDF）。
>- 
>- **为什么它有用**
>- 很多图形学积分维度高、解析解难。
>- 蒙特卡洛用随机采样把“难积分”转成“求平均”。
>- 
>- **无偏与方差**
>- 估计器在满足条件时无偏（期望等于真值）。
>- 噪声随样本数下降速度约为：
> $$
> O\left(\frac{1}{\sqrt{N}}\right)
> $$
>- 这意味着“加样本有效，但收益递减”。
>- 
>- **在渲染中的位置**
>- 路径追踪。
>- IBL 预积分。
>- 软阴影/景深/运动模糊采样型实现。
>- 
>- **常见坑**
>- 忘记除以 PDF（结果偏差巨大）。
>- PDF 写错单位（立体角/面积混用）。

>[!tip]- 代码示例
>```cpp
>// Monte Carlo估计（概念）
>float sum = 0.0f;
>for (int i = 0; i < N; ++i) {
>    Sample s = SampleByPDF();   // returns x and pdf p(x)
>    sum += f(s.x) / s.pdf;
>}
>float I = sum / N;
>```

>[!quote]- 通俗易懂的理解
>- 算不动就随机抽样再平均，但“怎么抽”决定你收敛快不快。

>[!example]- 面试回答简版模板
>`蒙特卡洛积分通过随机样本估计高维积分，估计式是平均f(x)/p(x)。它在路径追踪等问题中非常核心，优点是通用性强，难点是方差和降噪。`

## 题目：重要性采样的理解？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说目标**
>- 重要性采样是为了降方差。
>- 思路是让采样分布更贴近高贡献区域，而不是均匀乱采。
>- 
>- **为什么能降噪**
>- 若 `p(x)` 与 `|f(x)|` 形状接近，则 `f(x)/p(x)` 波动更小，方差下降更明显。
>- 
>- **图形学中的典型用法**
>- 按 BRDF 分布采样（镜面 lobes）。
>- 按光源面积/能量采样（Next Event Estimation）。
>- 环境贴图按亮度分布采样。
>- 
>- **与 MIS 的关系**
>- 单一采样策略往往有盲区。
>- MIS（Multiple Importance Sampling）把多种策略混合并加权，稳定性更好。
>- 
>- 常见 MIS 权重（balance heuristic）：
> $$
> w_i = \frac{n_i p_i}{\sum_j n_j p_j}
> $$
>- 
>- **常见坑**
>- 采样分布和 PDF 不一致。
>- 忘了在多策略下用正确权重，导致偏差或噪点异常。

>[!tip]- 代码示例
>```text
>Uniform sampling: 易实现，但噪声大
>Importance sampling: 实现更复杂，但相同样本数下更稳
>MIS: 综合多策略，降低“某一策略失效”风险
>```

>[!quote]- 通俗易懂的理解
>- 把预算花在最可能有贡献的方向上，才能“同样采样数，画面更干净”。

>[!example]- 面试回答简版模板
>`重要性采样通过让PDF贴近高贡献区域来降低方差。渲染中常按BRDF或光源分布采样，并结合MIS平衡多策略，获得更稳定收敛。`

## 题目：TBR和TBDR的了解？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说概念**
>- TBR（Tile-Based Rendering）：把屏幕分成很多 tile 组织渲染。
>- TBDR（Tile-Based Deferred Rendering）：通常先做图元分桶/可见性阶段，再在 tile 内执行延后处理。
>- 
>- **为什么移动 GPU 常用它**
>- 外部显存带宽和功耗昂贵。
>- 把更多计算留在片上 tile memory，可显著降带宽和功耗。
>- 
>- **典型思路**
>- 1. Binning：把图元分配到覆盖到的 tile。
>- 2. Tile Pass：每个 tile 局部处理颜色/深度/混合。
>- 3. 尽量减少中间结果写回外存。
>- 
>- **对渲染管线的启示**
>- 减少不必要的 framebuffer 回写。
>- 合理使用 subpass/input attachment（Vulkan）。
>- 谨慎设计多次全屏 pass，避免带宽炸裂。
>- 
>- **常见坑**
>- 桌面思路直接搬到移动端，导致功耗和带宽异常。
>- 忽略 tile 架构的最优路径，造成性能浪费。

>[!tip]- 代码示例
>```text
>移动端优化倾向：
>1) 尽量合并pass
>2) 减少中间RT写回
>3) 充分利用tile局部性
>```

>[!quote]- 通俗易懂的理解
>- 移动 GPU 更像“分区小厨房做菜”，尽量在厨房内完成，少去仓库搬材料。

>[!example]- 面试回答简版模板
>`TBR/TBDR通过tile局部处理减少外部带宽访问，尤其适合移动GPU。工程上要围绕tile局部性设计渲染流程，减少中间RT读写和不必要全屏pass。`

## 题目：流体模拟/粒子系统的了解？
>[!NOTE]- 标准准确的说法（详细版）
>- **粒子系统（Particle System）**
>- 三个阶段：
>- 1. 发射（Spawn）：位置、速度、寿命初始化。
>- 2. 更新（Update）：力场、碰撞、寿命衰减。
>- 3. 渲染（Render）：sprite/mesh/条带等表现。
>- 
>- **CPU vs GPU 更新**
>- CPU：易调试，适合小规模和复杂逻辑。
>- GPU（Compute）：并行度高，适合大规模粒子特效。
>- 
>- **流体模拟常见两大类**
>- Eulerian（网格法）：在固定网格上解速度场/压力场（如 Stable Fluids）。
>- Lagrangian（粒子法）：跟踪粒子运动（如 SPH）。
>- 
>- **实时项目的真实目标**
>- 不是追求离线级物理精确。
>- 更重视：稳定、可控、可调参、成本可预测。
>- 
>- **工程常见优化**
>- 粒子 LOD（远处减量）。
>- 屏幕空间裁剪。
>- 分桶更新和异步计算。
>- 
>- **常见坑**
>- 粒子透明排序代价高。
>- 粒子数量暴涨导致 fillrate 崩溃。
>- 流体迭代步数过低造成不稳定震荡。

>[!tip]- 代码示例
>```cpp
>// GPU粒子更新（概念）
>// for each particle i in compute shader:
>vel[i] += (gravity + forceField(pos[i])) * dt;
>pos[i] += vel[i] * dt;
>life[i] -= dt;
>if (life[i] <= 0) Respawn(i);
>```

>[!quote]- 通俗易懂的理解
>- 粒子系统是“很多小个体按规则运动”；流体是“让整体满足流动规律”。

>[!example]- 面试回答简版模板
>`粒子系统通常分发射、更新、渲染三步，大规模效果常用GPU并行更新。流体可用网格法或粒子法，实时项目更强调稳定和可控，而非离线级精确。`

## 题目：NPR/卡通渲染的方法？
>[!NOTE]- 标准准确的说法（详细版）
>- **NPR 是什么**
>- NPR（Non-Photorealistic Rendering）目标不是写实，而是风格化表达。
>- 卡通渲染是 NPR 的典型分支。
>- 
>- **常见方法1：Toon Shading（分段光照）**
>- 把连续光照量化成几档明暗：
> $$
> L = \text{step/ramp}(N\cdot L)
> $$
>- 形成漫画式明暗分层。
>- 
>- **常见方法2：描边（Outline）**
>- 几何法：沿法线外扩背面再渲染黑边。
>- 后处理法：深度/法线边缘检测后画线。
>- 
>- **常见方法3：风格纹理与噪声**
>- Ramp 纹理控制明暗色带。
>- 手绘噪声、半色调、笔触贴图增强风格。
>- 
>- **工程关键点**
>- 时序稳定：相机动时边线不要抖。
>- 风格统一：角色、场景、特效的风格语言要一致。
>- 艺术可控：参数可调比物理正确更重要。
>- 
>- **常见坑**
>- 只有 toon 量化但无风格系统，画面会“塑料感”。
>- 描边忽粗忽细、远处闪烁影响观感。

>[!tip]- 代码示例
>```mermaid
>flowchart TD
>  A[基础法线与光照] --> B[Toon量化/Ramp采样]
>  A --> C[Outline(几何或后处理)]
>  B --> D[风格纹理调制]
>  C --> E[边线合成]
>  D --> F[最终NPR输出]
>  E --> F
>```

>[!quote]- 通俗易懂的理解
>- NPR 不是“做得不真实”，而是“有意识地做成特定视觉语言”。

>[!example]- 面试回答简版模板
>`NPR核心是风格化表达。卡通常见组合是Toon分段光照、描边和Ramp纹理，并辅以噪声/笔触等风格元素。工程重点是风格一致性与时序稳定，而不是物理逼真。`

## 一页速记（考前30秒）
>[!NOTE]- 你最少要记住的几句话
>- HDR 是大动态范围计算空间，LDR 是显示范围。
>- Tone Mapping 负责把 HDR 压到可显示范围，曝光和曲线共同决定观感。
>- Gamma 流程：线性计算，sRGB 输出。
>- Bloom = 高亮提取 + 多级模糊 + 回叠。
