## 题目：视锥剔除的计算过程？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说人话**
>- 视锥剔除（Frustum Culling）就是：
>- 在真正发 DrawCall 前，先判断物体是否在相机可见范围里。
>- 完全看不见的物体直接不画，省 CPU 提交和 GPU 渲染成本。
>- 
>- **视锥是什么**
>- 相机的可见空间近似是一个截头锥体（近裁面/远裁面/左右上下 6 个平面）。
>- 任意点只要在这 6 个平面“内侧”，就可能可见。
>- 
>- **常见做法 1：从 VP 矩阵提取 6 个平面**
>- 设 `M = P * V`（投影乘视图矩阵，行/列主序会影响写法）。
>- 平面通常可由“第4行 ± 第1/2/3行”得到：
>- Left: `row4 + row1`
>- Right: `row4 - row1`
>- Bottom: `row4 + row2`
>- Top: `row4 - row2`
>- Near: `row4 + row3`
>- Far: `row4 - row3`
>- 提取后要做平面归一化，便于稳定计算距离。
>- 
>- 平面方程：
> $$
> ax + by + cz + d = 0
> $$
>- 点到平面的有符号距离：
> $$
> dist = n\cdot p + d
> $$
>- 若 `dist < 0` 常表示在平面外侧（符号约定依实现）。
>- 
>- **常见做法 2：包围体与平面相交测试**
>- **包围球测试（快）**
>- 设球心 `c`，半径 `r`，若对任意平面满足：
> $$
> n\cdot c + d < -r
> $$
>- 则球完全在该平面外，物体可直接剔除。
>- 
>- **AABB 测试（更紧）**
>- 取平面法线方向上的“最靠外顶点”做快速排斥（p-vertex 方法）。
>- 若最靠外顶点都在外侧，则 AABB 完全在外。
>- 比包围球更准确，但计算略重。
>- 
>- **完整流程（工程版）**
>- 1. 每帧更新相机 VP。
>- 2. 提取并归一化 6 个平面。
>- 3. 遍历物体包围体做平面测试。
>- 4. 结果分三类：Outside / Intersect / Inside。
>- 5. Outside 直接剔除；其余进入后续（遮挡剔除、LOD、渲染队列）。
>- 
>- **常见坑点**
>- 矩阵行列约定错，平面朝向会反。
>- 包围体未随动画/缩放更新，误剔除。
>- 只做逐物体剔除，CPU 仍可能重，需层次化（BVH/Octree）先粗筛。

>[!tip]- 代码示例
>```cpp
>struct Plane { float3 n; float d; }; // n 需归一化
>
>bool SphereOutsidePlane(const Plane& p, const float3& c, float r)
>{
>    return dot(p.n, c) + p.d < -r;
>}
>
>bool FrustumCullSphere(const Plane planes[6], const float3& c, float r)
>{
>    for (int i = 0; i < 6; ++i) {
>        if (SphereOutsidePlane(planes[i], c, r)) return true; // culled
>    }
>    return false; // maybe visible
>}
>```
>
>```mermaid
>flowchart TD
>  A[提取6个视锥平面] --> B[读取物体包围体]
>  B --> C[逐平面相交测试]
>  C --> D{是否完全在任一平面外?}
>  D -->|是| E[剔除]
>  D -->|否| F[保留并进入后续渲染流程]
>```

>[!quote]- 通俗易懂的理解
>- 视锥剔除是“先问一句：你在不在镜头里？不在就别排队画图”。

>[!example]- 面试回答简版模板
>`视锥剔除通常从VP矩阵提取六平面，再用包围球或AABB做平面相交测试。若包围体完全在任一平面外则直接剔除。工程里会配合层次结构先粗筛，减少CPU逐物体测试成本。`

## 题目：BVH和八叉树的区别？各自的适用范围？
>[!NOTE]- 标准准确的说法（详细版）
>- **一句话先区分**
>- BVH：按“物体分布”分层（对象驱动）。
>- 八叉树：按“空间规则”分块（空间驱动）。
>- 
>- **BVH（Bounding Volume Hierarchy）**
>- 每个节点存一组物体的包围体（如 AABB），子节点包围体更小更紧。
>- 构建常见策略：SAH（表面积启发）、中位数分割、LBVH（GPU友好）。
>- 查询时从根开始与查询体相交，逐层剪枝。
>- 
>- 特点：
>- 对非均匀分布场景更友好（例如城市、室内复杂堆叠）。
>- 包围体通常更紧，遍历效率高。
>- 动态更新可用 refit（只更新包围体）+ 局部重建。
>- 
>- **八叉树（Octree）**
>- 每层把空间均匀切成 8 块，递归细分到阈值。
>- 结构规则，定位和实现直观。
>- 
>- 特点：
>- 适合空间较均匀的体素/地形类数据。
>- 对稀疏或极不均匀分布会出现大量空节点，内存利用率变差。
>- 大量动态移动物体时，节点迁移维护成本高。
>- 
>- **核心对比（工程最常问）**
>- 划分依据：BVH 按物体，八叉树按空间。
>- 紧致性：BVH 通常更紧。
>- 规则性：八叉树更规则，易并行和索引。
>- 动态性：BVH 更容易做 refit；八叉树对象频繁跨块时开销大。
>- 
>- **适用范围建议**
>- 光追加速结构：BVH 主流（几乎标准答案）。
>- 碰撞/可见性（非均匀场景）：BVH 常更优。
>- 体素世界、规则大地形、空间查询：八叉树更直观。
>- 也可混合：大区块用八叉树，区块内物体再用 BVH。

>[!tip]- 快速对比表
>| 维度 | BVH | 八叉树 |
>|---|---|---|
>| 划分方式 | 按物体聚类 | 按空间等分 |
>| 非均匀分布 | 强 | 弱（易空节点） |
>| 实现复杂度 | 中-高 | 低-中 |
>| 动态更新 | refit 友好 | 跨块迁移成本高 |
>| 光追应用 | 主流 | 较少作为主加速结构 |

>[!quote]- 通俗易懂的理解
>- BVH 像“按人群自然分组”；八叉树像“先把地图切成固定网格”。

>[!example]- 面试回答简版模板
>`BVH是对象驱动层次包围体，八叉树是空间驱动规则分块。非均匀复杂场景通常BVH更高效，规则空间和体素类场景八叉树更直观。工程里会按数据分布和动态更新频率选型，必要时混用。`

## 题目：合批操作有哪些？GPU Instancing？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说目标**
>- 合批（Batching）核心是降低 CPU 提交成本：
>- 减少 DrawCall 数量、减少状态切换（材质/纹理/管线切换）。
>- 
>- **常见合批手段**
>- **1）静态合批（Static Batching）**
>- 把静态且长期不变的网格合并成更大网格。
>- 优点：DrawCall 显著下降。
>- 代价：内存上升、剔除粒度变粗（可能多画）。
>- 
>- **2）动态合批（Dynamic Batching）**
>- 运行时把小网格临时打包。
>- 优点：小物体多时可降提交。
>- 代价：CPU 变换/拷贝成本高，不一定总赚。
>- 
>- **3）材质归并 / 纹理图集**
>- 多材质合并为少材质，减少状态切换。
>- 典型是 atlas、数组纹理、统一 shader variant。
>- 
>- **4）GPU Instancing（重点）**
>- 同一网格 + 同一材质（同一 shader 变体）的大量实例，一次提交画多份。
>- 每个实例只传“差异数据”：变换矩阵、颜色、风摆参数等。
>- 
>- **Instancing 什么时候最划算**
>- 重复物体很多：草、树、路灯、栏杆、石头群。
>- 单个 mesh 顶点不高，但数量巨大。
>- 
>- **Instancing 典型流程**
>- 1. 组织实例数据缓冲（instance buffer）。
>- 2. 顶点着色器按 `instanceID` 读取每实例变换。
>- 3. 一次 DrawInstanced/Indirect 绘制 N 个实例。
>- 4. 配合实例级剔除/LOD，避免“全画一遍”。
>- 
>- **常见坑点（高频面试追问）**
>- 不是“开了 instancing 就一定快”。
>- 实例数据带宽过大，可能反吃性能。
>- 如果一个批里可见实例很少，剔除粒度差会造成过绘。
>- 材质变体太多会破坏批次，导致“批不起来”。

>[!tip]- 代码示例
>```cpp
>// 概念伪代码：实例化绘制
>BindMesh(mesh);
>BindMaterial(material);
>BindInstanceBuffer(instanceData); // world matrix / color / custom params
>DrawIndexedInstanced(indexCount, instanceCount);
>```
>
>```mermaid
>flowchart LR
>  A[大量重复物体] --> B[构建InstanceBuffer]
>  B --> C[按instanceID读取实例参数]
>  C --> D[一次Instanced Draw]
>  D --> E[减少DrawCall和状态切换]
>```

>[!quote]- 通俗易懂的理解
>- 合批是“少发命令”；Instancing 是“同一条命令让GPU画很多份同款物体”。

>[!example]- 面试回答简版模板
>`合批主要包括静态合批、动态合批、材质归并和GPU Instancing。Instancing适合同网格同材质的大量重复对象，能显著降低DrawCall，但要控制实例数据带宽并配合实例级剔除。`

## 题目：背面剔除怎么做？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说原理**
>- 三角形有“正反面”定义（由顶点绕序决定，CW/CCW）。
>- 背面剔除（Back-face Culling）就是丢弃背向相机的三角形。
>- 
>- **为什么有效**
>- 闭合实体模型大约一半三角形朝外、一半朝内。
>- 背面通常看不到，提前剔除可减少光栅化和片元着色压力。
>- 
>- **判断方式（概念）**
>- 方式 A：API 按屏幕空间绕序判断正反面（最常见）。
>- 方式 B：几何上看法线与视线方向夹角。
>- 若 `dot(N, V) >= 0`（具体符号随约定）可视作背面。
>- 
>- **API 常见配置**
>- 设正面绕序：`FrontFace = CCW` 或 `CW`。
>- 设剔除模式：`CullMode = Back / Front / None`。
>- 默认最常用：`Cull Back`。
>- 
>- **哪些情况不能直接开背剔除**
>- 透明双面材质（树叶、布料、纸片）。
>- 毛发、卡片式植被、某些特效面片。
>- 镜像缩放（负 scale）可能反转绕序，需要额外处理。
>- 
>- **工程常见做法**
>- 不透明实体：默认背面剔除。
>- 双面材质：关闭剔除或做 two-pass。
>- 阴影 pass：根据需求可改为 front-cull 减少阴影痤疮（与 bias 联动）。

>[!tip]- 代码示例
>```cpp
>// 概念状态设置
>RasterState rs;
>rs.frontFace = FrontFace::CCW;
>rs.cullMode  = CullMode::Back; // 常见默认
>SetRasterState(rs);
>```

>[!quote]- 通俗易懂的理解
>- 看不到的背面就不画，属于“免费午餐”级别的基础优化。

>[!example]- 面试回答简版模板
>`背面剔除通过顶点绕序或法线朝向识别背向三角形并丢弃，可显著减少无效光栅化。默认不透明模型开启Cull Back，双面材质和透明对象按效果需求单独处理。`

## 题目：GPU Driven了解吗？
>[!NOTE]- 标准准确的说法（详细版）
>- **先摆正概念**
>- GPU Driven 不是单个算法，而是一套“让 GPU 参与可见性与绘制列表生成”的渲染架构。
>- 目标：减少 CPU 逐物体提交瓶颈，提升大规模场景扩展能力。
>- 
>- **传统 CPU Driven vs GPU Driven**
>- 传统：CPU 遍历所有物体 -> 做剔除 -> 发大量 DrawCall。
>- GPU Driven：CPU 主要上传场景数据，GPU 用 compute 生成“要画的实例列表”和“间接绘制参数”。
>- 
>- **典型 GPU Driven 流程**
>- 1. CPU 准备静态场景缓冲（实例、包围体、材质索引、meshlet 等）。
>- 2. Compute Pass 做 frustum/occlusion/distance/LOD 剔除。
>- 3. 用 prefix-sum/compaction 把可见实例压紧到 visible list。
>- 4. 生成 indirect draw command buffer。
>- 5. 执行 `Draw*Indirect / MultiDrawIndirect`。
>- 
>- **为什么更快**
>- 少了 CPU 大量逐物体逻辑与 API 调用。
>- 可见性筛选在 GPU 并行完成，和渲染数据“就地处理”。
>- 
>- **常配技术**
>- Hi-Z 遮挡剔除。
>- Cluster/meshlet 粒度组织。
>- Bindless 资源访问。
>- GPU LOD 选择。
>- 
>- **挑战与坑点**
>- 调试复杂（不像 CPU 流程直观）。
>- 同步与数据一致性（barrier、pipeline stage）容易出错。
>- 需要精心设计 buffer 布局和内存访问模式。

>[!tip]- 代码示例
>```mermaid
>flowchart TD
>  A[Instance Buffer] --> B[Compute: Frustum/Occlusion/LOD]
>  B --> C[Compaction Visible List]
>  C --> D[Build Indirect Commands]
>  D --> E[Execute DrawIndirect]
>  E --> F[减少CPU提交瓶颈]
>```
>
>```cpp
>// 概念伪代码
>Dispatch(CullAndLodCS);
>Dispatch(BuildIndirectArgsCS);
>DrawIndexedInstancedIndirect(indirectArgsBuffer);
>```

>[!quote]- 通俗易懂的理解
>- GPU Driven 就是“GPU 不只负责画，还负责先决定画什么”。

>[!example]- 面试回答简版模板
>`GPU Driven把可见性筛选和间接绘制参数生成下放到GPU，通过Compute+Indirect Draw减少CPU逐物体提交开销。它非常适合大规模实例场景，但实现和调试复杂度更高。`

## 题目：大世界场景需要注意哪些问题？
>[!NOTE]- 标准准确的说法（详细版）
>- **先说核心矛盾**
>- 大世界不是“把所有内容同时渲染”，而是“在固定帧预算内，动态提供玩家当前需要的内容”。
>- 
>- **1）空间分块与流式加载（Streaming）**
>- 世界按 chunk/cell 分块管理。
>- 镜头附近高优先级加载，远处延迟或降级。
>- 必须异步 IO + 异步解压 + 异步上传，避免主线程卡顿。
>- 
>- **2）多级 LOD 体系（不仅几何）**
>- 几何 LOD：网格简化、HLOD、impostor。
>- 纹理 LOD：mip streaming、各向异性分级。
>- 阴影 LOD：远处降低分辨率/级联精度。
>- 动画 LOD：远处降骨骼更新频率。
>- 
>- **3）可见性管理（省的关键）**
>- 视锥剔除 + 遮挡剔除 + 距离剔除要叠加使用。
>- 城市场景常用 portal/room 系统，开放地形常用 HZB occlusion。
>- 
>- **4）精度问题（非常关键）**
>- 单精度浮点在超大坐标下抖动明显。
>- 常用方案：原点重定位（origin rebasing）/ 浮动原点。
>- 逻辑和网络层可用双精度，渲染层保持局部单精度。
>- 
>- **5）CPU/GPU/IO 三预算联动**
>- 不是只有 GPU 预算。
>- 需要同时监控：
>- CPU 线程峰值（逻辑、动画、剔除）
>- GPU 帧时（阴影、后处理、透明）
>- IO 带宽与突发延迟（磁盘/解压）
>- 
>- **6）资源生命周期管理**
>- 资源何时加载、何时保活、何时淘汰要有策略。
>- 防止“抖动加载”（刚卸载又立刻加载）和显存碎片。
>- 
>- **7）跨平台分级**
>- PC/主机/移动端应有不同质量档。
>- 档位不是简单开关，而是一整套预算策略（视距、LOD、阴影、AA、特效）。
>- 
>- **8）工程上可落地的监控指标**
>- 可见实例数、DrawCall、三角形数。
>- Streaming miss 率、IO 峰值、解压耗时。
>- LOD 切换频率、遮挡查询命中率。
>- 建议做线上 telemetry，按区域定位瓶颈。

>[!tip]- 实战流程示意
>```mermaid
>flowchart TD
>  A[玩家位置更新] --> B[决定激活Chunk集合]
>  B --> C[异步IO加载资源]
>  C --> D[构建可见性与LOD]
>  D --> E[提交渲染]
>  E --> F[回收远处资源]
>```
>
>```text
>大世界优化顺序建议：
>1) 先稳帧（避免尖峰）
>2) 再提均值帧率
>3) 最后追求极限画质
>```

>[!quote]- 通俗易懂的理解
>- 大世界优化本质是“只在对的时间、对的地点，加载并渲染刚好够用的细节”。

>[!example]- 面试回答简版模板
>`大世界重点是分块流式、LOD体系、可见性剔除、浮点精度管理和异步资源管线。目标不是把所有内容同时画出来，而是在CPU/GPU/IO固定预算下稳定输出玩家周围高质量画面，并避免帧时尖峰。`

## 一页速记（考前30秒）
>[!NOTE]- 你最少要记住的几句话
>- 视锥剔除：6平面 vs 包围体，Outside 直接丢。
>- BVH 是按物体分层，八叉树是按空间等分。
>- 合批与 Instancing 的核心是降 DrawCall 与状态切换。
>- 背面剔除对不透明闭合模型收益很高，双面材质要例外处理。
>- GPU Driven 是“Compute 筛选 + Indirect Draw”，核心是减 CPU 提交瓶颈。
>- 大世界是系统工程：Streaming + LOD + Culling + 精度 + 预算联动。
