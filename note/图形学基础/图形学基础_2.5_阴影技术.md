## 题目：Shadow Map的原理和实现？有什么缺点？
>[!NOTE]- 标准准确的说法
>- **核心原理**
>- Shadow Map 的本质是“深度可见性测试”：
>- 先在光源视角记录“离光源最近的深度”，再在相机视角判断当前片元是否比该深度更远。
>- 若更远，则说明光线被前方几何遮挡，片元处于阴影中。
>- **标准两阶段流程**
>- 1) Light Pass（阴影图生成）：
>- 以光源为相机，只写深度，得到 `shadowMap`。
>- 2) Camera Pass（主渲染）：
>- 片元坐标变换到光源裁剪空间，做透视除法与偏移映射到 `[0,1]` UV；
>- 采样阴影图并比较深度，得到 `shadow` 因子。
>- **关键坐标变换**
> $$
> p_{lightClip} = M_{lightVP} \cdot p_{world}
> $$
> $$
> p_{lightNDC} = \frac{p_{lightClip}.xyz}{p_{lightClip}.w}
> $$
> $$
> uv = 0.5 \cdot p_{lightNDC}.xy + 0.5
> $$
>- **常见伪影/缺点**
>- 1) 走样和闪烁（aliasing / shimmering）：阴影图分辨率与投影视角变化导致。
>- 2) 自遮挡（shadow acne）：深度比较误差造成表面被误判为阴影。
>- 3) Peter-panning：bias 过大导致阴影与物体“脱离”。
>- 4) 透视分配不均：近处/远处 texel 密度不平衡。
>- 5) 硬边阴影：不做滤波时边缘生硬。
>- **工程改进手段**
>- PCF/PCSS 软化边缘；
>- slope-scale bias + normal offset 处理精度问题；
>- CSM 解决方向光大场景分布不均；
>- 稳定化采样和级联混合减少抖动。

>[!tip]- 代码示例
>```mermaid
>flowchart TD
>  A[Light Pass: 渲染深度到 ShadowMap] --> B[Camera Pass: 片元转到光源空间]
>  B --> C[采样 ShadowMap 深度]
>  C --> D{currentDepth - bias > shadowDepth?}
>  D -->|是| E[在阴影中]
>  D -->|否| F[受光]
>```
>
>```cpp
>// 伪代码：基础ShadowMap比较
>float4 lightClip = mul(lightVP, float4(worldPos, 1.0));
>float3 lightNdc = lightClip.xyz / lightClip.w;
>float2 uv = lightNdc.xy * 0.5 + 0.5;
>float currentDepth = lightNdc.z;
>
>float shadowDepth = ShadowMap.SampleLevel(ShadowSampler, uv, 0).r;
>float bias = ComputeBias(N, L); // constant + slope-scale
>float shadow = (currentDepth - bias > shadowDepth) ? 0.0 : 1.0;
>```

>[!quote]- 通俗易懂的理解
>- 可以理解成“光源先拍一张只有深度的黑白照片”。
>- 主相机渲染时，每个像素都去问这张照片：
>- “我是不是比光看到的最近表面更靠后？”如果是，就在阴影里。

>[!example]- 面试回答简版模板
>`Shadow Map是先从光源视角生成深度图，再在主渲染中把片元变换到光源空间做深度比较。它通用且高效，但会有走样、acne、peter-panning等精度问题，工程上通常结合bias、PCF/PCSS和CSM优化。`

## 题目：CSM(级联阴影)是怎么做的？
>[!NOTE]- 标准准确的说法
>- **为什么需要 CSM**
>- 方向光阴影覆盖范围大，单张 shadow map 很难同时保证近处清晰和远处可用。
>- CSM（Cascaded Shadow Maps）通过把相机视锥按深度切成多段，把分辨率优先分给近处。
>- **基本流程**
>- 1) 将相机 frustum 按深度拆成 `N` 个 cascade。
>- 2) 对每个 cascade 计算独立的光源正交包围盒和 lightVP。
>- 3) 分别渲染每级阴影图。
>- 4) 主渲染按片元视空间深度选择 cascade（或进行级联混合）。
>- **分割策略（高频）**
>- 线性分割：近远分辨率分配不理想。
>- 对数分割：近处质量好，远处更稀疏。
>- 实际常用混合分割：
> $$
> d_i = \lambda \cdot d_i^{log} + (1-\lambda) \cdot d_i^{lin}
> $$
>- 其中 `\lambda` 常在 `0.5~0.9` 范围调参。
>- **稳定化（防抖动）**
>- 把 light-space 投影对齐到 shadow texel 网格，减少相机轻微移动引起的阴影游走。
>- 级联交界区做 `blend`，避免硬切换线。
>- **常见问题**
>- 1) 级联边界跳变（需混合）。
>- 2) 级联矩阵频繁变化导致闪烁（需稳定化）。
>- 3) 级联数过多成本上升（通常 3~4 级折中）。

>[!tip]- 代码示例
>```mermaid
>flowchart LR
>  A[Camera Frustum] --> B[Split0 Near]
>  A --> C[Split1 Mid]
>  A --> D[Split2 Far]
>  B --> E[ShadowMap0 高密度]
>  C --> F[ShadowMap1 中密度]
>  D --> G[ShadowMap2 低密度]
>```
>
>```cpp
>// 伪代码：按深度选择级联
>int cascadeIdx = SelectCascade(viewDepth, splitDepths);
>float shadow = SampleCascadeShadow(cascadeIdx, worldPos, N, L);
>
>// 可选：边界区域做级联混合
>shadow = BlendCascadeIfNeeded(cascadeIdx, viewDepth, splitDepths, shadow);
>```

>[!quote]- 通俗易懂的理解
>- CSM 像“分层预算”：
>- 离玩家近的区域分配更多阴影像素，远处接受更粗略阴影。
>- 这样同样总预算下，体感质量会明显提升。

>[!example]- 面试回答简版模板
>`CSM把相机视锥分成多级，为每级单独生成方向光阴影图，近处高密度、远处低密度。核心工程点是分割策略（常用线性+对数混合）、级联稳定化和边界混合。`

## 题目：PCF和PCSS的原理？
>[!NOTE]- 标准准确的说法
>- **PCF（Percentage Closer Filtering）**
>- 思路是“先比较、再滤波”：对邻域多个采样点做深度比较，把 0/1 阴影结果平均。
>- 结果是阴影边缘被软化，但半影宽度通常固定。
> $$
> shadow_{pcf} = \frac{1}{N}\sum_{i=1}^{N} step(d_i^{map}, d^{current}-bias)
> $$
>- **PCSS（Percentage Closer Soft Shadows）**
>- 在 PCF 基础上引入“可变核半影”：
>- 1) blocker search：在小核内统计遮挡体平均深度 `z_blocker`。
>- 2) penumbra estimate：根据光源尺寸和深度关系估计半影宽度。
>- 3) 用该宽度做自适应 PCF。
>- 常见近似关系：
> $$
> w_{penumbra} \propto \frac{z_{receiver}-z_{blocker}}{z_{blocker}} \cdot w_{light}
> $$
>- **术语白话版（重点）**
>- `半影（penumbra）`：阴影边缘那圈“灰色过渡带”，不是全黑本影，也不是全亮区域。
>- `核（kernel）`：采样邻域大小，可理解为“拿多大范围做平均”。
>- `可变核半影`：核大小不是固定值，而是每个像素按几何关系动态变化。
>- `receiver`：接收阴影的点（例如地面上的当前像素）。
>- `blocker`：挡光体（例如箱子边缘）。
>- 为什么要“可变核”：接收点离遮挡体越远，真实半影越宽，因此采样核应更大；离得近则核更小，边缘更硬。
>- 一个直观例子：手电筒照物体投到墙上，物体离墙越远，影子边缘越糊；PCSS就是把这个规律做进实时算法。
>- **效果与代价对比**
>- PCF：稳定、便宜、实现简单，但“近硬远软”不明显。
>- PCSS：真实感更好（半影随距离变化），但采样开销更大且更易噪声。
>- **工程优化要点**
>- 旋转泊松盘采样降低 banding；
>- 分离 blocker 搜索和过滤核采样数量；
>- 按质量档动态调整 sample count。

>[!tip]- 代码示例
>```cpp
>// 伪代码：PCF
>float ShadowPCF(float2 uv, float depth, float bias, float radius) {
>    float sum = 0.0;
>    for (int i = 0; i < K; ++i) {
>        float2 offset = PoissonDisk[i] * radius;
>        float z = ShadowMap.SampleLevel(Samp, uv + offset, 0).r;
>        sum += (depth - bias > z) ? 0.0 : 1.0;
>    }
>    return sum / K;
>}
>
>// 伪代码：PCSS（简化）
>float zBlocker = SearchAverageBlockerDepth(uv, depth);
>float penumbra = EstimatePenumbra(depth, zBlocker, lightSize);
>float shadow = ShadowPCF(uv, depth, bias, penumbra);
>```

>[!quote]- 通俗易懂的理解
>- PCF 是“固定模糊半径”的阴影软化。
>- PCSS 是“先估计遮挡关系，再动态调模糊半径”，因此能做出更像真实面积光的近硬远软。

>[!example]- 面试回答简版模板
>`PCF通过多点深度比较平均来软化阴影边；PCSS在此基础上增加blocker搜索和可变核过滤，能形成距离相关半影，视觉更真实但成本明显更高。`

## 题目：软阴影如何实现？
>[!NOTE]- 标准准确的说法
>- **先说人话：什么是软阴影**
>- 软阴影就是“阴影边缘不是硬切，而是有一圈渐变带（半影）”。
>- 这圈渐变出现的原因是：真实光源有面积，不是理想点光源。
>- **你最该记住的物理规律**
>- 遮挡体离接收面越远，半影通常越宽。
>- 光源面积越大，半影通常越宽。
>- 所以软阴影不是固定模糊，而是“半影宽度会变化”。
>- **实时实现可以分成 4 条路（从便宜到贵）**
>- 1) PCF：固定核过滤，最容易上手，稳定但真实感一般。
>- 2) PCSS：先估计半影宽度，再做可变核过滤，能做出“近硬远软”。
>- 3) 方差类（VSM/EVSM/MSM/VSSM）：支持大核预滤波，效率高，但要防漏光。
>- 4) 光追阴影（RT Shadows）：最接近真实，但成本最高，通常需要去噪。
>- **第3类详解：方差类（VSM / EVSM / MSM / VSSM）**
>- 核心思路：不直接存“单点深度比较结果”，而是存“深度统计量”，再用概率估计阴影可见性。
>- 为什么它常被说“滤波友好”：统计量可以线性过滤，所以很适合做大核软化。
>- 典型统计量（以 VSM 思路为例）：
> $$
> M_1 = E[z],\quad M_2 = E[z^2]
> $$
> $$
> \sigma^2 = M_2 - M_1^2
> $$
>- 然后用概率上界估计光照可见性（简化表达）：
> $$
> p_{lit} \le \frac{\sigma^2}{\sigma^2 + (z_r - M_1)^2}
> $$
>- 你会遇到的核心问题：漏光（light leaking）。
>- 漏光常见原因：统计上界偏松 + 大核滤波把前景/背景深度混在一起。
>- 常见缓解：EVSM（指数变换）、MSM（更高阶矩）、clamp/bleed-reduction、减小跨边界滤波。
>- 适用场景：想要“软阴影 + 可控性能 + 大核过滤”的中端实时项目。
>- **第4类详解：RT 阴影（Ray Traced Shadows）**
>- 核心思路：每个像素从表面点朝光源方向发射阴影射线，检查中途是否有几何体阻挡。
>- 跟 ShadowMap 的本质区别：RT 在做几何求交；ShadowMap 在做深度纹理比较。
>- 为什么更真实：面积光可用多条射线采样，天然得到半影变化，不需要手工估核宽。
>- 典型实时流程：发射阴影射线 -> 得到可见性 -> 时域/空域去噪 -> 与主光照合成。
>- 主要成本：射线追踪开销和去噪开销都高，低采样时会出现噪点。
>- 常见副作用：去噪可能带来拖影或过度平滑，需要与历史缓存和运动矢量配合调参。
>- 工程里通常不会“全场全灯都 RT”：常见做法是主光 RT、次光 PCSS/EVSM 的混合方案。
>- **如何选方案（项目视角）**
>- 手游/低端机：PCF（主）+ 少量接触阴影。
>- 中端平台：PCSS 或 EVSM（主）+ 时域稳定化。
>- 高端主机/PC：混合方案（主光 RT，次光 PCSS/EVSM）。
>- **为什么你会觉得难懂**
>- 很多资料把“软阴影”写成算法名堆叠，但本质只有一句：
>- 先决定“这点应该多软”，再用对应采样去实现这个软度。
>- **一个简单的工程落地流程**
>- 1) 先做可用的硬阴影（ShadowMap + bias）。
>- 2) 上 PCF，先解决硬边。
>- 3) 需要更真实时换 PCSS，加入 blocker search 和可变核。
>- 4) 遇到噪声/闪烁，再加时域稳定化和采样模式优化。
>- 5) 平台允许时再考虑 RT 混合提升上限。

>[!tip]- 代码示例
>```cpp
>// 伪代码：按质量档选择软阴影方案
>switch (shadowQuality) {
>case LOW:
>    shadow = ShadowPCF(...);      // 固定核，便宜稳定
>    break;
>case MEDIUM:
>    shadow = ShadowPCSS(...);     // 可变核，近硬远软
>    break;
>case HIGH:
>    shadow = HybridRTShadow(...); // RT + denoise + fallback
>    break;
>}
>```
>
>```mermaid
>flowchart TD
>  A[先有硬阴影] --> B[PCF: 固定核软化]
>  B --> C{是否需要更真实半影?}
>  C -->|否| D[保持PCF]
>  C -->|是| E[PCSS: 估半影+可变核]
>  E --> F{预算足够?}
>  F -->|否| G[PCSS+时域稳定化]
>  F -->|是| H[混合RT阴影]
>```

>[!quote]- 通俗易懂的理解
>- 把灯想成有面积的灯板，不是数学点。
>- 物体在墙上的影子边缘会有“灰带”，而且距离变化时灰带宽度也会变。
>- 软阴影算法做的就是把这个“灰带宽度变化”近似出来。

>[!example]- 面试回答简版模板
>`软阴影本质是模拟面积光产生的半影。实时实现通常分为PCF、PCSS、方差类和RT方案：PCF成本低但半影固定，PCSS通过可变核实现近硬远软，方差类滤波友好但需抑制漏光，RT最真实但成本高。工程上一般按平台预算做分档和混合。`

## 题目：大场景的阴影贴图怎么处理？
>[!NOTE]- 标准准确的说法
>- **核心矛盾**
>- 大场景中阴影覆盖范围很大，但阴影纹理预算有限，必须做“空间+时间+重要度”三维分配。
>- **常见策略组合**
>- 1) 空间分配：CSM（近高远低）、阴影距离裁剪。
>- 2) 时间分配：远级联低频更新、静态区域缓存。
>- 3) 重要度分配：角色/关键交互体优先高质量阴影。
>- 4) 资源组织：atlas/array 管理多光源与多级联。
>- **动态调度思路**
>- 根据屏幕占比、距离、光源重要度给阴影分辨率和更新频率打分。
> $$
> score = w_s \cdot screenArea + w_d \cdot (1/depth) + w_g \cdot gameplayImportance
> $$
>- 评分低的光源可降分辨率、降更新频率，甚至关闭阴影。
>- **典型问题**
>- 阴影 popping（分辨率/更新切换突变）
>- 远景抖动和闪烁
>- atlas 冲突与碎片化
>- **工程解决**
>- 分档过渡（hysteresis）避免来回切换；
>- 关键区域强制稳定采样；
>- 对动态物体局部补充高质量阴影通道。

>[!tip]- 代码示例
>```cpp
>// 伪代码：阴影预算调度
>for (Light light : visibleLights) {
>    float score = EvalShadowImportance(light, camera, gameplay);
>    ShadowBudgetDecision d = AllocateShadowBudget(score);
>    light.shadowResolution = d.resolution;
>    light.updateRate = d.updateRate;
>    light.castShadow = d.enable;
>}
>```

>[!quote]- 通俗易懂的理解
>- 大场景阴影不是“全都高清”，而是“把钱花在玩家看得见、看得清、看得在意的地方”。

>[!example]- 面试回答简版模板
>`大场景阴影要做预算调度：空间上CSM和距离裁剪，时间上分级降频更新，重要度上关键物体优先。核心是用有限纹理和带宽换取稳定的主观质量。`

## 题目：阴影自遮挡问题如何解决？
>[!NOTE]- 标准准确的说法
>- **先说人话：什么是阴影自遮挡（shadow acne）**
>- 你看到模型表面出现“脏条纹/黑点”，很多时候就是自遮挡：
>- 表面把“自己”误判成了挡光体。
>- **为什么会误判**
>- 阴影判断要比较两个深度值：
>- 当前片元深度 vs 阴影图里记录的深度。
>- 这两个值非常接近时，浮点误差、纹理精度和光栅化差异会让比较结果抖动。
>- **标准解决方法：给比较留容差（bias）**
>- 1) Constant bias：所有像素统一加一点偏移。
>- 2) Slope-scale bias：表面越斜，偏移越大（斜面更容易出误差）。
>- 3) Normal offset：沿法线轻推采样位置，降低“自己挡自己”的概率。
>- 常见近似写法：
> $$
> bias = b_c + b_s \cdot (1 - N\cdot L)
> $$
>- **副作用（必须同时理解）**
>- bias 太小：acne 还在。
>- bias 太大：Peter-panning（阴影和物体接触点分离，看起来“飘”）。
>- 所以这题的核心不是“有没有 bias”，而是“bias 多大才平衡”。
>- **最实用调参顺序**
>- 1) 先开小的 constant bias，消除大部分条纹。
>- 2) 再加 slope-scale bias，重点压斜面伪影。
>- 3) 局部顽固问题再加少量 normal offset。
>- 4) 边调边盯“接触阴影”是否脱离，防止 peter-panning。
>- 5) 最后再优化阴影图精度（分辨率、near/far、级联参数），减少对大偏移的依赖。
>- **工程建议**
>- 不同光源/不同级联使用不同 bias 参数，不要“一套参数打天下”。
>- 和 PCF/PCSS 一起联调，否则滤波会改变你看到的误差表现。

>[!tip]- 代码示例
>```cpp
>// 伪代码：常见 bias 策略
>float ndotl = saturate(dot(N, L));
>float bias = constBias + slopeBias * (1.0 - ndotl);
>
>// 可选：法线偏移（世界空间）
>float3 worldPosBiased = worldPos + N * normalOffset;
>
>float shadow = SampleShadow(worldPosBiased, bias);
>```

>[!quote]- 通俗易懂的理解
>- 可以把它理解成“尺子精度不够导致比较抖动”。
>- bias 就是在比较时预留一点容差。
>- 但容差过大又会把阴影推离物体，所以要在“干净”和“贴合”之间找平衡。

>[!example]- 面试回答简版模板
>`阴影自遮挡本质是深度比较精度误差导致的误判。工程上通常用 constant + slope-scale bias，并在需要时加 normal offset。关键是平衡 acne 与 peter-panning，同时通过阴影图精度和投影范围优化减少偏移需求。`

## 题目：VSSM的原理？漏光问题的原因？
>[!NOTE]- 标准准确的说法
>- **先澄清命名**
>- 实践里常把 VSM/EVSM/MSM/VSSM 归为“方差或矩统计阴影”一类。
>- 核心思想是存深度统计量（矩），用概率上界估计是否被遮挡，从而支持预滤波软阴影。
>- **典型统计形式（以 VSM 思路为例）**
>- 存一阶/二阶矩：
> $$
> M_1 = E[z],\quad M_2 = E[z^2]
> $$
>- 方差：
> $$
> \sigma^2 = M_2 - M_1^2
> $$
>- 用切比雪夫上界估计透光概率（简化）：
> $$
> p_{lit} \le \frac{\sigma^2}{\sigma^2 + (z_r - M_1)^2}
> $$
>- **为什么会漏光（light leaking）**
>- 1) 概率上界本身偏“松”，在多峰深度分布时会高估透光。
>- 2) 大核预滤波把前后景深度混合，遮挡边界被“平均化”。
>- 3) 接收面深度与统计矩不匹配时，概率估计失真。
>- **常见缓解方案**
>- EVSM：对深度做指数变换改善上界质量；
>- MSM：更高阶矩提升分布表达能力；
>- clamp/warp/min-variance 策略抑制泄露；
>- 分层或局部化滤波，减少跨遮挡边界混合。
>- **取舍**
>- 优点：滤波友好、软阴影效率高。
>- 缺点：存在统计近似误差与漏光风险，调参复杂度高。

>[!tip]- 代码示例
>```cpp
>// 伪代码：方差阴影简化计算
>float2 moments = MomentsTex.SampleLevel(Samp, uv, 0).rg; // M1, M2
>float mean = moments.x;
>float var  = max(moments.y - mean * mean, minVariance);
>
>float d = receiverDepth - mean;
>float pMax = var / (var + d * d); // Chebyshev上界近似
>float shadow = saturate(pMax);
>
>// 工程上常叠加 clamp / bleed reduction
>shadow = ReduceLightBleeding(shadow, bleedParam);
>```

>[!quote]- 通俗易懂的理解
>- 这类方法像“用统计学猜遮挡”。
>- 猜得快、滤波友好，但在复杂遮挡边界处会“猜太乐观”，于是出现漏光。

>[!example]- 面试回答简版模板
>`VSSM/方差类阴影通过存深度矩并做概率估计来实现可预滤波软阴影。漏光主要来自概率上界偏松与跨边界滤波混合。工程上常用EVSM/MSM与clamp策略缓解。`

## 题目：多个光源的阴影怎么处理？
>[!NOTE]- 标准准确的说法
>- **核心问题**
>- 多光源阴影成本接近“光源数 × 阴影成本”，如果全部最高质量开启会很快超预算。
>- **常见调度策略**
>- 1) 主光优先：太阳光/关键剧情灯固定高质量。
>- 2) 次光分级：按重要度分配分辨率、更新频率、过滤质量。
>- 3) 长尾光源裁剪：距离远、屏占比小、贡献低的光源关闭阴影。
>- 4) atlas/array 管理：减少资源切换和碎片。
>- 5) clustered/deferred 结合 light culling：减少每像素参与阴影光源数量。
>- **重要度评分示例**
> $$
> score = w_l \cdot luminance + w_s \cdot screenArea + w_d \cdot (1/depth) + w_g \cdot gameplayTag
> $$
>- 依据 `score` 决定 `castShadow / resolution / updateRate`。
>- **更新频率策略**
>- 主光每帧更新；
>- 次光按 2~8 帧更新一次；
>- 静态光源阴影可缓存。
>- **常见坑**
>- 阴影开关频繁导致 popping；
>- atlas 重排导致抖动；
>- 多灯重叠区域成本暴涨。
>- **解决思路**
>- 评分滞回（hysteresis）和平滑过渡；
>- 关键角色附近保底阴影；
>- 把“阴影质量档位”纳入整体渲染预算系统。

>[!tip]- 代码示例
>```mermaid
>flowchart TD
>  A[光源集合] --> B[计算重要度分数]
>  B --> C[主光: 高质量阴影]
>  B --> D[次光: 分级分辨率/降频更新]
>  B --> E[低优先级: 关闭阴影]
>  C --> F[进入阴影图集/数组]
>  D --> F
>```
>
>```cpp
>// 伪代码：多光源阴影预算调度
>for (Light l : visibleLights) {
>    float score = EvalLightShadowScore(l, camera, gameplayState);
>    ShadowTier tier = ChooseTier(score);
>    ApplyShadowTier(l, tier); // enable/resolution/updateRate/filterMode
>}
>```

>[!quote]- 通俗易懂的理解
>- 多灯阴影不是“人人同配”，而是“按贡献分层配给”。
>- 玩家最在意的光源要稳、要清楚；其余光源按预算动态让步。

>[!example]- 面试回答简版模板
>`多光源阴影本质是预算调度问题：主光高质量，次光按重要度分级，长尾光源可降频或关闭。配合atlas和light culling，在可控成本下保证主观画质稳定。`
