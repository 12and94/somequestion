## 题目：Early-Z是什么？有什么优劣？
>[!NOTE]- 标准准确的说法
>- **定义**
>- Early-Z 是指在片元着色器执行前，GPU 尽可能提前做深度测试（以及部分情况下的模板测试），把注定被遮挡的片元提前丢弃。
>- **核心价值**
>- 片元着色通常是图形管线最贵阶段之一。Early-Z 通过“先判可见性、后着色”减少无效 FS（Fragment Shader）执行次数。
>- **优势**
>- overdraw（过度绘制）高的场景收益明显。
>- 片元 shader 越重（多纹理采样、复杂 BRDF、阴影查询），收益越大。
>- **限制与代价**
>- 不是每个 draw call 都能完整享受 Early-Z。
>- 一旦片元阶段存在会改变覆盖/深度可见性的操作，硬件可能退化为 Late-Z 或部分保守模式。
>- **与 Late-Z 关系**
>- Late-Z 是在片元着色后再做深度测试。
>- 现代 GPU 常在不同条件下动态选择早测/晚测，甚至混合策略。

>[!tip]- 代码示例
>```mermaid
>flowchart LR
>  A[光栅化得到片元] --> B{Early-Z条件满足?}
>  B -->|是| C[先深度测试]
>  C -->|失败| D[丢弃, 不执行FS]
>  C -->|通过| E[执行FS]
>  B -->|否| E
>  E --> F[Late-Z/模板/混合]
>```
>
>```cpp
>// 伪代码：Early-Z思路
>for each fragment {
>    if (EarlyZEnabled && depthTestFail(fragment)) {
>        discard_without_shading();
>    } else {
>        shade_fragment();
>        late_depth_test_if_needed();
>    }
>}
>```

>[!quote]- 通俗易懂的理解
>- Early-Z 就像“先看这个像素是否已经被前面的物体挡住”。
>- 如果已经挡住，就不必做昂贵的上色计算，相当于先筛简历再面试，节省大量时间。

>[!example]- 面试回答简版模板
>`Early-Z 是把深度可见性判断前置，目标是减少无效片元着色。它在高 overdraw 和重型像素着色场景收益很大，但遇到会改变覆盖/深度行为的片元逻辑时可能退化。`

## 题目：Early-Z失效的情况有哪些？
>[!NOTE]- 标准准确的说法
>- **典型触发点**
>- 1) `discard/alpha test`：
>- 片元是否保留取决于 shader 运行结果，硬件难以提前确定覆盖。
>- 2) 显式写深度（如 `gl_FragDepth` / `SV_Depth`）：
>- 最终深度由 shader 决定，前置深度测试不再可靠。
>- 3) 依赖顺序的混合与特殊操作：
>- 某些 blending、读改写路径会削弱 Early-Z 的可用性。
>- 4) 透明渲染路径：
>- 透明通常关闭深度写入并依赖排序混合，Early-Z 价值和形态与不透明不同。
>- **注意**
>- “失效”不一定是 0 收益，而是收益下降或只在部分条件下生效。
>- 不同 GPU 厂商/架构有差异，需 profile 验证。

>[!tip]- 代码示例
>```glsl
>// 触发Early-Z受限的典型写法
>if (texture(alphaTex, uv).a < 0.5) {
>    discard;
>}
>
>gl_FragDepth = customDepth; // 显式改写深度
>```
>
>```mermaid
>flowchart TD
>  A[FS含discard?] -->|是| X[Early-Z收益下降]
>  B[FS写gl_FragDepth?] -->|是| X
>  C[透明+混合路径?] -->|是| X
>```

>[!quote]- 通俗易懂的理解
>- Early-Z 依赖“提前知道谁能看见”。
>- 只要你的 shader 会“运行后才决定要不要丢弃”或“运行后再改深度”，GPU 就很难提前拍板。

>[!example]- 面试回答简版模板
>`Early-Z 的本质前提是可见性可提前确定。discard、显式深度写、透明混合等会破坏这个前提，导致 Early-Z 失效或收益下降。`

## 题目：Pre-Z Pass是什么？
>[!NOTE]- 标准准确的说法
>- **定义**
>- Pre-Z（又叫 Z-Prepass）是“先跑一次深度预填充，只写深度不做复杂着色”，再执行正式颜色 pass。
>- **流程**
>- Pass1：深度通道，仅做必要顶点处理和光栅化，写深度。
>- Pass2：颜色通道，利用已有深度让被遮挡片元尽量在 Early-Z 阶段被剔除。
>- **适用场景**
>- 高 overdraw、重型片元 shader（复杂材质、阴影、体积效果）更可能收益明显。
>- **代价**
>- 增加一次几何遍历和 draw 提交，不是“永远赚”。
>- 小场景、低 overdraw、轻 shader 可能得不偿失。

>[!tip]- 代码示例
>```mermaid
>flowchart TD
>  A[Pass1: Pre-Z 仅写深度] --> B[Pass2: 正式着色]
>  B --> C[被挡片元在Early-Z被过滤]
>```
>
>```cpp
>// 伪代码
>RenderDepthOnlyPrepass();
>RenderShadingPassWithDepthTest();
>```

>[!quote]- 通俗易懂的理解
>- 先打一遍“深度草稿”，后面正式上色时就知道哪些像素根本看不见，可以直接跳过。

>[!example]- 面试回答简版模板
>`Pre-Z 是“先深度、后着色”的两阶段策略。它用额外几何 pass 换取更强的早期剔除，是否划算取决于 overdraw 和像素着色成本。`

## 题目：深度测试的过程？透明和非透明物体的渲染顺序？
>[!NOTE]- 标准准确的说法
>- **1. 深度测试是什么（先给定义）**
>- 深度测试（Depth Test）是比较“当前片元深度 `z_frag`”与“深度缓冲当前值 `z_buf`”，决定该片元是否可见。
>- 深度函数可为 `Less/LEqual/Greater...`，常见是 `Less` 或 `LEqual`。
>
>- **2. 一个片元经过深度测试的完整过程**
>- 1) 光栅化生成片元，并得到其深度 `z_frag`。
>- 2) 读取当前像素深度缓冲值 `z_buf`。
>- 3) 按深度函数比较（例：`z_frag < z_buf`）。
>- 4) 比较失败：丢弃片元。
>- 5) 比较通过：进入后续（着色/混合），并根据状态决定是否写回深度。
>
>- **3. 不透明物体为什么通常前到后（Front-to-Back）**
>- 不透明通常设置：`DepthTest On + DepthWrite On + Blend Off`。
>- 前到后让近处先写深度，后面的远处片元更容易被 Early-Z 或深度测试剔除，减少无效片元着色。
>- 目标是性能优先（减少 overdraw 成本）。
>
>- **4. 透明物体为什么通常后到前（Back-to-Front）**
>- 透明通常设置：`DepthTest On + DepthWrite Off + Blend On`。
>- 透明颜色要与背景按顺序混合，混合运算顺序相关（不可交换）。
>- 若顺序错误（前到后）会导致叠色结果偏差。
>
>- **5. 透明混合为什么“顺序相关”**
>- 常见混合：`C_out = C_src * a_src + C_dst * (1 - a_src)`。
>- 这里的 `C_dst` 是“当前缓冲颜色”，先后顺序不同会导致 `C_dst` 不同，最终颜色不同。
>
>- **6. 常见误区澄清**
>- 误区 1：透明应该也开深度写。
>- 更准确：透明通常关深度写，避免先画的透明层把后画透明层错误挡住。
>- 误区 2：透明不需要深度测试。
>- 更准确：透明通常保留深度测试，以正确被不透明遮挡。
>- 误区 3：所有材质都按同一顺序画。
>- 更准确：不透明优先前到后，透明通常后到前。

>[!tip]- 代码示例
>```cpp
>// 常见渲染顺序（伪代码）
>ClearColorAndDepth();
>
>// 1) 不透明：前到后，深度写开
>SetDepthTest(LessEqual);
>SetDepthWrite(true);
>DisableBlend();
>SortOpaqueFrontToBack();
>DrawOpaque();
>
>// 2) 透明：后到前，深度写关
>SetDepthTest(LessEqual);
>SetDepthWrite(false);
>EnableAlphaBlend();
>SortTransparentBackToFront();
>DrawTransparent();
>```
>
>```mermaid
>flowchart TD
>  A[片元到达] --> B[读取z_buf]
>  B --> C{DepthTest(z_frag, z_buf)}
>  C -->|失败| D[丢弃]
>  C -->|通过| E[片元着色/混合]
>  E --> F{DepthWrite?}
>  F -->|是| G[写回z_buf]
>  F -->|否| H[保留原深度]
>```
>
>```text
>// 透明顺序示意（概念）
>// 远层 T_far + 近层 T_near + 背景 B
>// 正确：先 T_far 再 T_near
>// 错误：先 T_near 再 T_far（结果会变）
>```

>[!quote]- 通俗易懂的理解
>- 深度缓冲像“每个像素的最近占位记录”。
>- 不透明先画近的，能尽早把远处无效像素挡掉，省算力。
>- 透明像“有颜色的玻璃叠加”，必须按从远到近一层层叠，不然颜色会串。

>[!example]- 面试回答简版模板
>`深度测试是比较 z_frag 与 z_buf 决定可见性。工程里不透明通常前到后渲染并开启深度写，以提升 Early-Z 剔除效率；透明通常后到前渲染，保留深度测试但关闭深度写，保证混合顺序正确。`

## 题目：Z-Buffer的实现？深度精度问题如何解决？
>[!NOTE]- 标准准确的说法
>- **Z-Buffer 实现要点**
>- 每个像素存一个深度值（常见 24-bit 或 32-bit 浮点深度）。
>- 栅格化后片元会与该值比较，决定可见性。
>- **精度问题来源**
>- 透视投影导致深度分布非线性：近处密、远处稀。
>- near 过小、far 过大时，有效精度被拉薄，远处更易抖动/冲突。
>- **常用优化**
>- 1) 收紧 near/far：尤其避免 near 设得过小。
>- 2) Reversed-Z + 浮点深度：
>- 把深度比较方向反过来并用浮点格式，通常显著改善远处精度。
>- 3) 分层渲染/分段深度：超大场景可按距离分层。
>- 4) 仅在必要场景用 log depth（有副作用需评估）。

>[!tip]- 代码示例
>```cpp
>// 深度范围配置（示意）
>camera.nearPlane = 0.1f;   // 不要盲目设成 0.001
>camera.farPlane  = 1000.0f;
>
>// Reversed-Z 思路（API细节依实现而异）
>// 使用浮点深度 + 反向比较函数
>SetDepthFunc(GreaterEqual);
>ClearDepth(0.0f);
>```
>
>```mermaid
>flowchart LR
>  A[near太小/far太大] --> B[深度分布更不均]
>  B --> C[远处精度不足]
>  C --> D[z-fighting概率上升]
>```

>[!quote]- 通俗易懂的理解
>- 深度缓冲像一把“非均匀刻度尺”：近处刻度很密，远处刻度很稀。
>- near/far 设不好就像拿错尺子量远处，容易量不准。

>[!example]- 面试回答简版模板
>`Z-Buffer 通过每像素深度比较决定可见性。精度问题主要来自透视非线性分布，工程上优先收紧 near/far，进阶可用 Reversed-Z + 浮点深度提升远处稳定性。`

## 题目：z-fighting怎么处理？
>[!NOTE]- 标准准确的说法
>- **定义**
>- z-fighting 是两个（或多层）表面深度非常接近时，受深度量化误差影响发生前后关系抖动。
>- **常见原因**
>- 共面或近共面几何（Decal、重复地面层）。
>- near/far 比例过大导致深度精度不足。
>- **处理策略（按优先级）**
>- 1) 改几何：避免共面，拉开微小间距。
>- 2) 改相机参数：收紧 near/far。
>- 3) 用 polygon offset（贴花等典型场景）。
>- 4) 用 Reversed-Z 或分层深度方案改善整体精度。
>- **注意**
>- 只靠“换更高分辨率深度格式”常常不够，参数和几何关系同样关键。

>[!tip]- 代码示例
>```cpp
>// Decal 场景常见做法（示意）
>EnablePolygonOffset(true);
>SetPolygonOffset(factor = 1.0f, units = 1.0f);
>DrawDecal();
>```
>
>```mermaid
>flowchart TD
>  A[z-fighting出现] --> B[先排查是否共面]
>  B --> C[调整near/far]
>  C --> D[贴花场景用polygon offset]
>  D --> E[需要时采用Reversed-Z]
>```

>[!quote]- 通俗易懂的理解
>- 两张纸几乎贴在一起，深度缓冲分不清谁更靠前，就会一会儿画这张、一会儿画那张，表现为闪烁。

>[!example]- 面试回答简版模板
>`z-fighting 是深度精度竞争问题。优先从几何关系和 near/far 入手，贴花类可用 polygon offset，远景精度不足时可采用 Reversed-Z。`
